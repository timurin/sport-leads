# Communication connector core

## Назначение

Слой `app.communications` отделяет внутреннюю модель переписки от протоколов
внешних сервисов. Конкретный Telegram, VK, email или WhatsApp-клиент должен
реализовать единый `CommunicationConnector`; остальной код работает только с
командами, результатами и `NormalizedMessage`.

На этапе v0.5.9 реализован только `MockCommunicationConnector`. Он не открывает
сетевые соединения, не сохраняет данные в PostgreSQL и не используется
frontend напрямую.

```text
Frontend
   │
   ▼
API / CommunicationService
   │
   ▼
ConnectorRegistry
   │
   ├── TelegramConnector  (будущий)
   ├── VkConnector        (будущий)
   ├── EmailConnector     (будущий)
   └── MockConnector
           │
           ▼
NormalizedMessage
           │
           ▼
Repository / Queue / Timeline  (будущие)
```

## Исходящий поток

1. Будущий API преобразует проверенный пользовательский ввод в
   `SendMessageCommand`.
2. `CommunicationService` запрещает внешнюю отправку для `internal` и
   текстовую отправку для `phone`.
3. `ConnectorRegistry` выбирает реализацию по `CommunicationChannel`.
4. Коннектор отправляет сообщение и возвращает `SendMessageResult`.
5. Сервис преобразует результат и исходную команду в `NormalizedMessage`.
6. Опциональная зависимость `MessageRepository` является точкой расширения
   для будущего сохранения. Реализации repository в текущем этапе нет.

## Входящий webhook

Будущий API должен передать канал, payload и заголовки в
`CommunicationService.process_webhook`. Выбранный коннектор проверяет подпись,
валидирует только известные ему поля и возвращает список нормализованных
сообщений. Постоянно работающий `receive_message` не нужен; email polling в
будущем сможет использовать тот же формат.

Mock payload содержит безопасные тестовые идентификаторы события, сообщения,
диалога и отправителя, текст, время и метаданные вложений. Полный внешний raw
payload в `NormalizedMessage` не копируется.

## Нормализованная модель

`NormalizedMessage` содержит внутренний UUID, канал, направление, статус,
внешние идентификаторы сообщения и диалога, отправителя и получателя, текст,
временные метки, вложения и JSON-метаданные. Также предусмотрены `lead_id`,
`contact_id`, `connector_name`, `raw_event_id`, ссылка на сообщение-родитель и
признак mock-данных.

`NormalizedAttachment` хранит только идентификаторы и метаданные: имя,
MIME-тип, размер, source URL и будущий storage key. Байты файла, токены и
секреты в схемах отсутствуют.

Значения каналов синхронизированы с frontend `LeadMessageChannel`. Backend
дополнительно поддерживает статус `queued`, предназначенный для будущей
очереди. До появления API frontend v0.5.8 продолжает использовать локальное
mock-состояние.

## Registry и зависимости

`ConnectorRegistry` создаётся вызывающим кодом и получает готовые экземпляры
коннекторов. Импорт пакета не создаёт подключений или глобального изменяемого
registry. Повторная регистрация канала запрещена, пока вызывающий код явно не
передаст `replace=True`.

## Ошибки

Все коммуникационные ошибки наследуются от `CommunicationError` и сообщают,
является ли ошибка временной, допустим ли retry, нужно ли уведомлять
администратора и следует ли окончательно помечать сообщение как `failed`.
Отдельные типы предусмотрены для конфигурации, аутентификации, rate limit,
временных и постоянных сбоев, отсутствующего connector и валидации сообщения.
Тексты ошибок не содержат секреты, payload или содержимое сообщений.

## Идемпотентность

Mock-коннектор дедуплицирует входящие данные в памяти экземпляра по сочетанию
`raw_event_id`, `external_message_id`, канала и имени коннектора. Повторный
webhook возвращает пустой список. Production-дедупликация должна опираться на
уникальное ограничение PostgreSQL либо атомарную проверку в Redis; in-memory
set не подходит для нескольких процессов и перезапусков.

## Безопасность

Mock webhook может быть защищён тестовым секретом из
`MockConnectorConfig`. Значения сравниваются через `hmac.compare_digest` по
хешам, секрет не логируется. Реальные коннекторы должны реализовать собственную
проверку подписи в `verify_webhook` до нормализации данных.

В журнал попадают только канал, connector, внутренний идентификатор сообщения,
event id и статус. Тексты, персональные данные, ключи и содержимое вложений не
логируются.

## Будущие коннекторы

- `TelegramConnector` адаптирует Telegram Bot API update и delivery semantics.
- `VkConnector` нормализует Callback API и результаты отправки VK.
- `EmailConnector` объединит SMTP-отправку с webhook или IMAP collector.
- `WhatsAppConnector` адаптирует WhatsApp Business API.
- `WebsiteConnector` нормализует события форм сайта.

Каждая реализация должна оставаться за контрактом `CommunicationConnector` и
не переносить особенности провайдера в API или бизнес-логику.

## Ограничения mock-этапа

Нет реальных API, webhook endpoint, БД сообщений, миграций, очереди, retry
worker, Redis, загрузки файлов или S3. Отправленные mock-команды и ключи
дедупликации существуют только в памяти конкретного экземпляра коннектора.
