# Контакты лида

Итерации `v0.6.1-lead-contact-persistence` и `v0.6.1-lead-contact-frontend` добавляют постоянный backend-контур контактов лида и подключают к нему существующие формы числовой detail-страницы, не создавая вторую модель `Lead`.

## Источник истины

Таблица `lead_contacts` является источником истины для контактов лида. Поля `leads.contact_name`, `leads.phone` и `leads.email` временно сохранены для обратной совместимости существующего API и конвертации и синхронизируются с основным контактом сервисом.

Конвертация сначала читает основной `LeadContact`, а при его отсутствии использует legacy-поля. Это сохраняет работу ранее созданных лидов и позволяет удалить совместимую проекцию в отдельной будущей миграции после перевода всех потребителей.

## Целостность

- внешний ключ `lead_contacts.lead_id → leads.id` использует `ON DELETE CASCADE`;
- частичный уникальный индекс `uq_lead_contacts_primary_per_lead` допускает максимум один `is_primary = true` на лид;
- первый созданный контакт автоматически становится основным;
- смена primary блокирует строку лида, снимает предыдущий признак и назначает новый в одной транзакции;
- основной контакт нельзя удалить; сначала необходимо назначить другой;
- неизвестные lead/contact возвращают 404;
- email валидируется Pydantic-схемой;
- timestamps имеют timezone-aware тип БД.

## API

- `GET /leads/{lead_id}` — возвращает лид вместе с контактами;
- `POST /leads/{lead_id}/contacts` — создаёт контакт;
- `PATCH /leads/{lead_id}/contacts/{contact_id}` — изменяет контакт;
- `DELETE /leads/{lead_id}/contacts/{contact_id}` — удаляет только дополнительный контакт;
- `POST /leads/{lead_id}/contacts/{contact_id}/set-primary` — атомарно меняет основной контакт.

API использует `LeadContactCreate`, `LeadContactUpdate` и `LeadContactRead`; ORM-объекты явно преобразуются в response schema.

## Frontend

- `getLeadDetails` выбирает источник явно: `lead-1`/`lead-2` читаются из demo-data, числовые идентификаторы — только из backend;
- `fromApiLead` преобразует `LeadRead.contacts` через общий mapper и не синтезирует контакт из legacy-полей;
- Server Actions выполняют create/update/delete/set-primary на сервере Next.js и возвращают форме актуальный список контактов;
- при ошибке API форма остаётся открытой, локальное состояние не выдаётся за сохранённое;
- `LeadPage` после успешной мутации вычисляет primary из полученного списка, поэтому шапка, карточка клиента и коммуникационная колонка синхронизируются одновременно;
- demo-маршруты сохраняют прежнее локальное поведение с явной пометкой.

## Миграция и проверки

Миграция `b6c4e2f91a07_add_lead_contacts.py` создаёт таблицу, индексы и основной контакт для каждого существующего лида из legacy-полей. Downgrade удаляет только новый контактный контур.

`backend/tests/test_lead_contacts.py` проверяет CRUD, единственный primary, синхронизацию проекции, 404, валидацию, запрет удаления primary и использование primary-контакта существующей конвертацией. `frontend/lib/sales/lead-contact-api.test.mjs` проверяет преобразование ответа API и нормализацию mutation payload.

## Ограничения

- `lead-1` и `lead-2` остаются demo/local и не используются как fallback при ошибке backend;
- `Client` пока хранит плоский контакт и не связан с `LeadContact`;
- поле messenger и остальные данные профиля клиента пока не входят в `LeadContact` API и остаются локальными;
- создание лидов и development seed не входят в эту итерацию;
- коммерческие данные, задачи, заметки, сообщения и connectors не изменялись.
