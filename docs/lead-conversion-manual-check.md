# Ручная проверка конверсии лида

## Backend API

1. Запустить PostgreSQL: `docker compose up -d postgres`.
2. Применить схему: `cd backend && ..\.venv\Scripts\python.exe -m alembic upgrade head`.
3. Запустить API: `..\.venv\Scripts\python.exe -m uvicorn app.main:app --reload`.
4. Открыть `http://127.0.0.1:8000/docs`.
5. Проверить справочник через `GET /lead-rejection-reasons?is_active=true`.
6. Для существующего активного лида вызвать `POST /leads/{lead_id}/convert`. Проверить один созданный заказ, `status=completed`, `result=converted` и взаимные ссылки `lead.converted_order_id` / `order.lead_id`.
7. Повторить запрос: API должен вернуть `409` и не создать второй заказ.
8. Для другого активного лида вызвать `POST /leads/{lead_id}/reject`. Для причин `competitor` и `other` комментарий обязателен. Проверить `result=rejected` и отмену открытых задач.
9. Проверить историю через `GET /leads/{lead_id}/history` и `GET /orders/{order_id}/history`, исходный лид — через `GET /orders/{order_id}/source-lead`.

API пока не содержит аутентификации. До появления auth-контекста операции принимают `completed_by_id`; по умолчанию используется системный пользователь с `id=1`, созданный миграцией.

## Frontend

1. Запустить `cd frontend && npm run dev` и открыть `http://127.0.0.1:3000/sales/leads`.
2. Переключить режимы «Активные», «Успешные», «Отказы», «Все».
3. Перетащить активный лид между рабочими стадиями и убедиться, что DnD работает.
4. Нажать «Завершить лид», выбрать успешный исход, скорректировать предварительные данные заказа и создать заказ.
5. Проверить результат, номер заказа и кнопку «Открыть заказ»; завершённая карточка не должна перетаскиваться.
6. Завершить другой лид отказом. Проверить клиентскую валидацию обязательного комментария для «Другое» или «Выбрал конкурента».
7. Открыть `/sales/dashboard`: KPI должны использовать результаты лидов, а таблица источников и блок причин отказов — показывать конверсии/отказы.
8. Открыть прежние страницы `/sales/deals`, `/sales/orders`, `/sales/tasks`, `/sales/clients` и проверить их Kanban/таблицы.

Frontend Sales остаётся демонстрационным локальным прототипом: изменения состояния не переживают перезагрузку страницы. Транзакционная запись и валидация реализованы в backend API.
