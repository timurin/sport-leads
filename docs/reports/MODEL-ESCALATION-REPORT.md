# Model escalation report — lead creation and archiving

Дата: 2026-07-18
Ветка: `feature/v0.6.1-lead-persistence`

## Выбранный пункт roadmap

Следующий незавершённый пункт текущего CRM-этапа — создание и архивирование лида через рабочий frontend/API-сценарий.

## Результат безопасного аудита

- `backend/app/api/leads.py` содержит read API, PATCH статуса/полей, contact API, convert и reject, но отдельного `POST /leads` нет.
- `backend/app/schemas/sales.py` содержит `LeadUpdate` и `LeadRead`, но отдельной схемы создания лида нет.
- В frontend список лидов получает API-данные, однако кнопка создания использует demo/local UI и не имеет backend-контракта.
- Архивирование не представлено отдельным полем, endpoint или подтверждённым правилом жизненного цикла.

## Причина эскалации

Реализация потребует нового API-контракта, схемы создания, правил обязательных полей, записи системного события, определения archive-состояния и согласования поведения списка/Kanban. Это новое архитектурное решение и изменение бизнес-жизненного цикла данных, а не безопасная локальная доработка существующего CRUD.

## Риски

- несовместимый контракт создания лида;
- незафиксированные правила архивирования и восстановления;
- расхождение между PostgreSQL, списком, Kanban и demo-сценариями;
- потеря или скрытие лидов при ошибочном статусе архива;
- необходимость миграции при выборе отдельного archive-поля.

## Продолжение

После ручного переключения на более сильную модель требуется сначала согласовать контракт `POST /leads`, archive-модель и frontend-сценарий, затем реализовать первую вертикальную часть с тестами и миграцией при необходимости.

Требуется более сильная модель. Перезапустите Codex и повторите задачу.

- Существующий `PATCH /leads/{lead_id}` выполняет `commit`, `refresh` и возвращает обновлённый `LeadRead`; дефект находится не в самом commit.

- `direction`, `product_type`, `kit_quantity`, `size_comment`, `preliminary_budget`, `discount_percent`, `probability`, даты заказа/мероприятия, адрес/способ/комментарий доставки, campaign, UTM и priority отсутствуют в `Lead`, `LeadUpdate` и `LeadRead`; после сохранения они существуют только в React state и исчезают при reload.
- Ответственный в `LeadHeader` также меняется только локально; backend-поле `responsible_id` существует, но текущий UI использует demo-список менеджеров и не вызывает PATCH.
- Профиль клиента, контакты и стадия уже имеют backend-контур, однако их необходимо включить в сквозной regression-набор изменения, очистки и повторного чтения.

## Причина эскалации

Полное выполнение критерия задачи требует изменения бизнес-модели и обратимой миграции для коммерческих и финансовых данных: предварительного бюджета, скидки, предполагаемой суммы и связанных условий заказа. Это прямо подпадает под правило Model escalation для финансов и затрагивает контракт `Lead` от БД до frontend DTO.

## Риски

- неверные типы или точность денежных значений;
- частичное сохранение карточки при расхождении схем frontend/backend;
- потеря значений при очистке (`undefined` против `null`);
- несовместимая миграция существующей таблицы `leads`;
- назначение несуществующего ответственного из demo-списка.

## Продолжение задачи

После ручного переключения модели:

1. Зафиксировать полный единый mapping всех редактируемых полей карточки.
2. Добавить nullable-колонки `Lead` с `Numeric/Decimal` для денежных значений и обратимую Alembic-миграцию.
3. Расширить `LeadUpdate`, `LeadRead`, frontend API DTO/mapping и существующий Server Action без нового endpoint.
4. Подключить ответственного к PATCH с backend-валидным списком `SalesUser` либо ограничить UI данными, подтверждёнными backend.
5. После ответа PATCH обновлять state только данными ответа backend.
6. Проверить изменение, заполнение пустого, очистку, повторное изменение и повторный GET для всех полей; затем выполнить полный обязательный набор проверок и обновить roadmap/ERP-check.

# Model escalation report — persistent custom lead stages

Дата: 2026-07-18
Ветка: `feature/v0.6.1-lead-persistence`
Task: `docs/tasks/patch-0.6-lead-ux-custom-stages.md`, пункт 2

## Выбранный пункт

Следующая незавершённая часть патча — постоянное хранение пользовательских стадий Kanban со стабильными идентификаторами, переименованием, сортировкой и сохранением стадии лида после reload.

## Результат безопасного аудита

- Конфигурация стадий существует только во frontend и хранится в `localStorage` под ключом `sport-leads.sales.lead-stages.v1`.
- Пользовательский идентификатор создаётся локально в формате `custom-N`; backend не является владельцем его уникальности.
- `Lead.status` хранится как PostgreSQL enum `lead_status` и типизирован Python enum `LeadStatus`, содержащим только `new`, `contact`, `qualification`, `proposal`, `waiting` и `completed`.
- `LeadUpdate.status` использует тот же закрытый enum, поэтому текущий `PATCH /leads/{lead_id}` не может принять пользовательскую стадию.
- Frontend-функция `getLeadStagePersistenceDecision` дополнительно отклоняет всё вне пяти заранее заданных рабочих стадий как `unsupported-stage`.
- Kanban и detail-карточка читают локальную конфигурацию стадий, но не имеют общего backend API для её загрузки и изменения.
- Отключение стадии с переносом лидов сейчас меняет только frontend-состояние и не является атомарной backend-операцией.

## Причина эскалации

Пункт требует нового архитектурного решения и миграции бизнес-данных. Необходимо определить:

1. Постоянную сущность стадии и владельца стабильного идентификатора.
2. Связь активного лида со стадией: foreign key либо иной контракт, совместимый с финальным состоянием `completed` и `Lead.result`.
3. Миграцию существующих enum-значений без потери стадий и нарушения ограничений завершения.
4. API и service/repository-границы для чтения, создания, переименования, сортировки, отключения и атомарного переноса лидов.
5. Область конфигурации стадий. Сейчас в системе нет организации/auth-контекста, поэтому преждевременная глобальная модель может осложнить будущую изоляцию данных.
6. Начальное заполнение системных стадий и правила неизменяемости финальных бизнес-действий.

Это не обычный CRUD поверх утверждённой модели: сейчас backend-модели стадии нет, а выбранная схема изменит контракт `Lead`, базу данных, Kanban и detail-карточку.

## Риски

- потеря или неверное сопоставление стадий существующих лидов при миграции enum;
- появление лидов со ссылкой на отсутствующую или отключённую стадию;
- расхождение Kanban и detail-карточки из-за двух источников истины;
- частичное сохранение порядка стадий или переноса лидов;
- смешение рабочих стадий с `converted`/`rejected`, которые должны оставаться бизнес-результатами;
- контракт, который позже нарушит изоляцию организаций.

## Продолжение задачи

После ручного переключения на более сильную модель:

1. Зафиксировать модель `LeadStage` и стратегию перехода от `LeadStatus`.
2. Создать миграцию с `upgrade`/`downgrade` и безопасным переносом пяти текущих стадий.
3. Реализовать schemas, service/repository и API конфигурации стадий.
4. Подключить Kanban к backend-источнику без успешного local fallback.
5. Покрыть создание, переименование, порядок, перенос лида и reload backend/frontend тестами.

Бизнес-код и схема базы в рамках этого аудита не изменялись.

Требуется более сильная модель. Перезапустите Codex и повторите задачу.

# Model escalation report — v0.7.0 sales order foundation

Дата аудита: 2026-07-18
Ветка: `feature/v0.6.1-lead-persistence`
Задача: `docs/tasks/v0.7.0-sales-order-foundation-and-project-status.md`

## Причина остановки

Следующая итерация v0.7.0 требует реализации карточки заказа покупателя и затрагивает несколько бизнес-контуров одновременно:

- backend-модель и read-контракт `SalesOrder`;
- связи `SalesOrder` с `Lead` и `Client`;
- API чтения заказа и обработку not-found/error состояний;
- frontend data layer, маршрут `/sales/orders/[orderId]` и навигацию из списка;
- синхронизацию roadmap, ERP-check и автономных HTML-отчётов.

Это новое архитектурное решение в бизнес-модуле продаж, а не локальное изменение CRUD. По правилам `AGENTS.md` и roadmap v0.6.1 до изменения бизнес-кода требуется ручная эскалация модели.

## Результаты безопасного аудита

- Рабочая ветка: `feature/v0.6.1-lead-persistence`.
- HEAD: `0c2d88f release: complete CRM Leads MVP v0.6.0`.
- До аудита в рабочем дереве находился только незакоммиченный task-файл v0.7.0; он сохранён.
- `SalesOrder` уже существует в `backend/app/models/sales.py`.
- Существуют `GET /orders`, `GET /orders/{order_id}`, `source-lead` и `history` в `backend/app/api/orders.py`.
- Существуют `SalesOrderRead` и `LeadConversionRead` в `backend/app/schemas/sales.py`.
- Конвертация `Lead → Client + SalesOrder` реализована транзакционно в `backend/app/services/lead_conversion.py` и покрыта тестами.
- Список `/sales/orders` уже загружает backend-данные; отдельной рабочей карточки заказа в маршруте `[orderId]` не обнаружено.
- Frontend имеет demo-типы и demo-шаблоны заказов, которые нельзя принимать за persistence-контур.
- В `SL-ROADMAP-v0.6.1` заказ отмечен как минимальный backend/read-контур, а полноценная карточка остаётся незавершённой.

## Проверки

- `.venv\Scripts\python.exe -m pytest`: `37 passed`.
- `git diff --check`: пройден.
- `scripts/check_project.py`: не пройден по причинам окружения:
  - PermissionError при записи `.pyc` в существующие `backend/alembic/**/__pycache__`;
  - UnicodeEncodeError при выводе символа `▲` frontend lint через cp1251.
- В ходе аудита файлы приложения, миграции и данные не изменялись.

## Риски без эскалации

Самостоятельное проектирование нового контракта карточки заказа может привести к дублированию существующего endpoint, расхождению nullable-полей, неверной связи с исходным лидом/клиентом и преждевременному повышению статуса модуля в документации. Поэтому дальнейшая реализация приостановлена.

## Продолжение после смены модели

После ручного выбора более сильной модели продолжить с первой законченной подзадачи: реализовать read-only foundation карточки заказа на существующем `GET /orders/{order_id}`, добавить mapping и состояния loading/not-found/API error, тесты и только затем обновить roadmap, ERP-check и два автономных HTML-отчёта.

## Результат после продолжения

После ручного переключения модели выполнена итерация `v0.7.0-sales-order-foundation`: существующий detail API возвращает имена клиента и ответственного, список открывает `/sales/orders/[orderId]`, а карточка обрабатывает loading, not-found, API error и nullable-поля без demo fallback. Миграция не потребовалась; полный набор проверок пройден.
