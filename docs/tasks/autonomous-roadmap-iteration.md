# Автономная итерация Sport-Lead по roadmap

Работай как ведущий разработчик проекта Sport-Lead.

## Источники истины

Сначала полностью прочитай:

- AGENTS.md;
- docs/roadmap/SL-ROADMAP-v0.3.0.md;
- docs/architecture/SL-ERP-CHECK-v0.3.0.md;
- последний документ в docs/releases;
- актуальную архитектурную документацию;
- текущий код и тесты.

## Цель

Самостоятельно определить и реализовать один следующий незавершённый пункт текущего этапа roadmap.

Не выбирать задачу по удобству.
Не перескакивать на следующий модуль.
Не заниматься визуальной полировкой, если она не блокирует основной сценарий.

## Шаг 1. Безопасная проверка

Выполни:

git status --short
git branch --show-current
git log --oneline --decorate -10
git diff --check

Если рабочее дерево содержит изменения:

- проанализируй их;
- не удаляй;
- не откатывай;
- определи, относятся ли они к текущей итерации;
- сохрани совместимость с ними.

Если есть незавершённый конфликт merge или rebase, остановись.

## Шаг 2. Выбор задачи

Сопоставь фактическое состояние проекта с roadmap и ERP-check.

Определи:

- текущий этап;
- завершённые пункты;
- частично реализованные пункты;
- первый логически следующий незавершённый пункт;
- его зависимости;
- риски для существующей реализации.

Выбери только один пункт.

Перед кодированием выведи краткий план, но не жди подтверждения, если:

- задача находится внутри текущего этапа;
- нет P0/P1;
- не требуется изменение утверждённого roadmap;
- не требуется удаление данных;
- не требуется переход к следующему модулю.

## Шаг 3. Реализация

Реализуй выбранный пункт полностью в рамках одной безопасной итерации.

Обязательные правила:

- сначала использовать существующие сущности;
- не создавать параллельные модели и сервисы;
- не ломать существующие API-контракты;
- не менять дизайн без необходимости;
- сохранять обратную совместимость;
- добавлять тесты;
- создавать обратимую миграцию при изменении БД;
- не включать следующий пункт roadmap.

При слишком большой задаче:

- раздели её на подзадачи;
- реализуй только первую законченную вертикальную часть;
- зафиксируй остаток в отчёте.

## Шаг 4. Самопроверка

Проведи review собственного diff:

- дублирование;
- нарушение слоёв;
- ошибки транзакций;
- необработанные исключения;
- несовместимость schemas;
- float для денег;
- naive datetime;
- отсутствие rollback;
- скрытая подмена API mock-данными;
- нарушение существующей конвертации лида;
- случайные изменения дизайна.

Исправь обнаруженные ошибки внутри текущей области.

## Шаг 5. Тесты

Запусти применимые проверки.

Backend:

python scripts/check_project.py
python -m pytest

Frontend:

cd frontend
npx tsc --noEmit
npm run lint
npm run build
cd ..

Git:

git diff --check
git status --short

При изменении database models дополнительно проверь Alembic:

- текущую цепочку revisions;
- upgrade head;
- downgrade на одну revision;
- повторный upgrade;
- alembic check.

Не считать задачу завершённой при падении обязательных тестов.

## Шаг 6. Документация

Обнови:

- SL-ROADMAP-v0.3.0;
- SL-ERP-CHECK-v0.3.0;
- связанную архитектурную документацию.

Меняй статусы только по подтверждённому результату.

## Ограничения

Не выполнять без отдельного разрешения:

- commit;
- push;
- tag;
- merge;
- rebase;
- force-команды Git;
- удаление данных;
- production-миграции;
- переход к следующему этапу roadmap;
- полный редизайн;
- массовый рефакторинг.

## Завершение

Покажи отчёт:

1. Выбранный пункт roadmap.
2. План и границы.
3. Реализованный результат.
4. Изменённые файлы.
5. Миграции и API.
6. Тесты.
7. P0–P3.
8. Обновления roadmap и ERP-check.
9. Остаток текущего пункта.
10. Следующую рекомендуемую итерацию.

После отчёта остановись.
Не начинай следующую итерацию автоматически.
