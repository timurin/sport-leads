# Patch v0.8.1 — выбор номенклатуры в заказе покупателя

## Контекст

В `v0.8.1-nomenclature-core` реализованы:

- persistent-справочник номенклатуры;
- API `/nomenclatures`;
- nullable-связь `SalesOrderItem.nomenclature_id`;
- snapshot-поля позиции заказа.

Во время визуальной проверки выявлен P1-дефект:

в карточке заказа покупателя в блоке «Товарные позиции» отсутствует возможность выбрать номенклатуру из справочника.

Перед работой изучить:

- `AGENTS.md`;
- `docs/architecture/project-structure.md`;
- `docs/architecture/erp-check.md`;
- `docs/roadmap/roadmap.md`;
- `ADR-003`;
- `ADR-005`;
- текущую реализацию страницы заказа;
- API и frontend справочника номенклатуры;
- Server Actions позиций заказа.

## Цель

Добавить в создание и редактирование позиции заказа выбор активной номенклатуры из persistent-справочника.

## Требуемое поведение

### Новая позиция заказа

В строке добавления позиции добавить поле выбора номенклатуры.

Пользователь должен иметь возможность:

- открыть список активной номенклатуры;
- найти позицию по наименованию или артикулу;
- выбрать номенклатуру;
- после выбора автоматически заполнить snapshot-поля позиции:
  - наименование;
  - базовую цену;
  - `nomenclature_id`;
- при необходимости вручную изменить наименование и цену до сохранения;
- заполнить размеры, персонализацию, цвет, количество и скидку;
- сохранить позицию через существующий backend.

### Существующая позиция

Для сохранённой позиции:

- отображать связанную номенклатуру;
- разрешить заменить выбранную номенклатуру;
- при замене обновлять `nomenclature_id`;
- копировать актуальные значения выбранной номенклатуры в snapshot только как осознанное действие пользователя;
- не менять позицию автоматически при последующем редактировании справочника;
- разрешить оставить позицию без связи с номенклатурой для старых и ручных строк.

## UI

Использовать текущий визуальный стиль страницы заказа.

Не добавлять обычный HTML `<select>`, если список может стать большим.

Предпочтительно реализовать компактный searchable combobox/autocomplete:

- отображение: `Артикул — Наименование`;
- поиск по артикулу и наименованию;
- только активные позиции;
- понятное пустое состояние;
- состояние загрузки;
- корректная работа с клавиатурой;
- адаптивность на desktop и mobile.

В строке заказа должно быть понятно:

- какая номенклатура выбрана;
- какие данные являются snapshot позиции;
- что ручные значения допустимы.

Не выполнять полный редизайн страницы заказа в рамках патча.

## Backend и persistence

## Критерии готовности

Патч готов, когда:

- в заказе можно найти и выбрать активную номенклатуру;
- выбор сохраняет `nomenclature_id`;
- наименование и цена копируются в snapshot;
- snapshot можно корректировать вручную;
- после перезагрузки связь сохраняется;
- изменение справочника не меняет старые позиции;
- ручные позиции без номенклатуры продолжают работать;
- все проверки проходят;
- отсутствуют P0/P1.
