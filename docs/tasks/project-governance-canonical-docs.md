# Каноническая документация и память проекта

## Контекст

Проекту нужна устойчивая система памяти, независимая от истории чатов.

Текущие рабочие документы включают несколько версий roadmap и ERP-check,
а логические коды SL-ROADMAP-v1 и SL-ERP-CHECK-v1 не привязаны к конкретным
физическим файлам.

Задача относится только к документации и правилам разработки.
Код приложения, модели, API, миграции и бизнес-логику не менять.

## Цель

Провести аудит существующей документации и создать единый набор
канонических источников истины:

- AGENTS.md;
- docs/architecture/project-structure.md;
- docs/architecture/erp-check.md;
- docs/roadmap/roadmap.md;
- docs/architecture/decisions/;
- docs/process/;
- два существующих HTML-статуса.

## 1. Аудит

Перед изменениями проверить:

- git status и последние коммиты;
- все файлы в docs/roadmap;
- все файлы в docs/architecture;
- docs/releases;
- docs/tasks;
- docs/erp/status;
- AGENTS.md и другие agent-файлы;
- README и внутренние ссылки на roadmap/ERP-check;
- дубли, временные отчёты и устаревшие документы.

Составить перечень:

- канонический источник;
- уникальная полезная информация;
- дубликат;
- исторический документ;
- временный файл;
- файл, который можно удалить;
- файл, который нужно сохранить в архиве.

Ничего не удалять до переноса уникальной информации.

## 2. Создать канонические документы

Создать или привести к каноническому виду:

### docs/roadmap/roadmap.md

Единый глобальный roadmap проекта.

Закрепить алиас:

SL-ROADMAP-v1 → docs/roadmap/roadmap.md

Сохранить утверждённые этапы 0–10 и фактический статус по коду.

### docs/architecture/project-structure.md

Глобальная архитектура модулей, сущностей и зависимостей.

Закрепить алиас:

SL-PROJECT-STRUCTURE-v1 → docs/architecture/project-structure.md

Явно разделить:

- CRM и продажи;
- номенклатуру и продукцию;
- модели, размеры и лекала;
- технологическую подготовку;
- материалы и ресурсы;
- производственные партии;
- спецификации;
- производственный факт;
- склад;
- закупки;
- финансы;
- интеграцию с 1С.

### docs/architecture/erp-check.md

Фактическая готовность ERP с чек-листом.

Закрепить алиас:

SL-ERP-CHECK-v1 → docs/architecture/erp-check.md

Статусы:

- [x] готово;
- [~] частично;
- [ ] не начато;
- [!] блокер или технический долг;
- [?] требует ручной проверки.

Статусы подтверждать кодом, API, миграциями и тестами, а не только старыми
документами.

## 3. Архитектурные решения

Создать каталог:

docs/architecture/decisions

Перенести подтверждённые решения в ADR:

- ADR-001 — Lead → SalesOrder;
- ADR-002 — SalesOrder → Organization;
- ADR-003 — SalesOrderItem является коммерческой позицией, Material не
  является номенклатурой заказа;
- ADR-004 — производственная цепочка и момент формирования спецификации.

Формат каждого ADR:

- статус;
- контекст;
- принятое решение;
- последствия;
- ограничения;
- связанные модули.

В ADR-004 закрепить цепочку:

Заказ покупателя
→ Позиции заказа
→ Номенклатура
→ Модель / артикул
→ Размеры и характеристики
→ Производственная партия
→ Утверждённая спецификация партии
→ Печать, раскрой, пошив и ОТК
→ Фактический расход и выпуск
→ Передача данных и документов в 1С.

Спецификация является плановым составом и формируется до запуска партии.
Производственный факт хранится отдельно.

## 4. Процесс разработки

Создать:

### docs/process/development-workflow.md

Закрепить роли:

- пользователь — владелец продукта и тестировщик;
- ChatGPT — архитектор;
- Codex — разработчик;
- Git — фактическая история;
- канонические документы — память проекта.

Цикл:

задача → реализация → автоматические проверки → визуальная проверка
→ patch при необходимости → commit → push → следующая задача.

### docs/process/codex-task-style.md

Закрепить:

- задачи компактные, но ёмкие;
- одна законченная функция на итерацию;
- не повторять весь roadmap в каждой задаче;
- всегда указывать цель, объём, ограничения, проверки и готовность;
- не переходить к следующему пункту;
- не создавать commit без разрешения;
- итоговый отчёт делать кратким.

### docs/process/release-checklist.md

Добавить checklist:

- тесты;
- миграции;
- TypeScript;
- lint;
- build;
- project check;
- git diff --check;
- ручная проверка;
- обновление документации;
- commit, tag и push для контрольной точки.

## 5. Обновить AGENTS.md

AGENTS.md должен требовать перед каждой задачей читать:

1. AGENTS.md;
2. docs/architecture/project-structure.md;
3. docs/architecture/erp-check.md;
4. docs/roadmap/roadmap.md;
5. связанные ADR;
6. task текущей итерации;
7. git status и последние коммиты.

Зафиксировать алиасы:

- SL-ROADMAP-v1 → docs/roadmap/roadmap.md;
- SL-ERP-CHECK-v1 → docs/architecture/erp-check.md;
- SL-PROJECT-STRUCTURE-v1 → docs/architecture/project-structure.md.

После каждой завершённой итерации обновлять только затронутые канонические
документы и оба HTML-статуса.

Запретить:

- создание параллельных master-roadmap;
- использование HTML как первичного источника;
- автоматическое закрытие пунктов без подтверждения кодом;
- commit и push без разрешения пользователя.

## 6. Миграция старой документации

Перенести всю уникальную информацию из существующих файлов, включая:

- docs/roadmap/SL-ROADMAP-v0.3.0.md;
- docs/roadmap/SL-ROADMAP-v0.6.1.md;
- docs/architecture/SL-ERP-CHECK-v0.3.0.md;
- релизные документы;
- актуальные архитектурные отчёты.

После переноса:

- исправить все внутренние ссылки;
- проверить упоминания старых путей;
- исторические release-документы сохранить;
- завершённые task-файлы не удалять, если они нужны для истории;
- устаревшие глобальные roadmap и ERP-check либо удалить, либо перенести
  в docs/archive — выбрать вариант по существующим правилам проекта;
- не оставлять конкурирующие master-документы.

Удалять только подтверждённые временные файлы и явные дубли.

В итоговом отчёте перечислить каждый удалённый или перемещённый файл и
обоснование.

## 7. HTML-статусы

Обновить:

- docs/erp/status/project-structure.html;
- docs/erp/status/roadmap.html.

Правило:

Markdown — источник истины.
HTML — визуальное представление.

HTML должен соответствовать новым каноническим документам, оставаться
автономным, адаптивным и без внешних зависимостей.

## Ограничения

Не изменять:

- backend;
- frontend;
- модели;
- API;
- миграции;
- тесты приложения;
- текущую функциональность.

Не исправлять в этой задаче связь SalesOrderItem → Material.
Только зафиксировать её в ADR и ERP-check как технический долг.

Commit и push не выполнять.

## Проверки

Выполнить:

- python scripts/check_project.py;
- git diff --check;
- поиск ссылок на старые roadmap и ERP-check;
- проверку отсутствия конфликтующих master-документов;
- проверку локального открытия обоих HTML;
- проверку отсутствия внешних зависимостей в HTML;
- проверку UTF-8.

## Критерии готовности

- существует один канонический roadmap;
- существует одна каноническая структура проекта;
- существует один канонический ERP-check;
- алиасы имеют физические пути;
- ключевые решения вынесены в ADR;
- правила работы закреплены в AGENTS.md и docs/process;
- уникальная информация старых документов не потеряна;
- старые дубли безопасно удалены или архивированы;
- HTML синхронизированы;
- код приложения не изменён;
- commit не создан.

## Итоговый отчёт

Кратко указать:

1. что было найдено;
2. какие документы признаны устаревшими или дублирующими;
3. какие канонические файлы созданы;
4. какие ADR созданы;
5. какие файлы перемещены или удалены;
6. какие ссылки исправлены;
7. что изменено в AGENTS.md;
8. результаты проверок;
9. известные ограничения;
10. что требуется проверить вручную.

После завершения остановиться.
