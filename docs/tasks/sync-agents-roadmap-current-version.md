# Задача Codex: синхронизировать AGENTS.md и roadmap с текущей версией

## Контекст

GitHub `main` содержит checkpoint `v0.5.0 leads functional`, но локальная рабочая копия может быть новее.

Нельзя автоматически считать `v0.5.0` текущей версией. Сначала определи фактическое состояние локальной ветки, последние checkpoint/release-документы и незакоммиченные изменения.

## Цель

1. Создать короткий корневой `AGENTS.md`.
2. Добавить правило ручного переключения модели.
3. Найти или создать актуальный roadmap.
4. Обновить его по фактическому состоянию проекта.
5. Зафиксировать, что Codex обязан сообщать владельцу, когда нужна более сильная модель.

## Перед началом

Проверь:

- текущую ветку;
- `git status`;
- последние 20 коммитов;
- теги;
- `docs/checkpoints`;
- `docs/roadmap`;
- `docs/architecture`;
- `docs/reports`;
- существующие инструкции Codex;
- фактические backend/frontend реализации.

Если локальная версия новее GitHub checkpoint `v0.5.0`, используй локальную подтвержденную версию.

## 1. Создать или обновить `AGENTS.md`

Добавь компактный раздел:

```md
## Model escalation

Используй текущую экономичную модель для документации, аудита по готовому checklist, обычного CRUD, локального UI, простых миграций и тестов.

До изменения бизнес-кода попроси владельца вручную переключить модель на более сильную, если задача включает хотя бы одно:

- новое архитектурное решение;
- authentication, sessions, passwords, tokens, RBAC или permissions;
- изоляцию данных организаций;
- финансы, оплаты, себестоимость, складские остатки или списания;
- сложную миграцию существующих данных;
- изменения сразу в нескольких бизнес-модулях;
- риск потери данных, нарушения прав или неверных остатков;
- две неудачные обоснованные попытки исправить одну ошибку.

При необходимости переключения:

1. не продолжай рискованные изменения;
2. выполни безопасный аудит;
3. создай `docs/reports/MODEL-ESCALATION-REPORT.md`;
4. укажи причины, риски и продолжение задачи;
5. сообщи владельцу: `Требуется более сильная модель. Перезапустите Codex и повторите задачу.`

Не утверждай, что модель переключена автоматически.
```

Не добавляй длинную классификацию LEVEL 1/2/3.

## 2. Определить актуальный roadmap

Приоритет поиска:

1. существующий roadmap с кодом, соответствующим текущей версии;
2. самый новый `SL-ROADMAP-*`;
3. если roadmap отсутствует — создать новый.

Имя нового документа:

`docs/roadmap/SL-ROADMAP-v<CURRENT_PLATFORM_VERSION>.md`

Пример: если подтверждена версия `v0.6.1`, код документа:

`SL-ROADMAP-v0.6.1`

## 3. Обновить roadmap по факту

Roadmap должен отражать только подтвержденное состояние.

Обязательно проверить и отметить:

### Платформа
- backend;
- frontend;
- PostgreSQL;
- Docker Compose;
- Alembic;
- проверки проекта;
- тесты;
- документацию;
- checkpoint/recovery.

### Лиды
- backend-модель;
- список лидов;
- карточка лида;
- контакты;
- коммерческое ядро;
- профиль клиента;
- загрузка списка из backend;
- изменение статуса;
- Kanban persistence;
- настраиваемые этапы;
- завершение успешно/отказ;
- транзакционная конвертация;
- создание сделки или заказа;
- аналитика;
- demo/in-memory остатки.

### Сделки и заказы
- фактические модели и API;
- frontend;
- связи с лидом;
- сохранение;
- товарные позиции;
- статусы;
- производство.

### Остальные модули
- продукция и материалы;
- модели и лекала;
- дизайн;
- производство;
- склад;
- закупки;
- логистика;
- финансы;
- системное ядро;
- интеграции;
- аналитика.

Для статусов используй:

- `[x]` — подтверждено кодом и проверками;
- `[~]` — частично, frontend-only, demo-only или неполный backend-контур;
- `[ ]` — отсутствует;
- `[!]` — сломано или требует стабилизации.

## 4. Обновить короткие коды

В roadmap укажи:

- `Platform version`;
- `Roadmap code`;
- `Structure code`, если актуальная структура существует;
- предыдущие коды как исторические ссылки.

Не переименовывай существующие документы без необходимости. При переходе на новый код создай новый документ и оставь старый как исторический.

## 5. Добавить раздел о модели в roadmap

В разделе правил разработки добавь короткий блок:

```md
## Когда просить переключение модели

Codex должен остановиться и попросить владельца вручную выбрать более сильную модель до реализации архитектуры, безопасности, RBAC, сложных миграций, изоляции организаций, финансов, склада, себестоимости или изменений нескольких бизнес-модулей.

Для CRUD, документации, UI, тестов и реализации уже утвержденной архитектуры используется экономичная модель.

После двух неудачных обоснованных попыток исправления одной ошибки Codex прекращает догадки и просит переключить модель.
```

## 6. Индекс документации

Создай или обнови:

`docs/README.md`

Добавь ссылки на:

- актуальный roadmap;
- актуальную структуру;
- последний checkpoint;
- `AGENTS.md`;
- предыдущий roadmap как исторический, если он существует.

## 7. Отчет

Создай:

`docs/reports/ROADMAP-SYNC-REPORT.md`

Укажи:

- фактически определенную версию платформы;
- источник версии;
- найденные roadmap/structure документы;
- измененные статусы;
- добавленное правило переключения модели;
- измененные файлы;
- выполненные проверки;
- расхождения между GitHub и локальной копией;
- следующую рекомендуемую задачу.

## Ограничения

- Не изменять бизнес-код.
- Не изменять API, модели и миграции.
- Не отмечать пункт `[x]` по названию коммита без проверки кода.
- Не понижать версию документации.
- Не считать GitHub `v0.5.0` текущей версией, если локальная копия новее.
- Не создавать несколько конкурирующих roadmap.
- Изменять только документацию.

## Проверки

Запусти:

- `python scripts/check_project.py`
- `docker compose config --quiet`
- `git diff --check`

Если доступны зависимости frontend:

- `cd frontend`
- `npm run lint`
- `npm run build`

Проверь:

- `AGENTS.md` находится в корне;
- правило model escalation короткое;
- roadmap соответствует фактической версии;
- ссылки в `docs/README.md` работают;
- `git diff` содержит только документацию.

## Критерии готовности

- Codex при старте читает понятное правило переключения модели;
- актуальный roadmap соответствует локальному проекту;
- короткие коды соответствуют версии платформы;
- GitHub/local расхождения отражены в отчете;
- документация готова к коммиту.
