## Текущее состояние

Проект преобразован в monorepo:

```text

frontend/

```

Backend находится в `backend/`.

Frontend находится в `frontend/`.

Скрипт `scripts/check_project.py` уже адаптирован к новой структуре.

Работай как разработчик. Сначала изучи фактическое состояние файлов, уже используемые UI-компоненты, маршрутизацию и стили. Не создавай второй конкурирующий UI-kit и не переписывай существующий layout.

# Задача

Реализовать демонстрационный frontend-раздел:

```text
Продажи
```

со страницами:

```text
/sales/leads
/sales/deals
/sales/orders
/sales/tasks
/sales/clients
```

## Цель

Создать целостный рабочий интерфейс CRM-раздела, который позже будет подключён к backend API.

На этом этапе использовать локальные типизированные demo-данные.

Backend API не изменять.

# 4. Общий заголовок

Каждая страница должна содержать:

* название страницы;
* краткое описание;
* основное действие;
* строку поиска;
* фильтры;
* счётчики по статусам, где это уместно.

Пример действий:

```text
Добавить лид
Создать сделку
Создать заказ
Добавить задачу
Добавить клиента
```

Кнопки пока могут открывать демонстрационное модальное окно или быть визуально активными без сохранения на backend.

Не оставлять кнопки без понятной реакции. Минимум — открыть dialog с сообщением, что подключение формы будет следующим этапом.

# 5. Универсальный Kanban

Создать переиспользуемые компоненты в:

```text
frontend/components/kanban/
```

Минимальный состав:

```text
kanban-board.tsx
kanban-column.tsx
kanban-card.tsx
kanban-types.ts
```

Названия можно адаптировать под существующий стиль проекта.

Компоненты должны поддерживать:

* набор колонок;
* заголовок колонки;
* количество карточек;
* сумму или дополнительную метрику колонки;
* карточки разных сущностей;
* badge приоритета или статуса;
* исполнителя;
* дату следующего действия;
* пустое состояние;
* горизонтальную прокрутку на небольших экранах.

На этом этапе drag-and-drop не обязателен.

Не добавлять тяжёлую DnD-библиотеку только ради визуальной демонстрации.

Архитектура компонента должна позволять добавить drag-and-drop позже.

# 13. Клиентские компоненты

Не добавлять `"use client"` на уровне всей страницы без необходимости.

Статические данные и layout по возможности оставить серверными компонентами.

Клиентскими делать только компоненты, которым действительно нужны:

* состояние поиска;
* фильтры;
* dialog;
* интерактивная сортировка.

Не переносить весь раздел в один огромный клиентский компонент.

# 14. Адаптивность

Интерфейс должен корректно работать:

* на desktop;
* на планшете;
* на узком экране.

Для Kanban на узком экране использовать горизонтальную прокрутку.

Не сжимать колонки до нечитаемой ширины.

Таблица клиентов может использовать горизонтальную прокрутку или существующий адаптивный table pattern проекта.
