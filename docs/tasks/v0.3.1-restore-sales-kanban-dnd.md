В Draft PR #1 обнаружена регрессия: после реализации нового Sales UI пропало существующее перетаскивание карточек Kanban.

Работай в текущей ветке:

feature/sales-workspace-ui

Не создавай новый PR. Исправление должно быть добавлено новым коммитом в существующий PR #1.

## Причина

В `frontend/components/kanban/kanban-board.tsx` была удалена существующая реализация на `@dnd-kit/core`, `@dnd-kit/sortable` и `@dnd-kit/utilities`, после чего Kanban стал статическим.

Нужно вернуть drag-and-drop, сохранив новую архитектуру, типы, фильтры, метрики и внешний вид Sales UI.

## Требуемое поведение

На страницах:

* `/sales/leads`
* `/sales/deals`
* `/sales/orders`
* `/sales/tasks`

карточки должны:

1. перетаскиваться между колонками;
2. менять порядок внутри одной колонки;
3. корректно перемещаться в пустую колонку;
4. визуально показывать перетаскиваемую карточку;
5. поддерживать мышь;
6. поддерживать touch/pointer;
7. по возможности сохранить keyboard sensor из прежней реализации;
8. не запускать drag при клике по кнопкам внутри карточки.

Страница клиентов drag-and-drop не требует.

## Архитектурные требования

Не возвращай старый монолитный файл целиком.

Сохрани текущее разделение:

* `kanban-board.tsx`
* `kanban-column.tsx`
* `kanban-card.tsx`
* `kanban-page.tsx`
* `kanban-types.ts`

Распредели ответственность:

* `KanbanBoard` — `DndContext`, состояние карточек, обработчики drag start/over/end/cancel;
* `KanbanColumn` — droppable-зона и `SortableContext`;
* `KanbanCard` — `useSortable`, drag handle и визуальное состояние;
* `kanban-types.ts` — общие строгие типы.

Не использовать `any`.

## Состояние данных

Сейчас страницы строятся из demo-данных и фильтров.

После перемещения карточки:

* новое положение должно сохраняться в React state до перезагрузки страницы;
* карточка должна получить статус целевой колонки;
* счётчики колонок должны обновиться;
* суммы и метрики колонок должны обновиться;
* поиск и фильтры не должны сбрасывать уже выполненные перемещения;
* очистка поискового запроса должна показывать карточку в новой колонке, а не возвращать исходное положение.

Не изменяй импортированные demo-данные напрямую.

Создавай локальную копию состояния.

## Типы

Добавь в общий тип карточки или адаптер явную возможность обновить статус.

Предпочтительный интерфейс:

```ts
onMove?: (cardId: string, targetColumnId: string, targetIndex: number) => void;
```

или эквивалентная строго типизированная модель.

Статус карточки после перемещения должен соответствовать union type конкретной сущности:

* `LeadStatus`;
* `DealStatus`;
* `OrderStatus`;
* `SalesTaskStatus`.

Не приводить значения через небезопасное `as any`.

## Важный сценарий с фильтрами

Drag-and-drop должен выполняться над полным локальным набором данных, а не только над отфильтрованным массивом.

Например:

1. пользователь применил фильтр;
2. перетащил видимую карточку;
3. сбросил фильтр;
4. остальные скрытые карточки не исчезли и не изменили порядок произвольно.

## Визуальное поведение

Вернуть:

* `cursor-grab`;
* `active:cursor-grabbing`;
* drag handle;
* подсветку droppable-колонки;
* `DragOverlay`;
* уменьшенную прозрачность исходной карточки при перетаскивании.

Использовать текущий дизайн карточек, а не старую разметку целиком.
