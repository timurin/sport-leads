## Предполагаемая причина

Сейчас `onDragOver` вызывает:

```ts
setColumns(...)
onColumnsChange(...)
```

`onColumnsChange` синхронно вызывает `setMetricColumns` в родительском `KanbanPage`.

При этом `moveKanbanCard()` всегда создаёт новый массив колонок, даже если карточка фактически остаётся в том же положении.

Во время активного `DndContext` это может вызвать повторяющуюся цепочку:

```text
dragOver
→ setColumns
→ onColumnsChange
→ parent setState
→ render
→ новая collision evaluation
→ dragOver
→ setState
```

Не ограничивайся обработкой исключения или перезагрузкой страницы. Устрани причину повторных обновлений.

## Требуемая архитектура

### 1. Не обновлять родителя на каждом `dragOver`

Во время движения карточки:

* обновляй только локальное состояние `KanbanBoard`;
* не вызывай `onColumnsChange` на каждом событии `onDragOver`;
* не обновляй метрики родителя десятки раз на одно движение указателя.

Вызови `onColumnsChange` один раз:

* после успешного `onDragEnd`;
* либо после окончательного подтверждения нового состояния.

При `onDragCancel` восстанови исходное состояние без публикации промежуточного результата родителю.

Метрики допустимо обновлять после отпускания карточки, а не во время движения.

### 2. Использовать функциональное обновление состояния

Избегай вычислений на потенциально устаревшем замыкании `columns`.

Используй подход:

```ts
setColumns((currentColumns) => {
  const nextColumns = moveKanbanCard(currentColumns, move);

  if (nextColumns === currentColumns) {
    return currentColumns;
  }

  return nextColumns;
});
```

или эквивалентную строго типизированную реализацию.

### 3. Сделать перемещение идемпотентным

`moveKanbanCard()` не должен возвращать новый массив, если:

* исходная и целевая колонка совпадают;
* карточка уже находится в требуемой позиции;
* `targetIndex` не изменяет порядок;
* событие полностью повторяет предыдущее перемещение;
* source/target/card не найдены.

В таких случаях возвращай исходный объект `columns`, а не его клон.

Это позволит React пропустить ненужный render.

### 4. Защита от повторного `dragOver`

Добавь `useRef` с последним применённым перемещением или эквивалентный механизм.

Не применяй повторно move, если совпадают:

```text
cardId
targetColumnId
targetIndex
```

Сбрасывай guard:

* в `handleDragStart`;
* после реального нового перемещения;
* в `handleDragEnd`;
* в `handleDragCancel`.

### 5. Не выполнять один move дважды

Сейчас перенос между колонками может применяться в `handleDragOver`, а затем повторно в `handleDragEnd`.

После перемещения между колонками `handleDragEnd` не должен повторно вставлять карточку в ту же позицию.

На `dragEnd`:

* для межколоночного переноса только зафиксируй уже полученное состояние;
* для перестановки внутри одной колонки выполни один окончательный move;
* убедись, что одна операция пользователя приводит максимум к одному фактическому изменению порядка.

### 6. Сохрани существующее поведение

После исправления должны работать:

* перенос между колонками;
* сортировка внутри колонки;
* перенос в пустую колонку;
* drag overlay;
* мышь;
* pointer/touch;
* keyboard sensor;
* фильтры;
* поиск;
* сохранение положения до перезагрузки;
* обновление метрик после завершения переноса;
* возврат исходного состояния при cancel.
