# Задача v0.3.3 — создать стабильную контрольную точку проекта

## Цель

Создать первую официальную стабильную контрольную точку проекта `sport-leads`, к которой можно безопасно вернуться при нарушении логики, структуры файлов или истории в последующих ветках.

Контрольная версия:

```text
v0.3.0-demo
```

Работа должна включать:

* завершение Draft PR #1;
* объединение Sales UI с родительской ERP-веткой;
* интеграцию результата в `main`;
* полную проверку проекта;
* создание аннотированного Git-тега;
* создание резервной stable-ветки;
* создание GitHub Release;
* подготовку инструкции восстановления.

---

## Исходное состояние

Репозиторий:

```text
timurin/sport-leads
```

Текущая функциональная ветка:

```text
feature/sales-workspace-ui
```

Текущий Draft PR:

```text
#1 Add sales workspace demo interface
```

Base PR #1:

```text
feature/erp-ui-prototype
```

В PR должны находиться как минимум следующие коммиты:

```text
9a48efa — Add sales workspace demo interface
92ede1d — Restore drag and drop in sales kanban
```

Также должен присутствовать последний коммит с исправлением ошибки:

```text
Maximum update depth exceeded
```

Перед началом найди его фактический SHA и включи в итоговый отчёт.

---

## Критические ограничения

Не начинать новую функциональную разработку.

Не изменять бизнес-логику Sales UI, если это не требуется для устранения ошибки сборки или конфликта интеграции.

Не выполнять force push.

Не переписывать существующую историю через:

```text
git rebase -i
git reset --hard
git push --force
```

Не удалять существующие ветки.

Не удалять теги.

Не включать в коммиты локальные служебные и неотслеживаемые файлы, если они не относятся к контрольной версии:

```text
AGENTS.md
docs/tasks/*
```

Файл текущей задачи также не должен автоматически попадать в интеграционные коммиты, если он ранее не отслеживался Git.

Перед любым merge обязательно проверить рабочее дерево.

При наличии непредвиденных изменений остановить merge и сначала определить их происхождение.

# Этап 1. Проверка исходного состояния

Перейти в корень проекта:

```powershell
cd D:\Projects\sport-leads
```

Выполнить:

```powershell
git status
git branch --show-current
git branch -vv
git log --oneline --decorate -10
git remote -v
git fetch origin --prune --tags
```

Проверить:

1. ветка `feature/sales-workspace-ui` существует локально и в `origin`;
2. локальная и удалённая ветки совпадают;
3. нет незакоммиченных изменений в отслеживаемых файлах;
4. PR #1 открыт;
5. PR #1 остаётся Draft до завершения проверки;
6. base PR #1 — `feature/erp-ui-prototype`;
7. head PR #1 — `feature/sales-workspace-ui`;
8. PR mergeable;
9. в PR присутствует исправление recursive React updates;
10. GitHub Actions, если настроены, не имеют незавершённых или упавших обязательных проверок.

Не считать неотслеживаемые `AGENTS.md` и `docs/` блокирующими, если они не входят в Git и не затрагивают проектные файлы.

# Этап 3. Проверка Kanban перед интеграцией

Запустить frontend:

```powershell
cd D:\Projects\sport-leads\frontend
npm run dev
```

Если браузер доступен, вручную проверить маршруты:

```text
/sales/leads
/sales/deals
/sales/orders
/sales/tasks
/sales/clients
```

Обязательно выполнить:

1. перенос карточки между колонками;
2. перестановку внутри одной колонки;
3. перенос в пустую колонку;
4. перенос при активном фильтре;
5. сброс фильтра после переноса;
6. быстрые последовательные перемещения;
7. отмену drag через Escape;
8. не менее 20 DnD-операций без обновления страницы;
9. отсутствие в console ошибок и warning;
10. отсутствие повторения:

```text
Maximum update depth exceeded
```

Если браузер недоступен, явно указать это в отчёте. Не выдавать автоматическую сборку за полноценную ручную проверку.

Поскольку пользователь уже сообщил, что ручная проверка прошла успешно, зафиксировать это в Release Notes как пользовательскую приёмку, но всё равно выполнить доступные автоматические проверки.

# Этап 4. Завершение PR #1

После успешных проверок:

1. убедиться, что PR #1 содержит только изменения Sales UI и DnD-исправления относительно `feature/erp-ui-prototype`;
2. убедиться, что нет конфликтов;
3. перевести PR #1 из Draft в Ready for review;
4. повторно проверить mergeability;
5. выполнить merge PR #1 в:

```text
feature/erp-ui-prototype
```

Предпочтительный метод:

```text
squash
```

Однако перед merge определить, что лучше сохраняет историю проекта.

Если отдельные коммиты:

```text
Add sales workspace demo interface
Restore drag and drop in sales kanban
Fix recursive updates in kanban drag and drop
```

важны как диагностические контрольные точки, использовать обычный merge.

Предпочтение для этой контрольной версии:

```text
merge commit
```

чтобы сохранить последовательность исправлений.

Не удалять head-ветку автоматически после merge.

После merge проверить:

```powershell
git fetch origin
git switch feature/erp-ui-prototype
git pull --ff-only
git log --oneline --decorate -10
```

# Этап 7. Создание контрольного Git-тега

После успешной проверки `main` определить точный SHA:

```powershell
git rev-parse HEAD
```

Убедиться, что тег отсутствует:

```powershell
git tag --list v0.3.0-demo
git ls-remote --tags origin v0.3.0-demo
```

Если тег уже существует, не перезаписывать его. Остановиться и сообщить об этом.

Создать аннотированный тег:

```powershell
git tag -a v0.3.0-demo -m "Stable demo checkpoint: ERP and Sales workspace"
git push origin v0.3.0-demo
```

Проверить:

```powershell
git show v0.3.0-demo --no-patch
git ls-remote --tags origin v0.3.0-demo
```

Тег должен указывать на проверенный HEAD ветки `main`.

---

# Этап 8. Создание резервной stable-ветки

Убедиться, что ветка отсутствует:

```powershell
git branch --list stable/v0.3.0-demo
git ls-remote --heads origin stable/v0.3.0-demo
```

Создать ветку точно от тега:

```powershell
git switch -c stable/v0.3.0-demo v0.3.0-demo
git push -u origin stable/v0.3.0-demo
```

Проверить совпадение SHA:

```powershell
git rev-parse stable/v0.3.0-demo
git rev-parse v0.3.0-demo
git rev-parse main
```

Все три значения в момент создания контрольной точки должны совпадать.

Вернуться в `main`:

```powershell
git switch main
```

Ветка `stable/v0.3.0-demo` считается защищённой контрольной веткой.

Не добавлять в неё новые функциональные коммиты.
