# Задача v0.4.1 — Dashboard раздела «Продажи»

## Цель

Разработать полноценную главную страницу проекта и dashboard продаж на основе существующих типизированных demo-данных.

Dashboard должен объединять ключевые показатели:

* лидов;
* сделок;
* заказов;
* клиентов;
* задач;
* конверсии;
* суммы продаж;
* источников привлечения;
* динамики за выбранный период.

На этом этапе backend и база данных не подключаются. Архитектура frontend должна быть подготовлена к последующей замене demo-данных на API.

# 2. Общий состав Dashboard

Dashboard должен содержать следующие блоки.

## 2.1. Заголовок

* название: `Dashboard продаж`;
* краткое описание выбранного периода;
* кнопка обновления demo-данных не требуется;
* отображение активных фильтров;
* кнопка сброса фильтров.

## 2.2. Основные KPI

Показать не менее восьми показателей:

1. новые лиды;
2. квалифицированные лиды;
3. активные сделки;
4. сумма активных сделок;
5. заказы;
6. сумма заказов;
7. конверсия лид → сделка;
8. конверсия сделка → заказ.

Дополнительно:

* новые клиенты;
* просроченные задачи;
* средний чек;
* прогноз выручки.

Каждый KPI должен иметь:

* название;
* значение;
* вспомогательное пояснение;
* изменение относительно предыдущего аналогичного периода;
* положительное, отрицательное или нейтральное состояние.

Не использовать случайные значения во время render.

Все данные должны быть детерминированы.

# 3. Фильтры

Dashboard должен поддерживать единый типизированный объект фильтров.

Обязательные фильтры:

## 3.1. Период

Предустановленные варианты:

```text
Сегодня
Вчера
7 дней
30 дней
Текущий месяц
Предыдущий месяц
Текущий квартал
Текущий год
Произвольный период
```

Для произвольного периода:

* дата начала;
* дата окончания;
* проверка корректности диапазона.

## 3.2. Источник

Минимальные значения из demo-данных:

```text
Сайт
Рекомендация
VK
Телефон
Email
Ручной ввод
```

## 3.3. Ответственный

Использовать сотрудников из существующих demo-данных.

## 3.4. Подразделение

Минимально:

```text
Продажи
Производство
Дизайн
Руководство
```

## 3.5. Статус

Фильтр должен уметь учитывать основные статусы:

* лидов;
* сделок;
* заказов;
* задач.

Не объединять несовместимые union types в небезопасный `string`.

Использовать отдельные типы или discriminated union.

## 3.6. Клиент

Поиск или select по клиенту.

## 3.7. Сброс

Кнопка сброса должна возвращать dashboard к периоду `30 дней`.

# 4. Статистические блоки

## 4.1. Воронка продаж

Показать этапы:

```text
Новые лиды
Квалифицированы
Сделки
Согласование
Заказы
Завершено
```

Для каждого этапа:

* количество;
* процент перехода с предыдущего этапа;
* визуальная длина;
* сумма, где применимо.

## 4.2. Динамика

Показать изменение по времени:

* лиды;
* сделки;
* заказы;
* сумма заказов.

Допускается использование существующей chart-библиотеки, если она уже есть.

Если chart-библиотеки нет:

* не добавлять тяжёлую зависимость без необходимости;
* реализовать простой SVG или CSS-график;
* вынести график в переиспользуемый компонент.

График должен корректно реагировать на выбранный период.

## 4.3. Источники лидов

Показать таблицу или диаграмму:

* источник;
* лиды;
* сделки;
* заказы;
* конверсия;
* сумма заказов.

## 4.4. Статусы сделок

Показать распределение сделок по статусам.

## 4.5. Заказы

Показать:

* новые;
* в работе;
* готовые;
* завершённые;
* просроченные;
* общую сумму.

## 4.6. Задачи

Показать:

* на сегодня;
* просроченные;
* ближайшие;
* завершённые;
* распределение по ответственным.

## 4.7. Последние события

Добавить demo-ленту последних событий:

* создан лид;
* изменён статус;
* создана сделка;
* создан заказ;
* добавлен комментарий;
* назначена задача;
* выполнен звонок.

Этот компонент позднее будет использоваться в карточке CRM-объекта.

# 5. Данные

Расширить типизированные demo-данные так, чтобы каждый объект имел даты, необходимые для Dashboard:

```text
createdAt
updatedAt
qualifiedAt
wonAt
orderedAt
completedAt
dueAt
```

Добавлять только применимые поля.

Не использовать все даты одновременно для каждой сущности.

Для источников добавить строгий тип, например:

```ts
export type SalesSource =
  | "website"
  | "referral"
  | "vk"
  | "phone"
  | "email"
  | "manual";
```

Добавить отображаемые labels отдельно.

Не использовать `any`.

Не мутировать импортированные константы.

# 6. Архитектура вычислений

Расчёт Dashboard должен быть отделён от React-компонентов.

Предлагаемая структура:

```text
frontend/
  app/(workspace)/sales/dashboard/page.tsx
  components/dashboard/
    dashboard-filters.tsx
    dashboard-kpi-grid.tsx
    sales-funnel.tsx
    sales-dynamics-chart.tsx
    lead-sources-table.tsx
    sales-status-summary.tsx
    orders-summary.tsx
    tasks-summary.tsx
    recent-activity.tsx
  lib/dashboard/
    date-range.ts
    sales-dashboard.ts
    sales-dashboard-types.ts
```

Чистые функции должны:

1. применять фильтры;
2. строить период;
3. рассчитывать предыдущий период;
4. считать KPI;
5. рассчитывать конверсии;
6. группировать по дням, неделям или месяцам;
7. группировать по источникам;
8. рассчитывать funnel;
9. формировать recent activity.

React-компоненты не должны содержать сложную бизнес-логику расчётов.

---

# 7. Поведение периода

Группировка динамики:

* до 31 дня — по дням;
* от 32 до 180 дней — по неделям;
* больше 180 дней — по месяцам.

Для периода без данных:

* показывать нули;
* не падать;
* выводить понятное пустое состояние.

Все даты интерпретировать последовательно.

Не смешивать локальное время и UTC без явной стратегии.

Для demo-версии допустимо хранить ISO-строки и интерпретировать их в локальном времени приложения.

---

# 8. Адаптивность

Dashboard должен корректно работать:

* desktop;
* tablet;
* узкое окно браузера.

KPI:

* 4 колонки на широком экране;
* 2 на среднем;
* 1 на узком.

Таблицы должны иметь горизонтальную прокрутку.

Графики не должны выходить за контейнер.

# 9. Переиспользование

Не дублировать:

* кнопки;
* карточки;
* select;
* пустые состояния;
* форматирование валюты;
* форматирование процентов;
* форматирование дат.

Использовать существующую UI-систему проекта.

Допускается создание общих компонентов:

```text
MetricCard
EmptyState
FilterField
DateRangeFilter
SectionCard
```

Не выполнять масштабный redesign существующего workspace.

---

# 10. Подготовка к backend

Dashboard должен получать данные через отдельный frontend-слой.

Не импортировать demo-массивы непосредственно во всех компонентах.

Создать функцию или provider, например:

```ts
getSalesDashboardDemoData()
```

или:

```ts
createSalesDashboardSnapshot(...)
```

Чтобы в следующей версии источник можно было заменить на API.

Не создавать backend endpoints в этой задаче.
