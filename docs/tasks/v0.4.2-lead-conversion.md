# v0.4 — Конверсия лида в заказ покупателя

## Цель

Реализовать бизнес-механику завершения лида двумя способами:

1. Успешно — создать заказ покупателя.
2. Отказ — завершить лид с обязательным указанием причины.

После успешной конверсии вся дальнейшая работа должна вестись в заказе покупателя. Исходный лид сохраняется и остаётся связанным с созданным заказом.

## Основной процесс

Источник обращения → Лид → один из результатов:

* Успешно → Заказ покупателя.
* Отказ → Причина отказа.

На этом этапе не использовать отдельную обязательную сущность «Сделка». Заказ покупателя является успешным результатом обработки лида.

---

## 1. Модель состояния лида

Разделить рабочую стадию и результат завершения.

### Рабочие стадии

* `new` — новый;
* `contact` — первичный контакт;
* `qualification` — квалификация;
* `proposal` — подготовка предложения;
* `waiting` — ожидание решения;
* `completed` — завершён.

### Результат завершения

* `converted` — создан заказ покупателя;
* `rejected` — отказ.

Для незавершённого лида результат должен быть пустым.

Недопустимое состояние:

* `status = completed`;
* `result = null`.

---

## 2. Сущность Lead

Добавить или подготовить следующие поля:

* `status`;
* `result`;
* `completed_at`;
* `completed_by_id`;
* `converted_order_id`;
* `rejection_reason_id`;
* `rejection_comment`.

Правила:

* незавершённый лид не имеет результата;
* успешно завершённый лид обязательно связан с заказом;
* отказной лид обязательно имеет причину отказа;
* один лид нельзя повторно конвертировать;
* завершение должно выполняться транзакционно.

---

## 3. Справочник причин отказа

Создать сущность `LeadRejectionReason`.

Поля:

* `id`;
* `code`;
* `name`;
* `category`;
* `requires_comment`;
* `is_active`;
* `sort_order`;
* `created_at`;
* `updated_at`.

Начальные причины:

### Клиент

* не выходит на связь;
* передумал;
* нет бюджета;
* отложил заказ;
* выбрал конкурента.

### Цена и условия

* высокая цена;
* не устроили сроки;
* не устроили условия оплаты;
* не устроила доставка.

### Производство

* не производим нужный товар;
* недостаточный тираж;
* невозможный срок;
* нет технической возможности.

### Качество лида

* дубликат;
* спам;
* ошибочное обращение;
* нецелевой клиент;
* другое.

Для причин «Другое» и «Выбрал конкурента» комментарий должен быть обязательным.

Причины нельзя физически удалять, если они уже использовались. Для отключения использовать `is_active`.

---

## 4. Заказ покупателя

Создать или расширить сущность `SalesOrder`.

Минимальные поля:

* `id`;
* `number`;
* `lead_id`;
* `client_id`;
* `status`;
* `responsible_id`;
* `title`;
* `description`;
* `product_category`;
* `sport`;
* `quantity`;
* `amount`;
* `desired_date`;
* `source`;
* `created_at`;
* `updated_at`.

Связь:

* один лид создаёт не более одного первичного заказа;
* заказ хранит ссылку на исходный лид;
* лид хранит ссылку на созданный заказ.

На первом этапе заказ может содержать предварительные данные без товарных позиций и производственной спецификации.

---

## 5. Успешная конверсия

Создать backend-операцию:

`POST /leads/{lead_id}/convert`

Операция должна:

1. Заблокировать или повторно проверить состояние лида.
2. Убедиться, что лид ещё не завершён.
3. Найти существующего клиента или создать нового.
4. Создать заказ покупателя.
5. Перенести данные из лида в заказ.
6. Связать заказ с лидом.
7. Установить:

   * `lead.status = completed`;
   * `lead.result = converted`;
   * `lead.converted_order_id`;
   * `lead.completed_at`;
   * `lead.completed_by_id`.
8. Создать запись истории.
9. Выполнить всё в одной транзакции.

При любой ошибке заказ не создаётся, а лид остаётся в исходном состоянии.

Повторный запрос не должен создавать второй заказ.

---

## 6. Данные, переносимые в заказ

Из лида автоматически переносить:

* клиента или контакт;
* название компании;
* телефон;
* email;
* город;
* источник;
* ответственного;
* вид спорта;
* категорию продукции;
* описание потребности;
* предварительное количество;
* предварительную сумму;
* желаемую дату.

Пользователь должен иметь возможность скорректировать данные перед созданием заказа.

---

## 7. Завершение отказом

Создать backend-операцию:

`POST /leads/{lead_id}/reject`

Входные данные:

* `rejection_reason_id`;
* `comment`.

Операция должна:

1. Проверить, что лид не завершён.
2. Проверить существование и активность причины.
3. Проверить обязательность комментария.
4. Установить:

   * `lead.status = completed`;
   * `lead.result = rejected`;
   * `lead.rejection_reason_id`;
   * `lead.rejection_comment`;
   * `lead.completed_at`;
   * `lead.completed_by_id`.
5. Записать событие в историю.
6. Закрыть или отменить незавершённые задачи лида.

Удалять отказные лиды нельзя.

---

## 8. История событий

Добавить события:

* `lead_created`;
* `lead_status_changed`;
* `lead_converted`;
* `lead_rejected`;
* `order_created`;
* `comment_added`;
* `task_created`;
* `task_completed`.

История исходного лида должна быть доступна из заказа покупателя.

На первом этапе допустимо отображать объединённую историю:

* события заказа;
* события связанного лида.

Не копировать историю физически без необходимости.

---

## 9. Интерфейс лида

В карточке лида добавить кнопку:

`Завершить лид`

После нажатия показать выбор:

* `Успешно — создать заказ покупателя`;
* `Отказ — указать причину`.

### Успешное завершение

Открыть форму предварительного заказа с автоматически заполненными данными лида.

После сохранения:

* закрыть модальное окно;
* обновить карточку лида;
* показать номер созданного заказа;
* добавить кнопку `Открыть заказ`.

### Отказ

Показать:

* выпадающий список причин;
* поле комментария;
* обязательность комментария в зависимости от причины;
* кнопку подтверждения.

После завершения карточка должна показывать:

* результат;
* причину;
* комментарий;
* дату;
* пользователя, завершившего лид.

## 10. Список и Kanban лидов

Завершённые лиды не должны оставаться среди активных стадий.

Добавить возможность просмотра:

* активных лидов;
* успешно завершённых;
* отказных;
* всех лидов.

Карточка завершённого лида должна иметь визуальный признак результата.

Drag-and-drop должен работать только для активных лидов.

## 11. Dashboard

После реализации связей изменить расчёт конверсии.

Не использовать формулу:

`общее число заказов / общее число лидов`

Использовать реальные результаты лидов.

Добавить показатели:

* всего создано лидов;
* завершено лидов;
* успешно конвертировано;
* отказов;
* конверсия всех входящих лидов;
* конверсия завершённых лидов;
* причины отказов;
* конверсия по источникам;
* среднее время от создания лида до завершения.

Источник заказа должен определяться через исходный лид.

## 13. Тесты

Обязательно проверить:

1. Успешная конверсия создаёт один заказ.
2. Лид получает результат `converted`.
3. Заказ связан с исходным лидом.
4. Повторная конверсия не создаёт дубликат.
5. При ошибке создания заказа лид не завершается.
6. Отказ без причины запрещён.
7. Обязательный комментарий проверяется.
8. Неактивную причину нельзя выбрать для нового отказа.
9. Успешный лид нельзя завершить отказом.
10. Отказной лид нельзя конвертировать.
11. Dashboard использует реальные связи.
12. Старые страницы Sales продолжают открываться.
13. Drag-and-drop активных лидов не ломается.

---

## 14. Ограничения версии

На этом этапе не реализовывать:

* товарные позиции заказа;
* материалы;
* спецификации моделей;
* расчёт себестоимости;
* производство;
* оплату;
* склад;
* интеграцию с 1С;
* автоматическое объединение дублей клиентов;
* сложные права доступа.

Сосредоточиться только на завершении лида и создании первичного заказа покупателя.
