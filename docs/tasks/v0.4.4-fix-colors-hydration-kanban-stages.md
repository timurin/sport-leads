# v0.4.4 — Исправление цветов, hydration warning и настройка стадий Kanban

## Контекст

После выполнения задачи v0.4.3 при ручной проверке обнаружены следующие проблемы.

### Проблема 1. Светлый текст в карточке лида

В диалоге карточки лида почти не читаются значения:

- источник;
- ответственный;
- оценка;
- следующий контакт;
- другие значения внутри информационных блоков.

На скриншоте подписи отображаются светло-синим цветом, а значения практически белым цветом на светлом фоне.

Предыдущее исправление затронуло преимущественно input, select и textarea, но не исправило весь текст внутри диалога.

### Проблема 2. Hydration mismatch

В браузерной консоли появляется предупреждение React/Next.js:

A tree hydrated but some attributes of the server rendered HTML didn't match the client properties. This won't be patched up.

Возможные причины, перечисленные React:

- различия между серверным и клиентским рендерингом;
- использование typeof window во время рендера;
- Date.now или Math.random;
- локализованное форматирование дат;
- изменение HTML внешними данными;
- некорректная вложенность HTML;
- данные из localStorage, прочитанные до завершения hydration.

Необходимо найти реальную причину предупреждения в текущем проекте. Не скрывать предупреждение через suppressHydrationWarning без точного понимания причины.

### Новая возможность

Добавить возможность редактировать стадии Kanban лидов, чтобы пользователь мог настроить рабочий процесс под себя.

## 1. Исправление цветов карточки лида

Проверить минимум файл:

frontend/components/sales/lead-completion-dialog.tsx

frontend/app/globals.css

### Требования

Корневой контейнер диалога должен иметь явный основной цвет текста:

text-slate-900

Информационный блок с данными лида должен использовать:

- подписи dt: text-slate-500;
- значения dd: text-slate-900;
- основной текст: text-slate-900;
- дополнительный текст: text-slate-600;
- даты и вторичные данные: не светлее text-slate-500.

Не оставлять важные значения без явного класса цвета, если они могут наследовать белый или слишком светлый цвет из глобальных стилей.

Проверить и исправить:

- название лида;
- контакт;
- город;
- вид спорта;
- источник;
- ответственного;
- оценку;
- следующий контакт;
- причину отказа;
- комментарий отказа;
- номер созданного заказа;
- поля формы создания заказа;
- поля формы отказа;
- текст выбранного значения в select;
- текст внутри textarea;
- текст внутри input.

На белом и светло-сером фоне основной текст должен иметь достаточный контраст.

Не менять размеры, отступы и общую композицию диалога без необходимости.

## 2. Диагностика hydration mismatch

Найти точный источник предупреждения.

Проверить минимум:

- frontend/app/layout.tsx;
- frontend/app/(workspace)/layout.tsx;
- frontend/components/sales/lead-workspace.tsx;
- frontend/components/sales/lead-completion-dialog.tsx;
- frontend/components/kanban/kanban-board.tsx;
- frontend/lib/demo-data/sales.ts;
- компоненты навигации и layout;
- чтение localStorage;
- формирование дат;
- генерацию идентификаторов;
- использование Date.now;
- использование Math.random;
- использование window и document;
- условный серверный и клиентский рендеринг;
- некорректную вложенность HTML;
- различия className или style между сервером и клиентом.

### Требования

Серверный и первый клиентский рендер должны создавать одинаковую HTML-структуру и одинаковые атрибуты.

Если используются данные localStorage:

- не читать localStorage непосредственно при серверном рендеринге;
- первый клиентский рендер должен совпадать с серверным;
- загрузить сохранённые настройки после mount;
- использовать useEffect или другой hydration-safe подход;
- при необходимости отображать стабильные значения по умолчанию до загрузки сохранённых настроек.

Если используется форматирование дат:

- не полагаться на различающуюся серверную и клиентскую локаль;
- использовать заранее подготовленную строку или одинаковый locale;
- обеспечить одинаковый результат на сервере и клиенте.

Если предупреждение создаёт браузерное расширение, сначала подтвердить это сравнением в режиме инкогнито или в чистом профиле. Не изменять код приложения только ради атрибутов, добавленных расширением.

Не использовать suppressHydrationWarning как основное исправление.

В итоговом отчёте указать точную найденную причину.

## 3. Настраиваемые стадии Kanban

Добавить возможность редактировать рабочие стадии Kanban лидов.

### Текущие системные результаты

Следующие результаты завершения не являются обычными редактируемыми рабочими стадиями:

- converted — успешно завершён;
- rejected — отказ.

Их нельзя переименовывать, удалять или использовать как активную рабочую стадию в рамках этой задачи.

### Рабочие стадии по умолчанию

- new — Новый;
- contact — Первичный контакт;
- qualification — Квалификация;
- proposal — Предложение;
- waiting — Ожидание решения.

### Интерфейс

На странице:

/sales/leads

добавить кнопку:

Настроить стадии

Кнопку разместить рядом с переключателями представления или в панели управления Kanban.

По нажатию открыть диалог настройки стадий.

В диалоге пользователь должен иметь возможность:

1. Переименовать стадию.
2. Изменить порядок стадии.
3. Добавить новую стадию.
4. Отключить стадию.
5. Включить ранее отключённую стадию.
6. Выбрать цветовой акцент стадии.
7. Сохранить настройки.
8. Отменить изменения.
9. Восстановить стадии по умолчанию.

### Ограничения

Должна всегда существовать хотя бы одна активная рабочая стадия.

Нельзя создать две стадии с одинаковым id.

Название стадии обязательно.

Название после trim не может быть пустым.

Системные результаты converted и rejected не должны редактироваться через этот интерфейс.

Если в отключаемой стадии находятся лиды:

- нельзя молча скрыть или потерять эти лиды;
- показать предупреждение;
- потребовать выбрать активную стадию, куда будут перемещены лиды;
- после подтверждения перенести лиды;
- только затем отключить стадию.

Удаление стадии физически на этом этапе не требуется. Использовать отключение через isActive.

## 4. Модель стадии на frontend

Создать понятный тип конфигурации стадии.

Пример полей:

- id;
- title;
- accentClass или color;
- isActive;
- sortOrder;
- isSystem;
- createdAt при необходимости.

Не хранить JSX-компоненты внутри конфигурации.

Не использовать случайные id через Math.random или Date.now во время рендера.

Для пользовательской стадии id должен формироваться только по действию пользователя.

Допустимо использовать crypto.randomUUID внутри обработчика кнопки добавления стадии, а не во время серверного рендера.

Также допустимо использовать стабильный slug с проверкой уникальности.

## 5. Хранение настроек

В текущей версии frontend работает на демонстрационных данных.

Поэтому разрешено временно хранить конфигурацию стадий в localStorage.

Ключ localStorage должен быть понятным и версионируемым, например:

sport-leads.sales.lead-stages.v1

### Требования к localStorage

- приложение должно иметь стабильный набор стадий по умолчанию;
- серверный рендер использует стадии по умолчанию;
- первый клиентский рендер также использует стадии по умолчанию;
- сохранённые настройки загружаются после mount;
- загрузка настроек не должна создавать hydration mismatch;
- повреждённые данные localStorage не должны ломать страницу;
- данные необходимо валидировать;
- при ошибке использовать стадии по умолчанию;
- восстановление по умолчанию должно очищать или заменять сохранённую конфигурацию.

В интерфейсе или документации явно указать, что в текущей версии настройки сохраняются только в текущем браузере.

Не добавлять backend-модель стадий в рамках этой задачи.

## 6. Интеграция стадий с Kanban

Удалить жёсткую зависимость интерфейса Kanban от константы definitions, если она мешает редактированию стадий.

Колонки активного Kanban должны строиться из текущей конфигурации стадий.

Учитывать:

- только активные рабочие стадии;
- sortOrder;
- пользовательское название;
- пользовательский цвет;
- системные колонки converted и rejected в соответствующих режимах просмотра.

DnD должен использовать id стадии, а не отображаемое название.

После переименования стадии лиды должны оставаться на прежней стадии.

После изменения порядка должны перестраиваться только колонки, а статусы лидов не должны меняться.

После добавления новой стадии в неё можно перемещать активные лиды.

Завершённые лиды нельзя перемещать.

## 7. Совместимость статусов

Текущий тип Lead может содержать ограниченный union статусов.

Не использовать небезопасные приведения типов по всему приложению.

Проверить типизацию и выбрать устойчивый вариант для пользовательских стадий.

Допустимые направления:

- выделить тип идентификатора рабочей стадии как string;
- отделить системный статус завершения completed от идентификатора рабочей стадии;
- добавить отдельное поле stageId для frontend-модели, если это минимально и не ломает остальную архитектуру.

Не изменять backend-модели и миграции в этой задаче.

В итоговом отчёте описать выбранный вариант совместимости.
