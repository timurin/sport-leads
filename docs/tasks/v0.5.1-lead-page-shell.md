# Задача Codex: v0.5.1 — Каркас страницы лида

## Контекст проекта

Проект: `sport-leads`

Текущий завершённый этап:

```text
v0.4.2
```

Следующий этап:

```text
v0.5.0 — Lead Workspace
```

Нужно начать разработку полноценного рабочего интерфейса карточки лида.

На данном шаге требуется создать только каркас страницы, маршрутизацию и базовые состояния интерфейса.

Не добавляй пока:

* редактирование данных;
* отправку сообщений;
* задачи;
* файлы;
* создание заказа;
* реальные API интеграции коммуникаций;
* сложную бизнес-логику.

---

# Цель задачи

Создать отдельную страницу лида, которая будет открываться из списка или канбана лидов и станет основой для последующего интерфейса менеджера.

Маршрут страницы:

```text
/sales/leads/[leadId]
```

Пример:

```text
/sales/leads/123
```

---

# 1. Сначала изучи проект

Перед внесением изменений:

1. Изучи текущую структуру frontend.
2. Определи используемый стек:

   * Next.js или другой frontend-фреймворк;
   * App Router или Pages Router;
   * способ работы с API;
   * существующие layout-компоненты;
   * UI-библиотеку;
   * стилизацию;
   * систему маршрутизации;
   * существующие страницы лидов.
3. Найди:

   * страницу списка лидов;
   * канбан лидов;
   * существующие карточки и таблицы;
   * общий layout ERP/CRM;
   * компоненты кнопок, карточек, badge, skeleton и alert.
4. Используй текущую архитектуру проекта.
5. Не создавай параллельную дизайн-систему и не дублируй существующие компоненты.

Перед написанием кода кратко сообщи:

* какие файлы были изучены;
* какой маршрут будет создан;
* какие существующие компоненты будут переиспользованы;
* какие файлы планируется изменить или добавить.

---

# 2. Создай маршрут карточки лида

Создай динамический маршрут:

```text
/sales/leads/[leadId]
```

Параметр `leadId` должен извлекаться из URL.

Страница должна корректно работать при прямом переходе по URL.

Пример:

```text
http://localhost:3000/sales/leads/123
```

---

# 3. Добавь переход в карточку лида

Из существующего интерфейса списка или канбана лидов пользователь должен иметь возможность открыть страницу лида.

Добавь переход:

* по нажатию на карточку лида в канбане;
* по нажатию на строку или название лида в таблице;
* либо через кнопку открытия, если это соответствует текущему UX проекта.

Не ломай существующие действия внутри карточки или строки.

Интерактивные кнопки внутри карточки не должны случайно вызывать переход, если они предназначены для другого действия.

---

# 4. Создай структуру страницы

Страница должна использовать общий layout проекта.

Основная структура:

```text
┌──────────────────────────────────────────────────────────────┐
│ Верхняя панель лида                                          │
├───────────────────────────────────────┬──────────────────────┤
│                                       │                      │
│ Основная рабочая область              │ Правая панель        │
│                                       │                      │
│                                       │                      │
├───────────────────────────────────────┴──────────────────────┤
│ История и коммуникации                                       │
└──────────────────────────────────────────────────────────────┘
```

Допускается адаптация структуры под текущую дизайн-систему проекта.

---

# 5. Верхняя панель

Создай базовую шапку страницы.

Пока достаточно вывести тестовые или полученные данные:

* номер лида;
* название клиента;
* текущий статус;
* ответственный менеджер;
* кнопку возврата;
* меню дополнительных действий.

Пример:

```text
Лид #123
ФК «Олимп»

Статус: Квалификация
Ответственный: Мария Иванова
```

Кнопка возврата должна возвращать пользователя к списку лидов.

Желательно использовать корректное поведение истории браузера, но предусмотреть безопасный переход на `/sales/leads`, если истории нет.

---

# 6. Основная рабочая область

Создай визуальные placeholder-блоки:

## Данные клиента

Пока только заголовок и заглушка:

```text
Данные клиента
Информация о клиенте будет добавлена на следующем этапе.
```

## Коммерческие данные

```text
Коммерческие параметры
Параметры потребности и будущего заказа будут добавлены позже.
```

## Дополнительная информация

```text
Дополнительная информация
```

Не создавай полноценные формы.

---

# 7. Правая панель

Создай отдельную боковую панель.

В ней пока должны находиться placeholder-блоки:

* ближайшая задача;
* ответственный;
* источник лида;
* дата создания;
* дата последней активности;
* связанные объекты.

Пример:

```text
Ближайшая задача
Задачи отсутствуют
```

```text
Связанные объекты
Заказы отсутствуют
```

---

# 8. История и коммуникации

В нижней части или центральной области страницы создай контейнер:

```text
История лида
```

Пока вывести пустое состояние:

```text
История пока пуста.
События, сообщения и комментарии появятся здесь.
```

Добавь визуальные вкладки, только если в проекте уже используется подобный компонент:

* Все события;
* Комментарии;
* Сообщения;
* Задачи.

На этом этапе вкладки могут быть статичными.

---

# 9. Данные страницы

Если в проекте уже существует API получения лида по ID, используй его.

Предполагаемый запрос:

```text
GET /api/leads/{leadId}
```

Но не придумывай endpoint без проверки проекта.

Если endpoint существует:

* подключи загрузку;
* типизируй ответ;
* используй существующий API client;
* не создавай второй API client.

Если endpoint пока отсутствует:

* создай отдельный mock-объект в frontend;
* не добавляй новый backend endpoint в этой задаче;
* явно пометь mock-данные;
* структура mock должна быть близка к будущей модели лида.

Пример mock:

```ts
const lead = {
  id: 123,
  title: 'ФК «Олимп»',
  status: 'qualification',
  statusLabel: 'Квалификация',
  responsible: {
    id: 1,
    name: 'Мария Иванова',
  },
  source: 'Форма сайта',
  createdAt: '2026-07-16T10:00:00Z',
  lastActivityAt: '2026-07-16T12:30:00Z',
}
```

---

# 10. Состояние загрузки

Добавь состояние загрузки страницы.

Используй существующий компонент `Skeleton`, если он есть.

Skeleton должен приблизительно повторять структуру:

* шапка;
* основная колонка;
* правая панель;
* история.

Не показывай пустой белый экран во время загрузки.

---

# 11. Состояние «лид не найден»

Если лид отсутствует или API возвращает `404`, показать отдельное состояние:

```text
Лид не найден
```

Текст:

```text
Лид с указанным идентификатором не существует или был удалён.
```

Кнопка:

```text
Вернуться к списку лидов
```

Кнопка должна вести на:

```text
/sales/leads
```

---

# 12. Состояние ошибки

При ошибке загрузки показать:

```text
Не удалось загрузить лид
```

Текст:

```text
Произошла ошибка при получении данных. Попробуйте повторить запрос.
```

Кнопки:

* повторить;
* вернуться к списку лидов.

Не показывай пользователю необработанный stack trace или технический ответ API.

---

# 13. Адаптивность

Основной приоритет — desktop.

На широком экране:

* основная рабочая область занимает большую часть ширины;
* правая панель имеет фиксированную или ограниченную ширину.

На узком экране:

* правая панель переносится под основную область;
* элементы шапки корректно переносятся;
* нет горизонтальной прокрутки;
* текст и кнопки не выходят за границы контейнера.

---

# 14. Типизация

Не используй `any`, если можно определить тип.

Создай или переиспользуй типы:

```ts
LeadDetails
LeadResponsible
LeadStatus
LeadSource
```

Не дублируй типы, если они уже есть в проекте.

Если backend использует `snake_case`, а frontend — `camelCase`, используй существующий механизм преобразования данных.

Не создавай новую систему преобразования без необходимости.

---

# 15. Компоненты

Разрешается разделить страницу на компоненты:

```text
LeadPage
LeadHeader
LeadMainContent
LeadSidebar
LeadTimelinePlaceholder
LeadPageSkeleton
LeadNotFoundState
LeadErrorState
```

Но не создавай чрезмерную декомпозицию.

Компоненты должны находиться в структуре, соответствующей текущим правилам проекта.

---

# 16. Ограничения

Не выполняй в этой задаче:

* формы редактирования;
* изменение статуса через API;
* сохранение данных;
* создание задач;
* отправку сообщений;
* загрузку файлов;
* создание заказа;
* подключение Telegram;
* подключение WhatsApp;
* подключение email;
* изменение backend-моделей;
* миграции базы данных;
* масштабный рефакторинг существующего frontend.

Не изменяй код, который не относится к текущей задаче.

---

# 17. Проверки

После реализации выполни доступные проверки проекта:

```text
lint
typecheck
tests
build
```

Используй команды из существующего `package.json`.

Не придумывай команды, которых нет в проекте.

Дополнительно проверь:

1. Страница открывается по прямому URL.
2. Страница открывается из списка лидов.
3. Кнопка возврата работает.
4. Неизвестный ID показывает состояние «лид не найден».
5. Ошибка API обрабатывается.
6. Загрузка отображает skeleton.
7. На узком экране нет горизонтального переполнения.
8. В консоли браузера нет ошибок.
9. Нет ошибок TypeScript.
10. Существующая страница списка лидов продолжает работать.

# Критерий готовности

Задача считается выполненной, если:

* существует отдельный маршрут карточки лида;
* лид можно открыть из текущего интерфейса;
* отображается базовый каркас рабочего пространства;
* реализованы loading, not found и error состояния;
* используется текущий layout проекта;
* страница адаптивна;
* существующие страницы не сломаны;
* TypeScript и сборка проходят без ошибок.
