# Задача Codex: v0.5.10 — Визуальное выравнивание Lead Workspace

## Контекст

Проект: `sport-leads`.

Реализованы этапы карточки лида:

* `v0.5.1` — каркас страницы;
* `v0.5.2` — шапка;
* `v0.5.3` — клиент и контакты;
* `v0.5.4` — коммерческие параметры;
* `v0.5.5` — история активности;
* `v0.5.6` — внутренние заметки;
* `v0.5.7` — задачи;
* `v0.5.8` — коммуникации;
* `v0.5.9` — ядро коннекторов.

Карточка доступна по маршруту:

```text
/sales/leads/[leadId]
```

Функциональность уже работает. Текущая задача должна привести внешний вид карточки к единой утверждённой стилистике CRM Lead Workspace.

Основной приоритет — визуальная структура, плотность интерфейса, иерархия блоков и удобство рабочего места менеджера.

Не добавлять новую бизнес-логику.

---

# Цель

Перестроить карточку лида в единое рабочее пространство менеджера:

```text
┌──────────────────────────────────────────────────────────────────────────┐
│ Навигация / хлебные крошки                                               │
├──────────────────────────────────────────────────────────────────────────┤
│ Компактная шапка лида: клиент, статус, сумма, ответственный, действия    │
├───────────────────────────────────────────────┬──────────────────────────┤
│                                               │                          │
│ Основная рабочая область                      │ Правая боковая панель    │
│                                               │                          │
│ Клиент / Коммерческие данные                  │ Ближайшая задача         │
│                                               │ Ответственный            │
│ Коммуникации / История                        │ Источник                 │
│                                               │ Связанные объекты        │
│                                               │ Ключевые даты            │
├───────────────────────────────────────────────┴──────────────────────────┤
│ Дополнительные разделы                                                   │
└──────────────────────────────────────────────────────────────────────────┘
```

Интерфейс должен выглядеть как единая CRM, а не как набор независимых крупных карточек.

# 2. Не менять рабочую бизнес-логику

Сохрани работающими:

* редактирование клиента;
* редактирование коммерческих параметров;
* контакты;
* задачи;
* заметки;
* закрепление заметок;
* timeline;
* фильтры;
* локальные сообщения;
* шаблоны сообщений;
* вложения;
* действия шапки;
* синхронизацию данных между блоками.

Не переписывай состояние страницы без необходимости.

Не меняй типы данных и mock-данные, если это не требуется для отображения.

---

# 3. Общий контейнер страницы

Сделай основную область более широкой и подходящей для CRM.

Рекомендуемые ограничения:

```text
max-width: 1600–1800 px
```

Страница должна:

* использовать доступную ширину рабочего пространства;
* не выглядеть узкой колонкой;
* иметь единые внешние отступы;
* не иметь чрезмерно больших промежутков между секциями;
* сохранять читаемость на больших мониторах.

Не задавай жёсткую фиксированную ширину всей страницы.

---

# 4. Сетка Lead Workspace

На desktop используй двухколоночную сетку.

Основная колонка:

```text
minmax(0, 1fr)
```

Правая колонка:

```text
320–380 px
```

Пример:

```css
grid-template-columns: minmax(0, 1fr) 340px;
```

На широком экране правая панель должна оставаться компактной.

На экранах меньше выбранного breakpoint правая колонка переносится вниз.

Не допускай горизонтальной прокрутки.

---

# 5. Компактная шапка лида

Шапка не должна занимать чрезмерно большую высоту.

Сделай структуру:

## Верхняя строка

* кнопка возврата;
* номер лида;
* название клиента;
* badge статуса;
* меню «Ещё».

## Вторая строка

* основной контакт;
* ответственный;
* источник;
* дата последней активности.

## Правая зона

* сумма;
* вероятность;
* основные действия.

Основные действия:

* добавить задачу;
* изменить статус;
* назначить ответственного;
* создать заказ;
* ещё.

На desktop кнопки располагаются компактной группой.

Второстепенные действия можно перенести в меню «Ещё».

---

# 6. Визуальная иерархия шапки

Приоритет текста:

1. название клиента;
2. сумма;
3. статус;
4. контакт и ответственный;
5. служебные даты и источник.

Не делай номер лида самым крупным элементом.

Пример иерархии:

```text
Футбольная академия «Олимп»      480 000 ₽
Лид #lead-1 · Сергей Волков      Вероятность 40%
```

Используй текущую типографику проекта.

---

# 7. Единый стиль секций

Все основные секции должны использовать единую структуру:

* небольшой заголовок;
* краткое дополнительное описание при необходимости;
* действия справа;
* содержимое;
* одинаковые border, radius и padding.

Не допускай, чтобы один блок имел огромный заголовок, а другой — маленький.

Рекомендуемая плотность:

```text
padding: 16–20 px
gap: 12–16 px
```

Избегай чрезмерных:

```text
padding: 32 px
gap: 32 px
```

---

# 8. Клиент и коммерческие параметры

На desktop размести блоки клиента и коммерческих параметров рядом, если ширина позволяет:

```text
┌─────────────────────────────┬─────────────────────────────┐
│ Данные клиента              │ Коммерческие параметры     │
└─────────────────────────────┴─────────────────────────────┘
```

Если содержимое слишком большое, используй вкладки:

```text
Клиент
Потребность
Контакты
```

Но не скрывай критически важную информацию слишком глубоко.

В режиме просмотра показывай только основные значения.

Редкие и второстепенные поля можно поместить в раскрываемый раздел:

```text
Показать дополнительные данные
```

Не удаляй поля и не ломай редактирование.

---

# 9. Контактные лица

Контакты должны выглядеть компактно.

Вместо больших вертикальных карточек предпочтительно:

* компактные строки;
* небольшой avatar/инициалы;
* имя и должность;
* телефон и email;
* основной канал;
* меню действий.

Основной контакт должен иметь текстовую отметку:

```text
Основной
```

Не выделяй каждый контакт чрезмерной рамкой и большим padding.

---

# 10. Коммерческие параметры

В режиме просмотра сгруппируй параметры:

## Потребность

* направление;
* вид спорта;
* продукция;
* количество.

## Финансы

* бюджет;
* сумма;
* скидка;
* вероятность.

## Сроки и доставка

* заказ;
* готовность;
* мероприятие;
* город;
* доставка.

Описание потребности должно занимать всю ширину блока.

Ключевые финансовые значения можно визуально выделить, но без дублирования шапки.

---

# 11. Правая боковая панель

Правая панель должна содержать компактные информационные блоки:

1. ближайшая задача;
2. ответственный;
3. источник;
4. ключевые даты;
5. связанные объекты;
6. краткие показатели.

Панель не должна повторять всю информацию из шапки.

Приоритет:

* ближайшее действие;
* важные сроки;
* ответственный;
* связи.

На широком desktop допускается sticky-поведение:

```css
position: sticky;
top: ...
```

Используй sticky только если он не ломает прокрутку и layout.

---

# 12. Задачи

Полный список задач размести в основной колонке.

Карточки задач должны быть компактными строками:

* checkbox или статус;
* название;
* срок;
* исполнитель;
* приоритет;
* меню.

Не использовать слишком высокие карточки для коротких задач.

Просрочка должна обозначаться:

* текстом;
* иконкой;
* аккуратным цветовым акцентом.

---

# 13. Коммуникации и timeline

Это ключевая рабочая зона менеджера.

Предпочтительная структура:

```text
┌──────────────────────────────────────────────────────────────┐
│ Вкладки: Переписка | История | Заметки | Задачи             │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ Выбранный раздел                                             │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

Допускается оставить коммуникации и timeline рядом или последовательно, если это лучше соответствует текущей реализации.

Главное:

* не показывать одновременно несколько очень длинных вертикальных лент;
* уменьшить высоту всей страницы;
* обеспечить быстрый доступ к переписке;
* не потерять существующие фильтры;
* сохранить внутренние заметки отдельно от клиентских сообщений.

---

# 14. Рекомендуемые вкладки центральной зоны

Рассмотри вкладки:

```text
Коммуникации
История
Задачи
Заметки
```

Правила:

* вкладка не должна сбрасывать локальное состояние;
* формы и draft не должны теряться;
* задачи остаются доступны из правой панели;
* закреплённые заметки могут отображаться над вкладками или внутри «Заметки».

Не вводи маршрутизацию для вкладок.

---

# 15. Панель коммуникаций

Сделай интерфейс похожим на рабочий чат:

* компактный header канала;
* ограниченная высота истории;
* прокручиваемая область;
* поле ввода закреплено снизу;
* входящие сообщения слева;
* исходящие справа;
* вложения компактны;
* статусы доставки не перегружают интерфейс.

Высота панели на desktop ориентировочно:

```text
500–700 px
```

Не позволяй панели бесконтрольно увеличивать высоту страницы.

---

# 16. Timeline

Timeline должен быть визуально легче.

Используй:

* тонкую вертикальную линию;
* компактные иконки;
* небольшой заголовок события;
* метаданные приглушённым текстом;
* описание без лишних рамок.

Не помещай каждое событие в отдельную тяжёлую большую Card, если это создаёт визуальный шум.

---

# 17. Цвета

Используй текущую палитру проекта.

Базовые поверхности:

* белый или текущий background;
* светло-серые разделители;
* нейтральный secondary background.

Акценты:

* синий — основные действия;
* зелёный — завершено/успех;
* красный — просрочено/ошибка;
* янтарный — предупреждение;
* фиолетовый или другой существующий цвет — коммуникационный канал, если уже используется.

Не добавляй много новых случайных цветов.

Не задавай inline hex-цвета, если проект использует Tailwind-токены.

---

# 18. Радиусы, границы и тени

Используй единый радиус компонентов.

Не смешивай:

* слишком круглые карточки;
* квадратные формы;
* разные типы теней.

Предпочтительно:

```text
border
subtle shadow или без shadow
единый radius
```

Не превращай карточку лида в набор «плавающих» панелей с тяжёлыми тенями.

---

# 19. Кнопки

Установи визуальный приоритет:

## Primary

* добавить задачу;
* отправить сообщение;
* сохранить форму.

## Secondary

* редактировать;
* изменить статус;
* назначить ответственного.

## Ghost / menu

* копировать;
* удалить;
* дополнительные действия.

Не показывай пять одинаково ярких primary-кнопок в шапке.

---

# 20. Формы

В режиме редактирования:

* поля должны иметь одинаковую высоту;
* labels — единый размер;
* ошибки не должны сдвигать layout чрезмерно;
* кнопки «Сохранить» и «Отмена» должны находиться в ожидаемом месте;
* длинные формы разбить на логические группы.

Не изменяй правила валидации.

---

# 21. Пустые состояния

Сделай пустые состояния компактными.

Не использовать крупные пустые карточки для текста:

```text
Не указано
```

В режиме просмотра пустые значения отображай приглушённо.

Полное пустое состояние секции может содержать:

* небольшую иконку;
* одну строку;
* компактную кнопку.

---

# 22. Skeleton

Обнови skeleton под итоговую сетку:

* компактная шапка;
* две основные карточки;
* правая панель;
* центральная вкладочная зона.

Skeleton не должен заметно отличаться по размерам от готовой страницы.

---

# 23. Адаптивность

## Desktop от 1280 px

* двухколоночная сетка;
* правая панель 320–380 px;
* компактная шапка;
* основные секции могут располагаться рядом.

## Планшет / небольшой ноутбук

* правая панель переносится вниз либо становится горизонтальным блоком;
* действия шапки переносятся;
* вкладки остаются доступными.

## Мобильная ширина

* одна колонка;
* шапка вертикальная;
* основные действия занимают доступную ширину;
* вкладки прокручиваются внутри контейнера;
* Dialog помещается в viewport;
* нет горизонтальной прокрутки страницы.

Основной приоритет — desktop CRM.

# 26. Визуальные проверки

После изменений визуально проверь `lead-1`.

Проверь:

1. Страница выглядит как единое рабочее пространство.
2. Шапка компактная.
3. Название клиента является главным визуальным элементом.
4. Сумма и статус хорошо видны.
5. Основная колонка шире правой.
6. Правая панель не перегружена.
7. Клиент и коммерческие данные легко сканируются.
8. Контакты отображаются компактно.
9. Задачи не занимают чрезмерную высоту.
10. Коммуникации похожи на рабочий чат.
11. Timeline не выглядит набором тяжёлых карточек.
12. На экране нет большого количества одинаковых рамок.
13. Основные действия визуально отделены от второстепенных.
14. Нет дублирования одной информации в трёх местах.
15. Нет горизонтальной прокрутки.
16. Нет обрезанных dropdown или Dialog.
17. Нет layout shift при переключении вкладок.
18. Локальное состояние и draft сохраняются.
19. В Console нет ошибок.
20. Нет hydration mismatch.

# Критерии готовности

Задача считается выполненной, если:

* Lead Workspace выглядит как единая CRM-карточка;
* структура соответствует утверждённой двухколоночной концепции;
* шапка стала компактной;
* основная колонка и правая панель имеют правильную иерархию;
* секции используют единый стиль;
* коммуникации и timeline визуально упорядочены;
* интерфейс адаптивен;
* вся существующая функциональность сохранена;
* отсутствуют ошибки TypeScript, lint, build и hydration.
