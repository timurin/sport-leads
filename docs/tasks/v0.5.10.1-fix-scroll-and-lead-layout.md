# Задача Codex: v0.5.10.1 — Исправить двойную прокрутку и структуру карточки лида

## Контекст

Проект: sport-leads.

После выполнения v0.5.10 визуальная проверка выявила две проблемы:

1. На странице карточки лида появилась двойная вертикальная прокрутка:
   - прокручивается окно браузера;
   - одновременно прокручивается внутренний контейнер Lead Workspace.

2. Внешний вид карточки всё ещё заметно отличается от утверждённой концепции:
   - слишком много рамок;
   - секции выглядят изолированными;
   - правая колонка похожа на отдельную высокую карточку;
   - блок ближайшей задачи занимает слишком много места;
   - недостаточно выражена единая рабочая зона менеджера;
   - коммуникации, история, задачи и заметки визуально не объединены.

Маршрут карточки:

/sales/leads/[leadId]

Не добавляй новую бизнес-логику.

## Цель

Исправить прокрутку и перестроить страницу в цельный CRM Lead Workspace:

- оставить только одну основную вертикальную прокрутку;
- сделать шапку компактной;
- оставить широкую центральную рабочую область;
- облегчить правую информационную панель;
- уменьшить количество вложенных рамок;
- объединить коммуникации, историю, задачи и заметки во вкладочную рабочую зону;
- сохранить всю существующую функциональность.

## 1. Сначала найди точную причину двойной прокрутки

Изучи:

- frontend/app/(workspace)/layout.tsx или фактический workspace layout;
- frontend/app/(workspace)/sales/leads/[leadId]/;
- frontend/components/sales/lead-page.tsx;
- frontend/components/sales/lead-header.tsx;
- frontend/components/sales/lead-communication-panel.tsx;
- frontend/components/sales/lead-activity-timeline.tsx;
- frontend/components/sales/lead-tasks.tsx;
- компоненты правой панели;
- глобальные CSS;
- родительские контейнеры страницы.

Особенно проверь комбинации классов:

- h-screen и внутренний overflow-y-auto;
- min-h-screen и фиксированная высота дочернего блока;
- overflow-hidden у layout и overflow-auto у content;
- height: calc(100vh - ...);
- sticky внутри overflow-контейнера;
- max-h-screen;
- h-full в цепочке контейнеров без корректной высоты родителя.

Перед внесением изменений кратко сообщи:

- какой контейнер создаёт внешний scrollbar;
- какой контейнер создаёт внутренний scrollbar;
- какие классы или CSS-правила будут изменены.

## 2. Оставь одну основную вертикальную прокрутку

На странице карточки лида должна быть только одна основная вертикальная прокрутка.

Предпочтительное поведение:

- прокручивается общий документ или основной workspace-контейнер проекта;
- сама карточка лида не имеет отдельного общего overflow-y-auto;
- правая колонка не имеет собственного вертикального scrollbar;
- Lead Workspace не ограничивается искусственной высотой viewport;
- страница растёт по содержимому.

Удали или скорректируй ненужные:

- h-screen;
- max-h-screen;
- h-[calc(100vh-...)];
- overflow-y-auto;
- overflow-hidden;
- фиксированную высоту общего контейнера карточки.

Не ломай layout других страниц.

Если глобальное изменение может повлиять на другие разделы, сделай локальное исправление для страницы лида.

## 3. Допустимые внутренние прокрутки

Внутренняя прокрутка допустима только внутри функциональных областей:

- список сообщений;
- длинный timeline внутри активной вкладки;
- выпадающий список;
- Dialog.

При этом:

- header коммуникаций не прокручивается;
- поле ввода сообщения не прокручивается;
- прокручивается только список сообщений;
- правая колонка движется вместе с основной страницей;
- внутренний scrollbar чата не должен создавать второй общий scrollbar Lead Workspace.

## 4. Перестрой общую структуру страницы

Используй структуру:

Хлебные крошки

Компактная шапка лида

Двухколоночная рабочая область:
- основная колонка;
- правая информационная колонка.

Основная колонка:
- данные клиента;
- коммерческие параметры;
- общая вкладочная рабочая зона.

Правая колонка:
- ближайшее действие;
- ответственный;
- источник;
- ключевые даты;
- связанные объекты.

Рекомендуемая desktop-сетка:

grid-template-columns: minmax(0, 1fr) 330px

Допустимый диапазон правой колонки:

320–340 px.

На небольшом экране правая колонка должна переноситься вниз.

Не допускай горизонтальной прокрутки.

## 5. Упрости правую панель

Правая панель не должна выглядеть как одна большая карточка внутри другой карточки.

Убери тяжёлую внешнюю рамку вокруг всей колонки либо сделай её визуально нейтральной.

Разделяй блоки через:

- тонкие separator;
- небольшие вертикальные отступы;
- компактные заголовки;
- лёгкие фоновые поверхности только при необходимости.

Структура:

Ближайшее действие

Ответственный

Источник лида

Последняя активность

Дата создания

Связанные объекты

Не дублируй всё содержимое шапки.

## 6. Сделай блок ближайшего действия компактнее

Сейчас блок задачи занимает слишком много высоты.

Показывай:

- название;
- тип;
- исполнителя;
- срок;
- просрочку;
- приоритет.

Действия размести компактно.

Предпочтительный вариант:

- одна primary-кнопка «Завершить»;
- действие «Открыть» как secondary, link или пункт меню.

Не делай две большие кнопки во всю ширину одна под другой без необходимости.

На desktop допустима одна строка действий:

Открыть | Завершить

## 7. Уменьши количество вложенных рамок

Избегай структуры:

Card
  Card
    bordered block
      bordered row

Используй:

- одну основную поверхность секции;
- separator между логическими группами;
- мягкий secondary background;
- рамку только для границы крупного функционального блока;
- компактные строки для контактных лиц и задач.

Клиент, коммерческие параметры и правая панель должны выглядеть частями одного интерфейса.

## 8. Шапка лида

Сохрани компактную высоту.

Главный визуальный элемент:

название клиента.

Рядом отображай:

- статус;
- сумму;
- вероятность.

Второстепенная информация:

- номер лида;
- основной контакт;
- ответственный;
- источник;
- дата последней активности.

Пример визуальной иерархии:

Футбольная академия «Олимп»          480 000 ₽

Лид lead-1 · Сергей Волков           Вероятность 40%

Статус должен быть рядом с названием или в первой строке.

Не превращай номер лида в главный заголовок.

## 9. Клиент и коммерческие параметры

На desktop размести блоки рядом:

- данные клиента;
- коммерческие параметры.

Используй две колонки одинаковой или близкой ширины.

В режиме просмотра показывай сначала ключевую информацию.

Редкие поля можно поместить в раскрываемый блок:

Показать дополнительные данные

Не удаляй поля.

Не ломай режим редактирования.

Не добавляй отдельную рамку вокруг каждого поля.

## 10. Контактные лица

Отображай контакты компактными строками.

Строка контакта должна содержать:

- инициалы или небольшой avatar;
- имя;
- должность;
- телефон;
- email;
- предпочтительный канал;
- отметку «Основной»;
- меню действий.

Не используй большие вертикальные карточки для каждого контакта.

## 11. Коммерческие параметры

Сгруппируй режим просмотра:

Потребность:
- направление;
- вид спорта;
- категория;
- продукция;
- количество.

Финансы:
- бюджет;
- сумма;
- скидка;
- вероятность.

Сроки и доставка:
- дата заказа;
- дата готовности;
- дата мероприятия;
- город;
- способ доставки.

Описание потребности занимает всю ширину блока.

Не дублируй сумму и вероятность слишком крупно, если они уже показаны в шапке.

## 12. Объедини рабочие разделы вкладками

Создай одну центральную рабочую область с вкладками:

- Коммуникации;
- История;
- Задачи;
- Заметки.

В каждый момент времени отображается только один длинный раздел.

Не показывай одновременно:

- длинный чат;
- полный timeline;
- полный список задач;
- полный список заметок

один под другим.

Переключение вкладок не должно:

- сбрасывать локальное состояние;
- очищать draft сообщения;
- удалять новые задачи;
- удалять заметки;
- сбрасывать фильтры без необходимости.

Не добавляй отдельную маршрутизацию вкладок.

## 13. Коммуникационная панель

Сделай структуру:

- header канала;
- прокручиваемый список сообщений;
- composer.

Ориентировочная общая высота панели на desktop:

560–680 px.

Только список сообщений получает overflow-y-auto.

Header и composer остаются на месте.

Не задавай overflow-y-auto всей коммуникационной секции.

Входящие сообщения — слева.

Исходящие сообщения — справа.

Вложения и статусы показывай компактно.

## 14. Timeline

Timeline внутри вкладки должен выглядеть легче.

Используй:

- тонкую вертикальную линию;
- небольшие иконки;
- компактный заголовок события;
- приглушённые дату и автора;
- описание без тяжёлой отдельной Card.

Не помещай каждое событие в большую карточку с собственной тенью.

Если timeline длинный, допускается внутренняя прокрутка только содержимого вкладки.

## 15. Задачи

Полный список задач размести во вкладке «Задачи».

Используй компактные строки или небольшие карточки.

Показывай:

- статус;
- название;
- срок;
- исполнителя;
- приоритет;
- меню.

Просрочку обозначай:

- текстом «Просрочено»;
- иконкой;
- аккуратным цветовым акцентом.

Не делай каждую короткую задачу высокой карточкой.

## 16. Заметки

Закреплённые заметки могут отображаться:

- в верхней части вкладки «Заметки»;
- либо компактным блоком над вкладками.

Не дублируй их крупным блоком в нескольких местах.

Обычные заметки должны быть визуально легче системных карточек.

Сохрани:

- редактирование;
- удаление;
- закрепление;
- упоминания.

## 17. Sticky-поведение

Правая панель может быть sticky только на широком desktop.

Используй:

position: sticky;
top: значение, соответствующее верхней панели;
align-self: start;

Не помещай sticky-блок внутрь контейнера с overflow-auto или overflow-hidden.

Если sticky вызывает второй scrollbar, обрезание или нестабильное поведение, откажись от sticky.

## 18. Общие отступы и плотность

Используй единые значения:

- внешние gap: 16–20 px;
- padding секций: 16–20 px;
- внутренние gap: 8–12 px.

Избегай:

- padding 32 px для обычных CRM-блоков;
- gap 32 px между связанными элементами;
- чрезмерной высоты пустых состояний;
- больших вертикальных отступов в правой панели.

## 19. Цвета, границы и тени

Используй текущую палитру проекта.

Не добавляй случайные inline hex-цвета.

Предпочтительно:

- нейтральный фон страницы;
- белые основные поверхности;
- светлые separator;
- единый radius;
- тонкая border;
- минимальная shadow либо без shadow.

Не смешивай разные радиусы и тяжёлые тени.

## 20. Кнопки

В шапке и блоках соблюдай приоритет:

Primary:
- добавить задачу;
- отправить сообщение;
- сохранить;
- завершить ближайшую задачу.

Secondary:
- редактировать;
- открыть;
- изменить статус;
- назначить ответственного.

Ghost или меню:
- копировать;
- удалить;
- прочие действия.

Не показывай несколько одинаково ярких primary-кнопок рядом.

## 21. Адаптивность

Desktop от 1280 px:

- две колонки;
- правая колонка 320–340 px;
- клиент и коммерческие параметры рядом;
- одна основная прокрутка.

Небольшой ноутбук:

- правая панель переносится вниз;
- кнопки шапки переносятся;
- нет второго scrollbar.

Мобильная ширина:

- одна колонка;
- вкладки прокручиваются горизонтально только внутри своего контейнера;
- Dialog помещается в viewport;
- нет горизонтальной прокрутки всей страницы;
- сообщения занимают доступную ширину.

Основной приоритет — desktop CRM.

После перестройки должны работать:

- переход из канбана;
- кнопка возврата;
- редактирование клиента;
- добавление и редактирование контакта;
- коммерческие параметры;
- сумма, вероятность и источник в шапке;
- создание задачи;
- редактирование задачи;
- завершение задачи;
- перенос срока;
- заметки;
- закрепление заметок;
- упоминания;
- отправка mock-сообщения;
- шаблоны;
- mock-вложения;
- фильтры сообщений;
- timeline;
- фильтры timeline;
- меню шапки.

Не меняй бизнес-правила.

## 23. Обязательная визуальная проверка

Проверь lead-1:

1. На странице только один общий вертикальный scrollbar.
2. Правая панель не имеет отдельной вертикальной прокрутки.
3. Общий Lead Workspace не имеет внутреннего общего scrollbar.
4. Внутренняя прокрутка есть только у списка сообщений или длинного содержимого активной вкладки.
5. Шапка компактная.
6. Название клиента — главный элемент.
7. Сумма и статус заметны.
8. Основная колонка шире правой.
9. Правая панель выглядит лёгкой.
10. Блок ближайшей задачи стал компактнее.
11. Уменьшилось количество вложенных рамок.
12. Клиент и коммерческие параметры расположены рядом.
13. Коммуникации, история, задачи и заметки организованы вкладками.
14. Не отображаются одновременно несколько длинных лент.
15. Черновик сообщения не теряется при переключении вкладок.
16. Локальные задачи и заметки не теряются.
17. Нет layout shift.
18. Нет горизонтальной прокрутки.
19. Нет обрезанных Dropdown или Dialog.
20. В Console нет ошибок.
21. Нет hydration mismatch.

Проверь также:

- lead-2;
- неизвестный lead-999;
- открытие из канбана;
- ширину около 1280 px;
- мобильную ширину.

## Критерии готовности

Задача считается выполненной, если:

- двойная прокрутка устранена;
- на странице только один основной вертикальный scrollbar;
- правая панель не создаёт отдельную прокрутку;
- общая карточка не имеет собственного вертикального scrollbar;
- клиент и коммерческие параметры расположены рядом;
- коммуникации, история, задачи и заметки объединены вкладками;
- количество вложенных карточек и рамок уменьшено;
- блок ближайшей задачи стал компактнее;
- интерфейс стал визуально ближе к цельному CRM Lead Workspace;
- вся существующая функциональность сохранена;
- typecheck, lint и build проходят.
