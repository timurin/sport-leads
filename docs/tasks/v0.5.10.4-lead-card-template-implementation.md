# v0.5.10.4 — Перенос дизайна карточки лида из HTML-шаблона

## Источник дизайна

Утверждённый визуальный шаблон:

docs/example-design/sales/lead/lead-card-template.html

Текущая карточка:

/sales/leads/[leadId]

## Цель

Сопоставить текущую реализацию карточки лида с HTML-шаблоном и привести интерфейс платформы максимально близко к его внешнему виду.

HTML-шаблон определяет:

- композицию страницы;
- расположение блоков;
- пропорции колонок;
- визуальную плотность;
- размеры карточек;
- этапы лида;
- метрики;
- историю;
- задачи;
- комментарии;
- каналы коммуникации;
- карточку клиента;
- чат;
- цвета, отступы, границы и типографику.

## Перед началом

Изучи:

- HTML и CSS шаблона;
- текущую страницу LeadPage;
- LeadHeader;
- данные клиента;
- коммерческие параметры;
- задачи;
- заметки;
- timeline;
- коммуникационную панель;
- текущий layout, sidebar и topbar;
- существующие UI-компоненты и Tailwind-токены.

Сначала кратко покажи:

1. соответствие секций шаблона текущим React-компонентам;
2. какие компоненты будут перестилизованы;
3. какие визуальные компоненты нужно добавить;
4. какие файлы будут изменены.

## Главное ограничение

Не менять:

- бизнес-логику;
- текущую структуру данных;
- типы данных;
- callbacks;
- локальное состояние;
- API-контракты;
- коннекторы;
- коммуникационные механизмы;
- задачи и заметки;
- timeline;
- формы редактирования;
- существующую навигацию;
- loading, error и not-found;
- backend.

Разрешено менять JSX-композицию и стили компонентов, если вся существующая механика остаётся рабочей.

Разрешено добавлять новые презентационные компоненты, необходимые для дизайна.

## Реализовать

### 1. Общая композиция

Приведи страницу к двухколоночной композиции шаблона:

Левая часть:

- заголовок и действия;
- этапы лида;
- основная информация;
- потребность клиента;
- ключевые метрики;
- история активности;
- задачи;
- комментарии менеджера.

Правая часть:

- коммуникационные каналы;
- компактная карточка клиента;
- активный чат;
- composer;
- быстрые действия.

Используй существующий sidebar и topbar платформы. Не создавай второй sidebar или второй глобальный header внутри карточки.

### 2. Заголовок

Воспроизведи структуру шаблона:

- номер лида;
- статус;
- название организации;
- основные действия справа.

Сопоставь действия шаблона с существующими действиями проекта.

Не добавляй рабочее действие «Создать заказ», если соответствующая бизнес-логика ещё не реализована. В таком случае оставь disabled или используй существующую заглушку.

### 3. Этапы лида

Добавь горизонтальную визуальную шкалу этапов в стиле шаблона.

Используй реальные статусы и существующую механику изменения статуса.

Шкала должна отражать:

- завершённые этапы;
- текущий этап;
- будущие этапы.

Не создавай отдельное состояние статуса.

### 4. Основная информация

Собери существующие данные клиента в компактную карточку:

- контакт;
- организация;
- телефон;
- email;
- город;
- вид спорта;
- источник;
- ответственный;
- приоритет.

Редактирование должно продолжать работать через существующие формы.

### 5. Потребность клиента

Отобрази в отдельной карточке:

- категорию;
- количество или тираж;
- предполагаемый бюджет;
- желаемую дату;
- описание потребности.

Используй текущие коммерческие данные лида.

Не дублируй данные новым локальным объектом.

### 6. Ключевые метрики

Создай компактную горизонтальную панель, визуально близкую шаблону:

- вероятность конверсии;
- последний контакт;
- следующий контакт;
- дней в работе;
- количество касаний;
- потенциальная сумма.

Используй доступные данные.

Если для отдельной метрики данных пока нет:

- покажи нейтральное состояние;
- либо вычисли её только из уже существующих данных;
- не добавляй новую бизнес-логику или фиктивное постоянное хранение.

### 7. История, задачи и комментарии

На desktop расположи их в нижней сетке, как в шаблоне:

- история активности;
- задачи;
- комментарии или внутренние заметки.

Сопоставление:

- история → текущий LeadActivityTimeline;
- задачи → текущий LeadTasks;
- комментарии → текущие внутренние заметки.

Сохрани все действия:

- создание;
- редактирование;
- завершение;
- удаление;
- перенос срока;
- закрепление;
- упоминания;
- фильтры.

Допускается открытие полного содержимого в существующей вкладке, drawer или dialog, если компактная карточка показывает только часть элементов.

### 8. Коммуникационные каналы

Верхняя часть правой колонки должна повторять шаблон:

- заголовок;
- описание;
- карточки каналов;
- активный канал;
- статус подключения.

Используй текущий registry коннекторов и существующие статусы каналов.

Не заменяй реальные статусы статическим HTML.

Клик по каналу должен использовать текущую механику переключения.

### 9. Карточка клиента справа

Добавь компактную презентационную панель:

- предпочтительные каналы;
- среднее время ответа;
- часовой пояс;
- согласия, только если эти данные уже существуют;
- действие открытия или редактирования клиента.

Для отсутствующих данных используй нейтральное состояние «Не указано».

Не создавай отдельную сущность клиента.

### 10. Чат

Приведи текущую коммуникационную панель к внешнему виду шаблона:

- компактный header;
- активный канал;
- фон сообщений;
- входящие и исходящие bubbles;
- вложения;
- composer;
- кнопка отправки;
- шаблоны;
- быстрые коммуникационные действия.

Сохрани:

- draft;
- mock-отправку;
- шаблоны;
- вложения;
- фильтры;
- статусы сообщений;
- текущие коннекторы.

Не создавать второй компонент чата с отдельной логикой.

### 11. Визуальный стиль

Максимально приблизь интерфейс к шаблону:

- светло-серый фон;
- белые карточки;
- тонкие границы;
- небольшие тени;
- radius около 8–12 px;
- плотные отступы;
- синие primary-действия;
- зелёные статусы;
- красные и янтарные предупреждения;
- компактная типографика;
- высокая информационная плотность.

Не копируй inline CSS шаблона напрямую в JSX.

Используй Tailwind, существующие utility-классы, CSS variables или стили проекта.

### 12. Размеры и прокрутка

Шаблон рассчитан на широкий desktop, но production-страница должна оставаться адаптивной.

Desktop:

- две основные колонки;
- правая коммуникационная область примерно 42–46%;
- левая область примерно 54–58%.

Ноутбук:

- допускается уменьшение правой колонки;
- блоки не должны перекрываться.

Небольшая ширина:

- правая колонка переносится ниже;
- карточки основной информации становятся одной колонкой;
- нижняя сетка перестраивается;
- горизонтальный overflow страницы отсутствует.

Сохрани один основной scrollbar страницы.

Внутренняя прокрутка разрешена только:

- истории сообщений;
- длинным спискам;
- dialog/dropdown.

Не копируй из шаблона `body { overflow: hidden }`, если это создаёт проблемы в приложении.

## Не делать

- не переписывать страницу с нуля;
- не заменять React статическим HTML;
- не использовать данные, жёстко записанные из шаблона;
- не менять API;
- не менять backend;
- не менять коннекторы;
- не менять бизнес-правила;
- не удалять текущие функции;
- не выполнять масштабный архитектурный рефакторинг;
- не делать commit;
- не переходить к следующей версии.

## Проверить

Frontend:

npx tsc --noEmit
npm run lint
npm run build

Ручные сценарии:

1. Открывается lead-1.
2. Открывается lead-2.
3. lead-999 показывает not-found.
4. Статус лида меняется.
5. Ответственный меняется.
6. Клиент редактируется.
7. Контакты добавляются и редактируются.
8. Коммерческие параметры редактируются.
9. Задача создаётся и завершается.
10. Срок задачи переносится.
11. Заметка создаётся и закрепляется.
12. Канал коммуникации переключается.
13. Mock-сообщение отправляется.
14. Draft не теряется.
15. Шаблоны и вложения работают.
16. Timeline работает.
17. Нет двойной прокрутки.
18. Нет горизонтального overflow.
19. В Console нет ошибок.
20. Нет hydration mismatch.

## Готово, если

- карточка визуально максимально близка к HTML-шаблону;
- сохранена текущая глобальная навигация платформы;
- левая часть содержит данные, метрики и рабочие блоки;
- правая часть является коммуникационным центром;
- этапы лида отображаются визуально;
- все данные поступают из текущего состояния;
- вся существующая логика сохранена;
- коннекторы продолжают работать;
- адаптивность не сломана;
- typecheck, lint и build проходят.

## Отчёт

Покажи:

- карту соответствия шаблона и React-компонентов;
- добавленные презентационные компоненты;
- изменённые файлы;
- какие элементы шаблона не реализованы из-за отсутствия данных или логики;
- подтверждение сохранения механики;
- результаты typecheck, lint и build.

Не делай commit.
