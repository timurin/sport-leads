# v0.5.11 — Единая дизайн-система портала

## Цель

Создать базовую дизайн-систему Sport Leads, которую будут использовать все текущие и будущие модули. Не переделывать страницы целиком в рамках этой задачи.

## Перед началом

Изучи текущие:

- globals.css;
- Tailwind-конфигурацию;
- components/ui;
- layout;
- sidebar и topbar;
- карточку лида;
- список лидов;
- используемые варианты кнопок, карточек, badge, input и dialog.

Найди дублирующиеся стили и существующие UI-примитивы. Переиспользуй их и не создавай параллельную библиотеку компонентов.

## Реализовать

### Дизайн-токены

Определи единые CSS variables или Tailwind tokens:

- page background;
- surface;
- secondary surface;
- border;
- основной и приглушённый текст;
- primary, success, warning, danger;
- радиусы;
- тени;
- интервалы;
- высоты control;
- ширина sidebar;
- максимальная ширина контента.

Не использовать случайные inline hex в компонентах.

### Общий layout

Приведи к единому API или создай при отсутствии:

- AppShell;
- AppSidebar;
- AppTopbar;
- PageHeader;
- PageContent;
- PageActions;
- ResponsiveGrid.

Не менять текущую навигационную механику.

### Общие презентационные компоненты

Создай или унифицируй:

- SectionCard;
- InfoCard;
- MetricCard;
- StatusBadge;
- EmptyState;
- ActionMenu;
- CompactTabs;
- DataList;
- EntityHeader.

Компоненты должны принимать данные и children, не содержать бизнес-логику конкретного модуля.

### Варианты размеров

Для основных UI-компонентов предусмотреть:

- compact;
- default;
- spacious только при необходимости.

CRM-экраны по умолчанию используют compact/default.

### Адаптивные правила

Зафиксировать breakpoints:

- wide desktop;
- desktop/laptop;
- tablet;
- mobile.

Общие правила:

- не использовать глобальный min-width страницы;
- один основной вертикальный scrollbar;
- внутренний scrollbar только для чата, dialog и действительно длинных списков;
- grid-компоненты должны поддерживать minmax(0, 1fr);
- контент не должен ломаться при уменьшении ширины;
- действия должны переноситься или уходить в меню;
- таблицы на мобильном превращаются в строки/карточки либо имеют локальный горизонтальный scroll.

## Ограничения

- не менять бизнес-логику;
- не менять API;
- не менять backend;
- не менять коннекторы;
- не выполнять полный редизайн карточки лида;
- не удалять существующие рабочие компоненты без безопасной миграции;
- не делать commit.

## Проверить

npx tsc --noEmit
npm run lint
npm run build

Дополнительно проверь:

- существующие страницы продолжают открываться;
- sidebar и topbar работают;
- нет конфликтующих CSS variables;
- нет циклических импортов;
- новые компоненты не содержат бизнес-логики модулей.

## Готово, если

В проекте существует единая переиспользуемая дизайн-система, документированы визуальные и адаптивные правила, а последующие страницы можно собирать из общих layout- и UI-компонентов без копирования стилей.

## Отчёт

Покажи:

- созданные токены;
- созданные и переиспользованные компоненты;
- найденные дубли;
- изменённые файлы;
- результаты проверок;
- страницы, которые позже нужно мигрировать.

Не делай commit.
