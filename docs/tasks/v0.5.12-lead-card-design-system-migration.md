# v0.5.12 — Миграция карточки лида на дизайн-систему

## Контекст

В проекте создана единая дизайн-система портала:

- AppShell;
- AppTopbar;
- PageContent;
- PageActions;
- ResponsiveGrid;
- глобальные portal-токены;
- portal-grid-small;
- portal-grid-medium;
- portal-grid-large;
- документация в docs/design-system.

Текущая карточка лида уже содержит всю необходимую механику, но её визуальная композиция и стили частично собраны локально и плохо адаптируются на некоторых разрешениях.

## Цель

Перевести карточку лида на единую дизайн-систему портала и привести её к устойчивой адаптивной композиции.

Не копировать внешний HTML-шаблон буквально.

Верстку карточки лида построить по фактическим потребностям страницы с использованием общих токенов и layout-компонентов.

## 1. Корневой layout страницы

Используй PageContent.

Для карточки лида предпочтительно:

- size="compact" или size="default";
- width="full" либо width="wide" в зависимости от текущей оболочки;
- один основной вертикальный scrollbar через AppShell;
- без собственного h-screen;
- без собственного глобального overflow-y-auto;
- без min-width страницы.

Корневой контейнер должен содержать:

- w-full;
- min-w-0;
- безопасные gap;
- отсутствие horizontal overflow.

## 2. Заголовок лида

Собери заголовок через единый presentation-подход:

- номер лида;
- название клиента или организации;
- статус;
- ответственный;
- основные действия.

Используй PageActions для кнопок.

На широком экране:

- информация слева;
- действия справа.

На узком экране:

- действия переносятся;
- основные кнопки остаются видимыми;
- второстепенные действия могут уходить в существующее меню;
- кнопки не обрезаются.

Не добавлять новые действия без существующей механики.

## 3. Стадии лида

Сохрани существующую механику стадий.

Исправить только представление:

- широкая ширина — горизонтальная шкала;
- средняя ширина — компактная шкала;
- узкая ширина — локальная горизонтальная прокрутка внутри блока стадий либо безопасный перенос;
- стадия не должна растягивать всю страницу;
- текст стадий не должен накладываться.

Не создавать отдельное состояние стадий.

## 4. Верхняя информационная зона

Собери блоки:

- основная информация;
- потребность клиента;
- коммерческие параметры;
- при необходимости связанные контакты.

Используй ResponsiveGrid либо аналогичную безопасную сетку.

Режимы:

- широкий desktop — две колонки;
- laptop — две колонки только при достаточной ширине;
- tablet и mobile — одна колонка.

Карточки не должны иметь фиксированную высоту.

Длинные email, ID и URL могут использовать overflow-wrap:anywhere.

Обычный текст должен переноситься по словам.

## 5. Ключевые метрики

Собери метрики через ResponsiveGrid.

Ориентир:

- широкая ширина — 5–6 компактных ячеек;
- laptop — 3 колонки;
- tablet — 2 колонки;
- mobile — 1–2 колонки.

Не использовать жёсткую grid-template-columns: repeat(6, 1fr) на всех разрешениях.

Не создавать фиктивные данные.

Для отсутствующих значений показывать нейтральное состояние.

## 6. История, задачи и заметки

Не держать эти три блока постоянно в трёх колонках.

Использовать следующую композицию:

### Широкая рабочая область

- история — более широкая колонка;
- задачи — средняя колонка;
- заметки — средняя колонка.

Только если каждая карточка получает комфортную ширину.

### Средняя ширина

- история на всю ширину;
- задачи и заметки ниже в две колонки.

### Узкая ширина

- все блоки последовательно в одну колонку.

Минимальная рекомендуемая ширина:

- история — около 320 px;
- задачи — около 300 px;
- заметки — около 320 px.

Не допускать разрыва слов по буквам.

## 7. Прокрутка рабочих блоков

Убрать постоянные соседние scrollbar у:

- истории;
- задач;
- заметок;
- карточки клиента.

Компактный режим должен показывать разумное количество элементов и существующее действие просмотра полного списка.

Локальная прокрутка допустима только:

- в сообщениях чата;
- в dialog;
- в dropdown;
- в раскрытом длинном списке;
- в действительно ограниченной рабочей области, где это обосновано.

Не должно быть нескольких вертикальных scrollbar рядом.

## 8. Коммуникационный блок

Сохранить коммуникации как отдельную рабочую область.

На широком экране:

- данные лида слева;
- коммуникации справа.

На средней ширине:

- коммуникации могут переноситься ниже основной информации;
- не сжимать чат и карточку клиента до нечитабельного состояния.

На tablet и mobile:

- коммуникации занимают всю ширину;
- клиентская карточка располагается над чатом либо сворачивается существующим способом.

Не менять registry коннекторов и статусы каналов.

## 9. Каналы связи

Сохранить реальные данные и текущую механику переключения.

Адаптивность:

- широкий экран — одна строка;
- средний экран — сетка либо компактная лента;
- узкий экран — локальная горизонтальная прокрутка только панели каналов.

Не допускать:

- слишком узких карточек каналов;
- разрыва названий;
- нечитаемых статусов;
- горизонтального overflow всей страницы.

## 10. Карточка клиента

На широком экране карточка клиента может располагаться рядом с чатом.

При недостатке ширины:

- переносить её над чатом;
- либо использовать существующий drawer или collapsible.

Минимальная ширина карточки клиента должна сохранять читаемость данных и кнопок.

Не добавлять новые поля клиента и не создавать новую сущность.

## 11. Чат

Сохранить:

- переключение каналов;
- draft;
- mock-отправку;
- шаблоны;
- вложения;
- статусы сообщений;
- быстрые действия;
- текущие коннекторы.

Исправить только композицию и стили:

- сообщения прокручиваются внутри своей области;
- composer всегда доступен;
- кнопка отправки не обрезается;
- bubbles имеют нормальную ширину;
- входящие слева;
- исходящие справа;
- быстрые действия переносятся или становятся компактной лентой;
- локальный scrollbar только у сообщений.

## 12. Использование токенов

Заменить случайные локальные значения на существующие portal-токены там, где это безопасно.

Использовать единые:

- background;
- surface;
- border;
- text;
- muted text;
- primary;
- success;
- warning;
- danger;
- radius;
- shadow;
- spacing.

Не создавать новый набор цветов для карточки лида.

Не копировать inline CSS из HTML-шаблона.

## 13. Адаптивные режимы

Проверить и настроить минимум следующие ширины:

### 1920 px

- две основные колонки;
- коммуникации справа;
- нижние блоки могут быть в три колонки при достаточной ширине;
- нет чрезмерно широких пустых областей.

### 1440 px

- две основные колонки только если обе остаются читаемыми;
- нижняя зона перестраивается в 1+2 при необходимости;
- карточка клиента не сжимается;
- каналы читаемы.

### 1280 px

- допускается перенос коммуникаций ниже основной части;
- история, задачи и заметки не должны быть в трёх узких колонках;
- метрики перестраиваются.

### 1024 px

- одна основная колонка;
- коммуникации ниже;
- верхние информационные карточки могут быть в две колонки только при достаточной ширине;
- без horizontal overflow.

### 768 px

- одна колонка;
- действия переносятся;
- стадии имеют локальную прокрутку или компактный вид;
- метрики в 2 колонки;
- чат занимает всю ширину.

### 390 px

- одна колонка;
- кнопки не обрезаются;
- метрики в 1–2 колонки;
- bubbles до 90–92% ширины;
- composer полностью доступен;
- нет horizontal overflow страницы.

## Ручная проверка

Проверить страницу:

/sales/leads/lead-1

На ширинах:

- 1920;
- 1440;
- 1280;
- 1024;
- 768;
- 390.

Проверить:

1. Нет horizontal overflow страницы.
2. Нет второго основного вертикального scrollbar.
3. Нет нескольких соседних scrollbar.
4. Обычный текст не разбивается по буквам.
5. Кнопки не обрезаются.
6. Стадии не накладываются.
7. Метрики корректно перестраиваются.
8. История, задачи и заметки читаемы.
9. Коммуникации не сжимают основную область.
10. Карточка клиента читаема.
11. Чат работает.
12. Composer доступен.
13. В Console нет ошибок.
14. Нет hydration mismatch.

## Готово, если

- карточка лида использует компоненты и токены дизайн-системы;
- layout устойчив на всех указанных разрешениях;
- нет горизонтального overflow;
- нет лишних scrollbar;
- вся текущая логика сохранена;
- данные не подменены статическими значениями;
- коннекторы продолжают работать;
- typecheck, lint и build проходят.
