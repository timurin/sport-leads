# v0.5.12.1 — Корректная миграция карточки лида

## Проблема

В задаче v0.5.12 изменения были внесены преимущественно в:

frontend/components/sales/lead-workspace.tsx

Этот компонент содержит Kanban, метрики и вкладки списка лидов и не является карточкой конкретного лида по маршруту:

/sales/leads/[leadId]

Нужно определить фактическую страницу и компоненты карточки конкретного лида, затем выполнить миграцию именно там.

## Цель

Перевести страницу конкретного лида на единую дизайн-систему проекта без изменения списка лидов, Kanban, данных, типов и бизнес-логики.

## Перед началом

Найди маршрут:

/sales/leads/[leadId]

Определи:

1. Файл route/page.
2. Главный компонент карточки конкретного лида.
3. Компоненты заголовка, стадий, информации, метрик, задач, заметок, timeline и коммуникаций.
4. Где сейчас задаются layout, grid, height и overflow.
5. Какие изменения задачи v0.5.12 были ошибочно внесены в страницу списка лидов.

Сначала покажи карту:

маршрут → page → главный компонент → дочерние блоки.

Не начинай изменения, пока не определён фактический компонент карточки конкретного лида.

## Исправление изменений списка лидов

Проверь изменения в:

frontend/components/sales/lead-workspace.tsx
frontend/components/kanban/kanban-card.tsx
frontend/components/kanban/kanban-types.ts

Не удаляй полезные изменения автоматически.

Откати только те изменения, которые:

- были сделаны в рамках v0.5.12;
- относятся к ошибочной миграции списка лидов;
- не требуются ранее реализованной механике;
- не используются другими страницами.

Особенно проверь добавление:

href: `/sales/leads/${lead.id}`

Не удаляй его, если переход по Kanban-карточке был ранее согласован или уже используется. В таком случае оставь и отметь это в отчёте как изменение вне текущей миграции.

Не изменяй список лидов дополнительно.

## Карточка конкретного лида

На фактической странице /sales/leads/[leadId] используй существующие компоненты дизайн-системы:

- PageContent;
- PageActions;
- ResponsiveGrid;
- portal-токены;
- существующие UI-компоненты карточек, метрик и вкладок.

Не создавать новую параллельную страницу.

Не заменять рабочие компоненты статической разметкой.

## Корневой layout

Карточка должна:

- занимать доступную ширину;
- использовать w-full и min-w-0;
- не использовать собственный h-screen;
- не создавать второй основной overflow-y-auto;
- прокручиваться через data-app-shell-main;
- не создавать горизонтальный overflow.

## Заголовок

Сохранить:

- номер лида;
- клиента или организацию;
- статус;
- ответственного;
- существующие действия.

Использовать PageActions.

На узком экране действия должны переноситься или уходить в существующее меню.

Не добавлять новые рабочие действия.

## Стадии

Сохранить существующую механику.

Исправить только адаптивное представление:

- desktop — горизонтально;
- tablet — компактно;
- mobile — локальная горизонтальная прокрутка внутри блока стадий.

Не создавать новое состояние.

## Информационные блоки

Собрать через безопасные grid-компоненты:

- основная информация;
- контакты;
- потребность;
- коммерческие параметры.

Правила:

- широкая ширина — две колонки;
- средняя ширина — две колонки только при достаточном месте;
- узкая ширина — одна колонка;
- без фиксированной высоты;
- обычный текст переносится по словам;
- email и технические значения могут использовать overflow-wrap:anywhere.

## Метрики

Использовать адаптивную сетку:

- desktop — до 5–6 колонок;
- laptop — 3 колонки;
- tablet — 2 колонки;
- mobile — 1–2 колонки.

Не менять расчёты и данные.

## История, задачи и заметки

Не держать три блока в узких колонках.

Композиция:

- широкая область — три колонки только при достаточной ширине;
- средняя область — история на всю ширину, задачи и заметки ниже;
- узкая область — одна колонка.

Не добавлять постоянные scrollbar каждому блоку.

Полные списки должны использовать существующие вкладки, dialogs, drawers или раскрытие.

## Коммуникации

Сохранить:

- registry коннекторов;
- статусы каналов;
- переключение;
- сообщения;
- draft;
- шаблоны;
- вложения;
- mock-отправку;
- быстрые действия.

На широком экране коммуникации могут быть справа.

При недостаточной ширине коммуникации переходят ниже данных лида.

Не сжимать клиентскую карточку и чат до нечитабельного состояния.

Локальная вертикальная прокрутка разрешена только области сообщений.

## Данные и типы

Не изменять:

frontend/types/sales.ts
frontend/lib/demo-data/sales.ts

кроме случая, когда это строго необходимо для исправления TypeScript-ошибки, вызванной текущей версткой.

Запрещено:

- добавлять поля ради дизайна;
- добавлять новые бизнес-значения;
- менять priority;
- расширять типы сообщений;
- создавать статические данные под внешний вид.

## Глобальные стили

Сохранить уже созданные portal-токены.

Проверь оставшуюся часть frontend/app/globals.css.

В глобальном CSS допустимы:

- portal-токены;
- portal-grid-small;
- portal-grid-medium;
- portal-grid-large;
- общие layout-правила;
- общие UI utilities.

Не добавлять большой набор классов, относящихся только к карточке лида.

Локальные стили карточки размещать рядом с её компонентами либо выражать Tailwind-классами.

## Адаптивность

Проверить:

- 1920 px;
- 1440 px;
- 1280 px;
- 1024 px;
- 768 px;
- 390 px.

На всех ширинах:

1. Нет horizontal overflow.
2. Нет второго основного scrollbar.
3. Обычный текст не разбивается по буквам.
4. Кнопки не обрезаются.
5. Стадии не перекрываются.
6. Метрики перестраиваются.
7. История, задачи и заметки читаемы.
8. Коммуникации не сжимают данные лида.
9. Карточка клиента читаема.
10. Composer полностью доступен.

## Сохранение механики

Проверить:

- смену статуса;
- смену ответственного;
- редактирование клиента;
- контакты;
- коммерческие данные;
- задачи;
- заметки;
- timeline;
- каналы;
- сообщения;
- draft;
- шаблоны;
- вложения;
- lead-1;
- lead-2;
- lead-999;
- loading;
- error;
- not-found.

- не менять типы и demo-данные ради дизайна;
- не переделывать список лидов;

- не делать commit.

3. Изменения списка лидов, оставленные или отменённые.
4. Использованные элементы дизайн-системы.
5. Изменённые файлы.
6. Результаты адаптивной проверки.
