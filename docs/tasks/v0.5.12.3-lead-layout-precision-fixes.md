# v0.5.12.3 — Точечные исправления верстки карточки лида

## Контекст

Карточка конкретного лида успешно переведена на общую дизайн-систему.

Основная композиция и существующая механика должны быть сохранены.

По результатам визуальной проверки обнаружены точечные проблемы:

- пересечение подписей и значений в карточке «Потребность клиента»;
- чрезмерно узкая рабочая область на широких мониторах;
- нестабильные пропорции основной сетки;
- чрезмерное сжатие коммуникационного блока на промежуточных разрешениях.

## Важное замечание о кодировке

Русский текст в браузере отображается правильно.

Вывод Windows PowerShell Get-Content без параметра -Encoding может показывать UTF-8 как mojibake.

Не выполнять массовую перекодировку файлов.

Для проверки содержимого использовать:

Get-Content <файл> -Encoding UTF8

Не добавлять BOM и не менять кодировку корректных файлов.

## 1. Исправить карточку «Потребность клиента»

Устранить пересечение полей:

- направление;
- вид спорта;
- категория;
- тип продукции;
- количество изделий;
- количество комплектов;
- размеры.

Каждая пара label/value должна иметь собственное место.

Не использовать сетку, в которой две пары label/value помещаются в одну строку при недостаточной ширине.

На широком экране допустимы две внутренние колонки только при достаточной ширине карточки.

На средней и узкой ширине использовать одну колонку пар label/value.

Рекомендуемый принцип:

- label имеет стабильную, но не чрезмерную ширину;
- value использует min-width: 0;
- обычный текст переносится по словам;
- для длинных технических значений разрешён overflow-wrap:anywhere;
- значения не должны накладываться на соседние строки.

Не менять сами поля и их значения.

## 2. Ширина рабочей области

Для detail-страницы лида использовать доступную ширину AppShell.

Проверь использование PageContent.

Для карточки лида предпочтительно:

- width="full";
- разумные горизонтальные padding;
- отсутствие искусственного ограничения всей страницы значением --portal-content-max.

Не менять глобальное значение --portal-content-max для остальных страниц без необходимости.

Разрешено задать width="full" только странице конкретного лида.

На очень широком мониторе интерфейс не должен превращаться в узкую центральную колонку с большими пустыми полями.

При этом не растягивать текстовые значения до неудобной длины: ограничивать ширину можно внутри отдельных карточек.

## 3. Основная desktop-сетка

Настроить основную композицию по доступной ширине контейнера.

На широком desktop:

- левая информационная область ориентировочно 55–60%;
- правая коммуникационная область ориентировочно 40–45%;
- обе области используют min-width: 0.

Не задавать правой области ширину меньше комфортной ширины чата.

Использовать minmax с безопасными минимальными значениями.

При нехватке ширины коммуникационный блок должен переноситься ниже, а не сжимать карточки до пересечения текста.

## 4. Верхние информационные карточки

Основная информация и потребность клиента:

- на широком desktop — две колонки;
- потребность может получать немного больше ширины;
- на средней ширине — одна колонка, если пары полей начинают пересекаться;
- без фиксированной высоты;
- нижние границы карточек не обязаны совпадать ценой пустого пространства или обрезания текста.

## 5. Коммуникационная область

Сохранить текущую механику.

Проверить:

- карточки каналов;
- карточку клиента;
- чат;
- composer;
- быстрые действия.

На средней ширине:

- карточка клиента может переноситься над чатом;
- каналы могут перестраиваться в несколько строк;
- чат не должен становиться чрезмерно узким.

Локальный вертикальный scrollbar оставить только области сообщений.

## 6. Адаптивные режимы

Проверить:

- 2560 px;
- 1920 px;
- 1600 px;
- 1440 px;
- 1280 px;
- 1024 px;
- 768 px;
- 390 px.

На 2560 и 1920 px:

- рабочая область использует доступную ширину;
- нет чрезмерных пустых боковых полей;
- информационные карточки читаемы;
- чат имеет достаточную ширину.

На 1600 и 1440 px:

- поля потребности не пересекаются;
- коммуникации не сжимают левую часть;
- стадии остаются читаемыми.

На 1280 и 1024 px:

- коммуникации могут переноситься ниже;
- основные карточки перестраиваются;
- нет horizontal overflow.

На 768 и 390 px:

- одна колонка;
- действия переносятся;
- стадии используют локальный горизонтальный scroll при необходимости;
- composer полностью доступен;
- страница не имеет горизонтальной прокрутки.
