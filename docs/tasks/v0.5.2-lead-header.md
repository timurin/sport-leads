# Задача Codex: v0.5.2 — Шапка карточки лида

## Контекст

Проект: `sport-leads`

Завершённая задача:

```text
v0.5.1 — Каркас страницы лида
```

Текущая задача:

```text
v0.5.2 — Lead Header
```

В проекте уже создан маршрут карточки лида:

```text
/sales/leads/[leadId]
```

Также созданы базовые компоненты страницы лида, состояния загрузки, ошибки и отсутствия данных.

На этом этапе нужно доработать только верхнюю часть карточки лида.

Не переходи к реализации данных клиента, коммерческих параметров, задач, сообщений, файлов и заказа покупателя.

---

# Цель задачи

Создать полноценную шапку карточки лида, которая показывает ключевую информацию и основные действия менеджера.

Шапка должна быть визуально близка к интерфейсу CRM и использовать текущую дизайн-систему проекта.

---

# 1. Перед началом изучи текущую реализацию

Изучи:

* `frontend/app/(workspace)/sales/leads/[leadId]/`;
* `frontend/components/sales/lead-page.tsx`;
* `frontend/components/sales/lead-back-button.tsx`;
* `frontend/components/sales/lead-page-skeleton.tsx`;
* `frontend/components/sales/lead-page-state.tsx`;
* `frontend/lib/sales/lead-details.ts`;
* существующие компоненты кнопок;
* существующие badge-компоненты;
* dropdown/menu-компоненты;
* компоненты выбора пользователя;
* компоненты выбора статуса;
* текущие типы лидов;
* стадии существующей воронки;
* модель данных лида.

Переиспользуй существующие компоненты.

Не создавай новый Button, Badge, Dropdown или Select, если такие компоненты уже есть.

Перед изменениями кратко сообщи:

* какие компоненты найдены;
* какие данные лида уже доступны;
* какие файлы будут изменены;
* потребуется ли расширение mock-данных.

# 2. Создай или доработай компонент LeadHeader

Рекомендуемое название:

```text
LeadHeader
```

Компонент должен отвечать только за шапку лида.

Не помещай всю страницу в один большой компонент.

Возможный файл:

```text
frontend/components/sales/lead-header.tsx
```

Используй другую структуру только в том случае, если она лучше соответствует существующей архитектуре проекта.

Покажи:

* номер лида;

# 4. Структура шапки

На desktop шапка должна включать три логические зоны.

## Левая зона

* кнопка возврата;
* номер лида;
* название клиента;
* контактное лицо.

## Центральная зона

* статус;
* ответственный;
* источник;
* даты.

## Правая зона

* сумма;
* вероятность;
* основные кнопки действий;
* меню дополнительных действий.

Допускается другая компоновка, если она лучше соответствует текущему интерфейсу проекта.

---

# 5. Кнопка возврата

Используй уже созданный компонент возврата.

Кнопка должна:

1. возвращать пользователя на предыдущую страницу через историю браузера;
2. при невозможности возврата вести на:

```text
/sales/leads
```

Не дублируй компонент возврата.

---

# 6. Статус лида

Покажи статус в виде цветного badge или кнопки-select.

Статус должен использовать реальные стадии существующей воронки.

Не создавай отдельный несвязанный список статусов.

Найди текущие определения стадий в проекте и переиспользуй:

* код стадии;
* название;
* цвет;
* порядок.

На текущем этапе допускается выбор нового статуса в интерфейсе без сохранения на backend только в том случае, если реального endpoint обновления нет.

При отсутствии API:

* явно оставь действие локальным;
* не создавай фиктивный backend endpoint;
* не показывай ложное сообщение об успешном сохранении;
* после перезагрузки допустим возврат исходного значения.

Если API обновления статуса уже существует, используй его.

---

# 7. Ответственный менеджер

Покажи:

* имя ответственного;
* аватар или инициалы, если такие элементы уже используются;
* dropdown или select для смены ответственного.

Если в проекте уже существует список пользователей или менеджеров, используй его.

Если API отсутствует:

* используй текущего ответственного;
* допускается локальное переключение;
* не добавляй новый backend endpoint.

Не создавай фиктивный список сотрудников без проверки проекта.

При необходимости временно используй минимальный mock-набор, явно помеченный как mock.

---

# 8. Источник лида

Покажи источник:

```text
Форма сайта
```

или реальное значение из данных лида.

Источник пока не должен редактироваться в шапке.

Если источник отсутствует, показать:

```text
Источник не указан
```

---

# 9. Сумма и вероятность

Предполагаемая сумма должна форматироваться через локаль `ru-RU`.

Пример:

```text
480 000 ₽
```

Вероятность должна отображаться в процентах:

```text
40%
```

Границы вероятности:

```text
0–100
```

Не допускай отображение:

* `undefined`;
* `NaN`;
* отрицательного значения;
* значения больше 100%.

Если данных нет:

```text
Сумма не указана
```

```text
Вероятность не указана
```

---

# 10. Даты

Покажи:

* дату создания;
* дату последней активности.

Форматируй даты по локали `ru-RU`.

Дата создания:

```text
16 июля 2026
```

Последняя активность может отображаться в относительном формате:

```text
Сегодня, 14:30
Вчера, 18:10
15 июля 2026, 12:40
```

Используй существующую утилиту форматирования дат, если она уже есть.

Не добавляй тяжёлую библиотеку только ради форматирования.

Учитывай timezone проекта.

Не вызывай hydration mismatch из-за различий времени между сервером и клиентом.

Если относительная дата может вызвать несовпадение SSR и клиента, используй безопасный стабильный формат или клиентский компонент.

---

# 11. Основные действия

Добавь кнопки:

```text
Изменить статус
```

```text
Назначить ответственного
```

```text
Добавить задачу
```

```text
Создать заказ
```

```text
Ещё
```

На текущем этапе:

* изменение статуса может работать локально или через существующий API;
* назначение ответственного может работать локально или через существующий API;
* «Добавить задачу» может быть disabled или показывать понятную заглушку;
* «Создать заказ» может быть disabled или показывать понятную заглушку;
* меню «Ещё» должно открываться.

Не реализуй полноценную задачу или заказ в рамках этой версии.

Не используй `alert()`.

Для ещё не реализованных функций используй:

* disabled-состояние;
* tooltip;
* существующий toast;
* аккуратную подпись «Будет доступно в следующем этапе».

---

# 12. Меню дополнительных действий

В меню «Ещё» предусмотри пункты:

* редактировать лид;
* копировать ссылку;
* закрыть лид;
* удалить лид.

На текущем этапе реально реализуй только безопасное действие:

```text
Копировать ссылку
```

Остальные действия:

* должны быть disabled;
* либо помечены как будущая функция.

Не реализуй удаление лида.

Не реализуй закрытие лида через backend.

Копирование ссылки должно:

* использовать текущий URL;
* работать только на клиенте;
* показывать существующий toast или визуальное подтверждение;
* корректно обрабатывать ошибку clipboard API.

---

# 13. Типы данных

Доработай тип `LeadDetails`, если это необходимо.

Предполагаемые поля:

```ts
type LeadDetails = {
  id: string
  number?: string
  title: string
  contactName?: string
  status: LeadStatus
  responsible?: LeadResponsible
  source?: string
  createdAt?: string
  lastActivityAt?: string
  estimatedAmount?: number
  probability?: number
}
```

Не копируй этот тип буквально, если в проекте уже есть подходящие типы.

Не используй `any`.

Не создавай дублирующие модели лида.

# 15. Loading skeleton

Обнови skeleton шапки, чтобы он соответствовал новой структуре.

Skeleton должен включать:

* кнопку возврата;
* название;
* строку контакта;
* badge статуса;
* ответственного;
* сумму;
* несколько кнопок действий.

Не меняй skeleton остальной страницы без необходимости.

---

# 16. Адаптивность

На широком экране:

* заголовок и данные располагаются компактно;
* действия находятся справа;
* шапка не занимает чрезмерную высоту.

На среднем экране:

* кнопки могут переноситься на следующую строку;
* поля не перекрываются.

На мобильной ширине:

* информация располагается вертикально;
* кнопки занимают доступную ширину;
* меню остаётся доступным;
* нет горизонтальной прокрутки.

Основной приоритет — desktop.

---

# 17. Доступность

Проверь:

* у кнопок есть доступные названия;
* dropdown работает с клавиатуры;
* focus-state виден;
* цвет статуса не является единственным способом передачи информации;
* disabled-кнопки корректно помечены;
* кнопки без текста имеют `aria-label`;
* ссылка возврата доступна с клавиатуры.

---

# 18. Ограничения

В этой задаче не выполнять:

* полноценное редактирование лида;
* формы клиента;
* коммерческие поля;
* создание задач;
* создание заказа покупателя;
* отправку сообщений;
* загрузку файлов;
* аудит;
* права доступа;
* миграции базы данных;
* создание новых backend endpoints без необходимости;
* масштабный рефакторинг страницы;
* изменение канбана, если это не требуется для шапки.

---

# 19. Проверки

После реализации выполни из папки `frontend` доступные команды:

```text
npm run lint
npm run typecheck
npm run build
```

Если `typecheck` отсутствует:

```text
npx tsc --noEmit
```

Используй команды, реально объявленные в `package.json`.

Дополнительно проверь:

1. Карточка лида открывается.
2. Шапка отображает номер и название лида.
3. Статус имеет правильное название и цвет.
4. Ответственный отображается.
5. Сумма форматируется в рублях.
6. Вероятность отображается корректно.
7. Даты не вызывают hydration mismatch.
8. Меню «Ещё» открывается.
9. Копирование ссылки работает.
10. Неактивные действия не вызывают ошибок.
11. Loading skeleton соответствует новой шапке.
12. На узком экране отсутствует горизонтальная прокрутка.
13. В браузерной консоли нет ошибок.
14. Страница лидов и канбан продолжают работать.

# Критерии готовности

Задача считается выполненной, если:

* создан полноценный `LeadHeader`;
* отображаются ключевые данные лида;
* статус использует стадии текущей воронки;
* ответственный отображается корректно;
* сумма, вероятность и даты форматируются правильно;
* доступны основные действия;
* меню дополнительных действий работает;
* копирование ссылки работает;
* отсутствуют hydration-ошибки;
* интерфейс адаптивен;
* lint, typecheck и build проходят успешно.
