# Задача Codex: v0.5.3 — Данные клиента в карточке лида

## Контекст

Проект: `sport-leads`

Завершённые задачи:

* `v0.5.1 — Каркас страницы лида`
* `v0.5.2 — Шапка карточки лида`

Текущая задача:

* `v0.5.3 — Lead Customer Details`

Маршрут карточки лида уже существует:

```text
/sales/leads/[leadId]
```

Шапка лида уже отображает основные данные, статус, ответственного, источник, сумму и вероятность.

На этом этапе нужно реализовать содержимое блока данных клиента и контактных лиц.

Не переходи к коммерческим параметрам, задачам, сообщениям, файлам и заказам.

---

# Цель задачи

Создать полноценный блок данных клиента внутри карточки лида.

Блок должен позволять менеджеру просматривать и редактировать:

* сведения о клиенте;
* основное контактное лицо;
* дополнительные контакты;
* телефоны;
* email;
* адресные и регистрационные данные;
* предпочтительный канал связи.

Используй существующую дизайн-систему проекта.

## Основной контакт

* имя;
* должность;
* телефон;

## Комментарий

* текст комментария;
* пустое состояние при отсутствии.

Добавь кнопку:

```text
Редактировать
```

# 5. Режим редактирования

По нажатию «Редактировать» переключай блок в режим формы.

Добавь поля:

* тип клиента;
* название организации;
* имя контактного лица;
* должность;
* телефон;
* email;
* сайт;
* город;
* регион;
* адрес;
* ИНН;
* комментарий;
* предпочтительный канал связи.

Кнопки:

```text
Сохранить
Отмена
```

На текущем этапе:

* если API обновления клиента существует, используй его;
* если API отсутствует, сохраняй изменения локально в состоянии страницы;
* не создавай фиктивный backend endpoint;
* не показывай ложное сообщение о сохранении в базе;
* явно укажи в коде, что используется локальный режим.

После локального сохранения данные должны обновляться на странице до её перезагрузки.

Кнопка «Отмена» должна возвращать исходные значения.

---

# 6. Валидация

Добавь базовую клиентскую валидацию.

## Email

* проверять корректный формат;
* пустое значение допустимо.

## Телефон

* не требовать строгого российского формата;
* разрешить `+`, цифры, пробелы, скобки и дефисы;
* пустое значение допустимо.

## ИНН

Допустимые значения:

* 10 цифр для организации;
* 12 цифр для ИП или физического лица;
* пустое значение допустимо.

Не блокируй интерфейс чрезмерно строгой валидацией.

Ошибки должны отображаться рядом с полями.

Не используй `alert()`.

---

# 7. Контактные лица

Добавь секцию:

```text
Контактные лица
```

Один контакт должен содержать:

* id;
* имя;
* должность;
* телефон;
* email;
* мессенджер;
* основной контакт;
* предпочтительный канал связи.

Поддержи несколько контактов.

Для каждого контакта покажи:

* имя;
* должность;
* телефон;
* email;
* канал связи;
* отметку «Основной».

# 9. Редактирование контакта

Для каждого контакта добавь действие:

```text
Редактировать
```

Используй тот же Dialog.

При открытии форма должна быть заполнена текущими данными.

После сохранения локальные данные обновляются.

---

# 10. Удаление контакта

Добавь действие:

```text
Удалить
```

Перед удалением покажи подтверждение через существующий Dialog.

Не используй `window.confirm`, если в проекте есть компонент подтверждения.

Нельзя удалить последний основной контакт без назначения другого основного контакта.

Если удаляется основной контакт и есть другие контакты:

* либо запрети удаление;
* либо предложи назначить другой контакт основным.

На текущем этапе достаточно безопасно запретить удаление основного контакта с понятным сообщением.

---

# 11. Основной контакт

В каждый момент времени только один контакт может быть основным.

При назначении нового основного контакта:

* предыдущий основной контакт автоматически теряет статус;
* данные в шапке лида должны отображать нового основного контакта, если архитектура страницы позволяет безопасно связать состояние;
* если это потребует значительного рефакторинга, обнови только блок клиента и оставь комментарий для следующей задачи.

Не создавай глобальный state manager только ради этой функции.

Расширь существующий `LeadDetails`, если требуется.

Предполагаемая структура:

```ts
type LeadCustomerType = "person" | "sole_proprietor" | "company"

type LeadContact = {
  id: string
  name: string
  position?: string
  phone?: string
  email?: string
  messenger?: string
  preferredChannel?: CommunicationChannel
  isPrimary: boolean
}

type LeadCustomer = {
  type?: LeadCustomerType
  organizationName?: string
  website?: string
  city?: string
  region?: string
  address?: string
  taxId?: string
  comment?: string

}
```

Не копируй эту модель буквально, если в проекте уже существуют подходящие типы.

Не используй `any`.

# 17. Состояние сохранения

Если используется API:

* показывай loading на кнопке;
* блокируй повторное сохранение;
* обрабатывай ошибку;
* не закрывай форму при ошибке.

Если используется локальное состояние:

* обновляй UI сразу;
* не показывай сетевую загрузку;
* можно показать нейтральное уведомление:

```text
Изменения сохранены только локально
```

Используй существующий toast, если он есть.

Если toast отсутствует, не создавай новую глобальную систему уведомлений в этой задаче.

# 18. Адаптивность

На desktop:

* сведения о клиенте отображаются в 2–3 колонки;
* контакты расположены компактными карточками или строками.

На узком экране:

* поля и данные переходят в одну колонку;
* кнопки не выходят за границы;
* Dialog помещается в viewport;
* отсутствует горизонтальная прокрутка.

---

# 19. Доступность

* у всех полей есть `label`;
* ошибки связаны с полями через `aria-describedby`;
* Dialog имеет заголовок;
* focus возвращается на кнопку после закрытия;
* кнопки без текста имеют `aria-label`;
* формы доступны с клавиатуры;
* кликабельные телефон и email имеют понятный текст.

# Критерии готовности

Задача считается выполненной, если:

* блок данных клиента заменяет заглушку;
* сведения клиента отображаются корректно;
* режим просмотра и редактирования работают;
* поддерживаются несколько контактных лиц;
* контакт можно добавить и изменить;
* удаление выполняется безопасно;
* поддерживается один основной контакт;
* реализована базовая валидация;
* отсутствуют ошибки TypeScript и hydration;
* интерфейс адаптивен;
* lint, typecheck и build проходят.
