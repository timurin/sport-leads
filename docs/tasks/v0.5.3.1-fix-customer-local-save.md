# Задача Codex: v0.5.3.1 — Исправить локальное сохранение клиента и контактов

## Контекст

Проект: `sport-leads`

Текущая карточка лида:

```text
/sales/leads/[leadId]
```

После выполнения задачи `v0.5.3` обнаружен дефект:

* отредактированные данные клиента не сохраняются в интерфейсе;
* добавленный контакт не появляется или исчезает после закрытия формы;
* изменения откатываются к исходным mock-данным;
* основной контакт не синхронизируется с шапкой лида.

На этом этапе backend API сохранения отсутствует или не используется.

Требуется исправить локальное состояние страницы. Данные должны сохраняться до перезагрузки браузера.

Не добавляй backend endpoint, базу данных, localStorage или глобальный state manager.

---

# Цель

Сделать единым источником данных карточки лида локальное состояние на уровне страницы или общего клиентского контейнера.

Изменения клиента и контактов должны сразу отображаться:

* в блоке данных клиента;
* в списке контактов;
* в шапке карточки, когда изменяется основной контакт.

После обновления страницы допускается возврат исходных mock-данных.

---

# 1. Найди причину

Изучи:

* `frontend/components/sales/lead-page.tsx`;
* `frontend/components/sales/lead-header.tsx`;
* `frontend/components/sales/lead-customer-details.tsx`;
* `frontend/lib/sales/lead-details.ts`;
* компоненты форм и диалогов клиента;
* обработчики сохранения и добавления контакта.

Определи:

* где хранится состояние формы;
* где хранится состояние клиента;
* не пересоздаётся ли состояние из props после каждого render;
* вызываются ли callbacks сохранения;
* не закрывается ли Dialog до обновления состояния;
* не используется ли изменение объекта или массива напрямую.

Перед исправлением кратко сообщи причину дефекта.

---

# 2. Единое локальное состояние

Создай один локальный источник состояния для данных лида.

Предпочтительная схема:

```ts
const [lead, setLead] = useState<LeadDetails>(initialLead)
```

Состояние должно находиться на уровне, который одновременно управляет:

* `LeadHeader`;
* `LeadCustomerDetails`.

Передавай дочерним компонентам:

```ts
lead
onCustomerChange
onContactsChange
```

или один безопасный callback:

```ts
onLeadChange
```

Не создавай глобальный store.

Не дублируй состояние клиента одновременно в нескольких компонентах.

---

# 3. Сохранение данных клиента

После нажатия «Сохранить»:

1. Проверь валидацию.
2. Создай новый объект данных клиента.
3. Обнови общее состояние лида через setter или callback.
4. Закрой режим редактирования только после успешного обновления состояния.
5. Покажи новые значения в режиме просмотра.

Не мутируй существующий объект.

Плохо:

```ts
lead.customer.name = value
```

Правильно:

```ts
setLead((current) => ({
  ...current,
  customer: {
    ...current.customer,
    name: value,
  },
}))
```

Кнопка «Отмена» не должна менять общее состояние.

---

# 4. Добавление контакта

После нажатия «Добавить»:

1. Проверь форму.
2. Создай уникальный локальный ID.
3. Добавь новый контакт в новый массив.
4. Обнови общее состояние лида.
5. Очисти форму.
6. Закрой Dialog.
7. Сразу покажи новый контакт в списке.

Не используй индекс массива как постоянный ID.

Допускается:

```ts
crypto.randomUUID()
```

Предусмотри fallback, если API недоступен.

---

# 5. Редактирование контакта

После сохранения редактирования:

* найди контакт по `id`;
* замени его новым объектом;
* не изменяй массив напрямую;
* сохрани порядок контактов;
* сразу обнови отображение.

---

# 6. Удаление контакта

Удаление должно:

* обновлять общий массив контактов;
* работать для неосновного контакта;
* запрещать удаление основного контакта согласно текущим правилам;
* не восстанавливать удалённый контакт после закрытия Dialog.

---

# 7. Основной контакт

При назначении контакта основным:

1. Установи `isPrimary: false` у остальных контактов.
2. Установи `isPrimary: true` у выбранного контакта.
3. Обнови общее состояние лида.
4. Обнови контакт в `LeadHeader`.

В шапке должны измениться доступные поля основного контакта, например:

* имя;
* телефон;
* email.

Не создавай отдельное состояние основного контакта в шапке.

`LeadHeader` должен получать актуальный `lead` через props.

---

# 8. Инициализация состояния

Начальное состояние должно создаваться один раз из загруженных или mock-данных.

Не выполняй безусловно:

```ts
setLead(initialLead)
```

в обычном render или effect, который срабатывает после каждого локального изменения.

Если данные загружаются асинхронно, инициализируй состояние только при смене `leadId` или при завершении новой загрузки.

Не допускай сброса локальных изменений при открытии и закрытии Dialog.

---

# 9. Контролируемые формы

Проверь, что поля формы являются контролируемыми:

```ts
value={form.name}
onChange={...}
```

При открытии редактирования:

* копируй текущие данные в draft-состояние формы;
* не редактируй объект из props напрямую.

При отмене:

* удаляй draft;
* исходные данные остаются без изменений.

---

# 10. Ограничения

Не выполнять:

* backend API;
* сохранение в PostgreSQL;
* localStorage;
* sessionStorage;
* глобальный state manager;
* React Context для всей CRM;
* изменение коммерческого блока;
* задачи;
* сообщения;
* заказ покупателя;
* commit;
* переход к `v0.5.4`.

---

# 11. Проверки

Из папки `frontend` выполни:

```text
npm run lint
npm run typecheck
npm run build
```

Если `typecheck` отсутствует:

```text
npx tsc --noEmit
```

Обязательно вручную или логически проверь:

1. Редактирование названия организации сохраняется в UI.
2. Редактирование города сохраняется в UI.
3. «Отмена» не сохраняет изменения.
4. Новый контакт появляется сразу.
5. Новый контакт остаётся после повторного открытия Dialog.
6. Контакт можно отредактировать.
7. Неосновной контакт можно удалить.
8. Основной контакт нельзя удалить.
9. Новый основной контакт отображается в шапке.
10. Закрытие Dialog не сбрасывает данные.
11. Перерисовка страницы не возвращает исходные props.
12. После полной перезагрузки допустим возврат mock-данных.
13. В Console нет ошибок React.
14. Нет изменения state во время render.
15. Нет мутации props.
