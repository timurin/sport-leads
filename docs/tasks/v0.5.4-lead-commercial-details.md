# Задача Codex: v0.5.4 — Коммерческие параметры лида

## Контекст

Проект: `sport-leads`.

Завершены этапы:

* `v0.5.1 — Каркас страницы лида`;
* `v0.5.2 — Шапка карточки лида`;
* `v0.5.3 — Данные клиента и контакты`;
* `v0.5.3.1 — Локальное состояние клиента и контактов`.

Карточка лида доступна по маршруту:

```text
/sales/leads/[leadId]
```

Сейчас данные работают на mock-источнике и локальном состоянии страницы. Полноценное сохранение в PostgreSQL будет реализовано отдельным backend-этапом.

Текущая задача должна реализовать блок коммерческих параметров лида.

Не переходи к задачам, коммуникациям, файлам или созданию заказа покупателя.

---

# Цель

Заменить заглушку «Коммерческие параметры» полноценным блоком, содержащим информацию о потребности клиента и предварительных условиях будущего заказа.

Менеджер должен иметь возможность:

* просматривать коммерческие параметры;
* переходить в режим редактирования;
* изменять значения;
* сохранять их в локальном состоянии карточки;
* отменять несохранённые изменения.

После ухода со страницы и повторного открытия допускается восстановление исходных mock-данных.

Не использовать `localStorage`.

# 7. Режим просмотра

По умолчанию блок открывается в режиме просмотра.

Структура может быть разделена на секции:

* потребность;
* количество и бюджет;
* сроки;
* доставка;
* источник и приоритет.

Для отсутствующих значений показывай:

```text
Не указано
```

Не отображай:

* `undefined`;
* `null`;
* `NaN`;
* пустые подписи;
* некорректные даты.

Добавь кнопку:

```text
Редактировать
```

---

# 8. Режим редактирования

После нажатия «Редактировать» покажи форму.

Кнопки:

```text
Сохранить
Отмена
```

Требования:

* форма использует draft-состояние;
* изменение полей не должно сразу менять общий объект;
* «Отмена» возвращает исходные значения;
* «Сохранить» обновляет единое локальное состояние карточки;
* после сохранения блок возвращается в режим просмотра;
* данные сохраняются до ухода с карточки.

Не создавай отдельный store.

Используй текущий общий `lead` state и callback обновления.

---

# 9. Количество

Поля количества должны принимать только целые неотрицательные значения.

Правила:

* пустое значение допустимо;
* минимальное значение — `1`, если поле заполнено;
* не допускай отрицательные значения;
* не показывай `0 изделий`, если значение отсутствует.

При выводе используй правильную подпись:

```text
1 изделие
2 изделия
5 изделий
```

Допускается простая утилита склонения без новой библиотеки.

---

# 10. Бюджет и сумма

Финансовые поля:

* предварительный бюджет;
* предполагаемая сумма.

Требования:

* хранить числом;
* отображать через `Intl.NumberFormat("ru-RU")`;
* валюта — рубли;
* не допускать отрицательных значений;
* не сохранять `NaN`.

Пример:

```text
480 000 ₽
```

Если значение отсутствует:

```text
Не указано
```

Предполагаемая сумма должна синхронизироваться с суммой в `LeadHeader`.

Не создавай отдельное состояние суммы в шапке.

---

# 11. Вероятность сделки

Поле принимает значение от `0` до `100`.

При сохранении:

* меньше `0` привести к `0` либо показать ошибку;
* больше `100` привести к `100` либо показать ошибку;
* пустое значение допустимо.

Изменение вероятности должно обновляться и в шапке карточки.

---

# 12. Скидка

Скидка задаётся в процентах:

```text
0–100%
```

Пустое значение допустимо.

Не рассчитывай итоговую цену автоматически в этой задаче.

Не связывай скидку со спецификацией, производством или заказом.

---

# 13. Даты

Поддержи:

* планируемую дату заказа;
* желаемую дату готовности;
* дату мероприятия.

Используй существующий компонент даты или HTML `input type="date"`.

Валидация:

* желаемая дата готовности не должна быть раньше планируемой даты заказа;
* дата мероприятия может быть пустой;
* дата готовности желательно должна быть не позже даты мероприятия, но это предупреждение, а не блокирующая ошибка.

Не добавляй тяжёлую библиотеку дат.

Формат отображения:

```text
16 июля 2026
```

Используй стабильное форматирование без hydration mismatch.

# 16. Источник и рекламная кампания

Источник лида должен использовать текущее значение из общего состояния.

При изменении источника в коммерческом блоке значение должно синхронизироваться с шапкой.

Не создавай второй независимый источник.

Рекламная кампания и UTM-описание могут быть обычными строковыми полями.

Пример:

```text
VK — форма для футбольных академий
utm_campaign=football_uniform_2026
```

---

# 17. Описание потребности

Добавь многострочное поле:

```text

```

Пример:

```text

```

Ограничение длины:

```text
до 3000 символов
```

Покажи счётчик только в режиме редактирования, если это не усложняет компонент.

# 22. Адаптивность

На desktop:

* параметры располагаются в 2–3 колонки;
* описание занимает всю ширину;
* финансовые значения визуально выделены;
* блок не должен быть чрезмерно высоким.

На узком экране:

* поля переходят в одну колонку;
* кнопки не выходят за контейнер;
* отсутствует горизонтальная прокрутка;
* даты и суммы не обрезаются.

---

# 23. Доступность

Проверь:

* у каждого поля есть `label`;
* ошибки связаны с полями;

* кнопки доступны с клавиатуры;
* focus-state виден;
* значения badge имеют текст;
* числовые поля имеют разумные `min` и `max`.

# 25. Проверки

Из папки `frontend` выполни доступные команды:

```text
npm run lint
npm run typecheck
npm run build
```

Если `typecheck` отсутствует:

```text
npx tsc --noEmit
```

Проверь сценарии:

1. Коммерческий блок отображается вместо заглушки.
2. Режим редактирования открывается.
3. «Отмена» не изменяет данные.
4. «Сохранить» обновляет интерфейс.
5. Сумма обновляется в шапке.
6. Вероятность обновляется в шапке.
7. Источник обновляется в шапке.
8. Количество не принимает отрицательные значения.
9. Вероятность ограничена диапазоном `0–100`.
10. Скидка ограничена диапазоном `0–100`.
11. Некорректный порядок дат показывает ошибку.
12. Даты отображаются без hydration mismatch.
13. Финансовые значения форматируются в рублях.
14. Пустые значения отображаются как «Не указано».
15. На мобильной ширине нет горизонтальной прокрутки.
16. В Console нет ошибок React.
17. Карточка клиента продолжает работать.
18. Канбан лидов продолжает работать.

# Критерии готовности

Задача считается выполненной, если:

* заглушка заменена полноценным коммерческим блоком;
* данные отображаются в режиме просмотра;
* редактирование, сохранение и отмена работают;
* сумма, вероятность и источник синхронизированы с шапкой;
* количества, проценты и даты валидируются;
* финансовые значения форматируются правильно;
* отсутствуют ошибки TypeScript и hydration;
* интерфейс адаптивен;
* lint, typecheck и build проходят успешно.
