# Задача Codex: v0.5.5 — История активности лида

## Контекст

Проект: `sport-leads`.

Завершены этапы:

* `v0.5.1 — Каркас страницы лида`;
* `v0.5.2 — Шапка карточки лида`;
* `v0.5.3 — Данные клиента и контакты`;
* `v0.5.3.1 — Локальное состояние клиента`;
* `v0.5.4 — Коммерческие параметры`.

Карточка лида доступна по маршруту:

```text
/sales/leads/[leadId]
```

На текущем этапе данные карточки работают через mock-источник и локальное состояние. Постоянное сохранение в PostgreSQL будет реализовано отдельным backend-этапом.

Текущая задача должна заменить заглушку истории лида полноценной лентой активности.

Не реализуй полноценные коммуникации, задачи, файлы и создание заказа.

---

# Цель

Создать единую хронологическую ленту событий лида.

Менеджер должен видеть:

* когда создан лид;
* изменения статуса;
* изменения ответственного;
* комментарии;
* звонки;
* сообщения;
* письма;
* задачи;
* файлы;
* изменения данных;
* создание заказа или сделки.

На этом этапе используются mock-события и локальное добавление внутренних комментариев.

# 3. Типы событий

Поддержи следующие типы:

```text
lead_created
status_changed
responsible_changed
comment_added

task_completed
incoming_call
outgoing_call
incoming_message
outgoing_message
email_received
```

Если в проекте уже есть подходящий тип событий, расширь его.

Не создавай дублирующую модель без необходимости.

---

# 4. Структура события

Каждое событие должно содержать:

* `id`;
* тип;
* дату и время;
* автора;
* заголовок;
* описание;
* направление входящее или исходящее, если применимо;
* канал коммуникации, если применимо;
* связанные данные;
* вложения, если применимо;
* системное или пользовательское событие.

Пример структуры:

```ts
type LeadActivity = {
  id: string
  type: LeadActivityType
  occurredAt: string
  author?: {
    id: string
    name: string
  }
  title: string
  description?: string
  direction?: "incoming" | "outgoing"
  channel?: "phone" | "email" | "telegram" | "whatsapp" | "vk" | "internal"
  metadata?: Record<string, string | number | boolean>
  attachments?: LeadActivityAttachment[]
  isSystem?: boolean
}
```

Не используй `any`.

Если `Record` не нужен, определи более строгие типы.

---

# 5. Хронология

События должны отображаться в обратном хронологическом порядке:

```text
новые сверху
старые снизу
```

Сортировка должна выполняться по `occurredAt`.

Не полагайся только на порядок mock-массива.

При одинаковом времени сохраняй стабильный порядок.

---

Каждая запись должна показывать:

* иконку типа события;
* заголовок;
* дату и время;
* автора;
* описание;
* канал;
* направление;
* вложения;


# 7. Визуальное разделение типов

Визуально различай:

* системные события;
* внутренние комментарии;
* звонки;
* сообщения;
* письма;
* задачи;
* файлы;
* коммерческие изменения.

Цвет не должен быть единственным способом различения.

Используй:

* иконку;
* текстовую подпись;
* badge канала;
* направление.

Не делай ленту чрезмерно цветной.

---

# 8. Вкладки и фильтры

Добавь вкладки:

```text
Все события
Комментарии
Сообщения
Задачи
Файлы
```

Правила:

* «Все события» показывает всё;
* «Комментарии» — внутренние комментарии;
* «Сообщения» — входящие и исходящие сообщения, письма и звонки;
* «Задачи» — создание и завершение задач;
* «Файлы» — события с вложениями.

Если существующий компонент Tabs отсутствует, используй компактные кнопки-фильтры.

Не создавай сложную систему маршрутизации вкладок.

---

# 9. Добавление внутреннего комментария

В верхней части истории добавь форму:

```text
Добавить внутренний комментарий
```

Поля:

* многострочный текст;
* кнопка «Добавить комментарий».

Правила:

* пустой комментарий нельзя добавить;
* пробелы в начале и конце удаляются;
* максимальная длина — 3000 символов;
* после добавления поле очищается;
* комментарий сразу появляется сверху;
* комментарий помечается как внутренний;
* данные сохраняются только до ухода со страницы.

Не использовать `alert()`.

Не создавать backend endpoint.

Если текущий пользователь отсутствует, используй явно помеченный mock:


# 11. События изменений карточки

Если архитектура позволяет без большого рефакторинга, добавляй локальные события при:

* изменении клиента;
* изменении коммерческих параметров;
* смене основного контакта;
* изменении суммы;
* изменении вероятности;
* изменении источника.

Пример:

```text
Коммерческие параметры обновлены
Сумма изменена с 480 000 ₽ на 520 000 ₽.
```

Если это требует существенного рефакторинга, подготовь callbacks и реализуй минимум:

* `customer_updated`;
* `commercial_updated`.

Не создавай полноценный аудит полей — он будет отдельной задачей.

# 13. Вложения

Событие файла должно отображать:

* имя файла;
* тип;
* размер, если доступен;
* кнопку или ссылку просмотра.

На текущем этапе файл может быть mock-объектом без реального скачивания.

Не создавай upload-компонент.

```text

```

---

# 14. Форматирование дат

Используй стабильный формат `ru-RU`.

Пример:

```text
16 июля 2026, 14:30
```

Для событий текущего дня допускается:

```text
Сегодня, 14:30
```

Но только если это не вызывает hydration mismatch.

Предпочтительно использовать абсолютный стабильный формат на сервере и клиенте.

Не добавляй новую библиотеку дат без необходимости.

---

# 15. Пустое состояние

Если событий нет:

```text
История пока пуста
```

Описание:

```text
События, комментарии и коммуникации появятся здесь.
```

Форма добавления внутреннего комментария при этом должна оставаться доступной.

---

# 16. Локальное состояние

История должна быть частью единого локального состояния карточки либо управляться на уровне `LeadPage`.

Требования:

* новые комментарии не исчезают при переключении вкладок;
* сортировка сохраняется;
* изменения не мутируют исходный массив;
* при уходе со страницы допустим сброс mock-данных;
* не использовать `localStorage`;
* не использовать глобальный store.

---

# 17. Производительность

Не рендери всю историю повторно без необходимости.

Допускается `useMemo` для:

* сортировки;
* фильтрации;
* подсчёта количества событий.

Не добавляй виртуализацию списка на этом этапе.

---

# 18. Адаптивность

На desktop:

* timeline занимает всю доступную ширину основной области;
* дата и метаданные читаемы;
* форма комментария компактна.

На узком экране:

* события переходят в одну колонку;
* дата не перекрывает текст;
* вкладки можно прокручивать горизонтально внутри контейнера;
* нет горизонтальной прокрутки всей страницы.

---

# 19. Доступность

Проверь:

* форма комментария имеет label;
* кнопка имеет понятное имя;
* вкладки доступны с клавиатуры;
* focus-state виден;
* иконки не являются единственным источником информации;
* события имеют понятную структуру для screen reader;
* disabled-действия корректно помечены.

Из папки `frontend` выполни:

```text

npm run build
```

Если `typecheck` отсутствует:

```text

```

Проверь сценарии:

1. История отображается вместо заглушки.
2. События отсортированы от новых к старым.
3. Вкладка «Все события» показывает всё.
4. Фильтры работают корректно.
5. Комментарий нельзя добавить пустым.
6. Новый комментарий появляется сверху.
7. Поле очищается после добавления.
8. Переключение вкладок не удаляет комментарий.
9. Системные и пользовательские события различаются.
10. Каналы отображаются текстом и badge.
11. Даты отображаются без hydration mismatch.
12. Пустое состояние работает.
13. На мобильной ширине нет горизонтальной прокрутки.
14. В Console нет ошибок.
15. Клиентский и коммерческий блоки продолжают работать.
16. Канбан лидов продолжает работать.

# Критерии готовности

Задача считается выполненной, если:

* заглушка истории заменена полноценным timeline;
* поддерживаются основные типы событий;
* события сортируются по времени;
* работают вкладки и фильтры;
* можно добавить внутренний комментарий;
* комментарий появляется сразу;
* даты отображаются стабильно;
* интерфейс адаптивен;
* отсутствуют ошибки TypeScript и hydration;
* lint, typecheck и build проходят успешно.
