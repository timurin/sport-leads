# Задача Codex: v0.5.6 — Внутренние заметки и комментарии лида

## Контекст

Проект: `sport-leads`.

Завершены этапы:

* `v0.5.1 — Каркас страницы лида`;
* `v0.5.2 — Шапка карточки лида`;
* `v0.5.3 — Данные клиента и контакты`;
* `v0.5.4 — Коммерческие параметры`;
* `v0.5.5 — История активности лида`.

Карточка лида доступна по маршруту:

```text
/sales/leads/[leadId]
```

В истории активности уже можно добавить простой внутренний комментарий.

Текущая задача должна расширить внутренние комментарии до полноценной системы заметок:

* создание;
* редактирование;
* удаление;
* закрепление;
* упоминание сотрудников;
* визуальное отделение от сообщений клиенту и системных событий.

Пока используется локальное состояние. Постоянное backend-сохранение будет реализовано отдельно.

Не реализуй отправку сообщений клиенту, задачи, файлы или заказ покупателя.

---

# Цель

Сделать внутренние заметки полноценным инструментом работы менеджера внутри карточки лида.

Менеджер должен иметь возможность:

* создать заметку;
* отредактировать свою заметку;
* удалить свою заметку;
* закрепить важную заметку;
* открепить заметку;
* упомянуть сотрудника;
* видеть закреплённые заметки отдельно;
* отличать внутренние записи от клиентских сообщений и автоматических событий.

Переиспользуй текущие типы и компоненты.

# 2. Модель внутренней заметки

Внутренняя заметка должна содержать:

```ts
type LeadInternalNote = {
  id: string
  text: string
  author: UserSummary
  createdAt: string
  updatedAt?: string
  isPinned: boolean
  mentionedUserIds: string[]
}
```

Допускается хранить заметку как `LeadActivity` типа `comment_added`, если это не усложняет редактирование и закрепление.

Не используй `any`.

Не дублируй автора, дату и текст в нескольких независимых структурах.

---

# 3. Создание заметки

Используй существующую форму добавления внутреннего комментария.

Требования:

* текст обязателен;
* пробелы по краям удаляются;
* максимальная длина — 3000 символов;
* пустая заметка не создаётся;
* после создания поле очищается;
* заметка сразу появляется в истории;
* заметка помечается как внутренняя;
* автором является текущий пользователь;
* создаётся уникальный локальный ID;
* дата создаётся в ISO-формате.

Не используй `alert()`.

---

# 4. Редактирование заметки

Для заметок текущего пользователя добавь действие:

```text
Редактировать
```

Редактирование можно реализовать:

* через Dialog;
* либо inline-режимом.

Требования:

* форма открывается с текущим текстом;
* «Отмена» не меняет заметку;
* «Сохранить» обновляет текст;
* сохраняется исходная дата создания;
* добавляется `updatedAt`;
* отображается подпись:

```text
Изменено
```

или дата редактирования.

Нельзя редактировать системные события, звонки, сообщения и заметки других пользователей.

---

# 5. Удаление заметки

Для заметок текущего пользователя добавь действие:

```text
Удалить
```

Перед удалением покажи подтверждение через существующий Dialog.

Не используй `window.confirm`, если в проекте есть компонент подтверждения.

После удаления:

* заметка исчезает из истории;
* заметка исчезает из закреплённых;
* остальные события не меняются;
* сортировка сохраняется.

Нельзя удалять:

* системные события;
* клиентские сообщения;
* звонки;
* задачи;
* заметки другого пользователя.

---

# 6. Закрепление заметки

Добавь действия:

```text
Закрепить
Открепить
```

Закреплять можно только внутренние заметки.

Закреплённые заметки должны отображаться в отдельном блоке над timeline:

```text
Закреплённые заметки
```

Требования:

* закреплённая заметка остаётся и в общей истории;
* в блоке закреплённых показываются только заметки с `isPinned: true`;
* сортировка закреплённых — новые сверху;
* заметку можно открепить из блока или из меню события;
* при отсутствии закреплённых заметок блок не занимает лишнее место.

Не создавай второй независимый массив закреплённых заметок. Используй фильтрацию единого массива событий или заметок.

---

# 7. Упоминания сотрудников

Добавь возможность упомянуть сотрудника в заметке.

Поддержи синтаксис:

```text
@Имя Фамилия
```

или выбор через существующий Popover/Combobox.

Используй существующий список менеджеров проекта.

При выборе сотрудника:

* его имя добавляется в текст;
* его ID добавляется в `mentionedUserIds`;
* повторное упоминание не должно дублировать ID;
* упомянутый сотрудник визуально выделяется в тексте.

На текущем этапе не отправляй уведомления.

Не создавай backend или notification service.

Если полноценный mention autocomplete требует чрезмерного объёма, реализуй кнопку:

```text
Упомянуть сотрудника
```

с выбором из существующего списка.

---

# 8. Отображение заметки

Внутренняя заметка должна показывать:

* автора;
* дату создания;
* текст;
* отметку редактирования;
* закрепление;
* упомянутых сотрудников;
* меню действий.

Визуально заметка должна отличаться от:

* системного события;
* сообщения клиента;
* исходящего сообщения;
* звонка;
* задачи.

Используй:

* подпись «Внутренняя заметка»;
* отдельную иконку;
* нейтральный фон;
* текстовый badge.

Цвет не должен быть единственным отличием.

---

# 9. Права на действия

Пока полноценная система ролей отсутствует.

Используй временное правило:

* текущий пользователь может редактировать и удалять только свои заметки;
* закреплять и откреплять можно любые внутренние заметки;
* системные события менять нельзя.

Правила должны быть вынесены в понятные проверки, например:

```ts
const canEdit = note.author.id === currentUser.id
const canDelete = note.author.id === currentUser.id
const canPin = note.type === "comment_added"
```

Не размазывай условия по всему JSX.

---

# 10. Вкладка «Комментарии»

Вкладка «Комментарии» должна показывать:

* все внутренние заметки;
* закреплённые и обычные;
* новые сверху.

При удалении или редактировании изменения должны сразу отражаться и во вкладке «Комментарии», и во вкладке «Все события».

Переключение вкладок не должно сбрасывать состояние.

---

# 11. Mock-данные

Для `lead-1` добавь минимум:

* одну обычную заметку;
* одну закреплённую заметку;
* одну заметку другого пользователя;
* одну заметку с упоминанием сотрудника.

Пример закреплённой заметки:

```text
Важно: клиент просит согласовать резервный комплект до 20 июля.
```

Пример с упоминанием:

```text
@Мария Иванова, проверь, пожалуйста, сроки производства.
```

Не добавляй одинаковые заметки всем лидам.

---

# 12. Локальное состояние

Заметки должны использовать единое состояние истории карточки.

Требования:

* редактирование не мутирует объект напрямую;
* удаление создаёт новый массив;
* закрепление создаёт новый объект события;
* упоминания сохраняются до ухода со страницы;
* после переключения вкладок состояние сохраняется;
* после ухода со страницы допустим сброс;
* не использовать `localStorage`;
* не использовать глобальный store.

---

# 13. Доступность

Проверь:

* меню действий доступно с клавиатуры;
* Dialog имеет заголовок;
* Textarea имеет label;
* кнопки без текста имеют `aria-label`;
* focus возвращается после закрытия Dialog;
* упоминания доступны не только через цвет;
* закреплённая заметка имеет текстовую отметку;
* удаление требует подтверждения.

---

# 14. Адаптивность

На desktop:

* закреплённые заметки отображаются компактно;
* меню не перекрывает контент;
* длинный текст переносится.

На мобильной ширине:

* действия доступны;
* текст не выходит за контейнер;
* Dialog помещается в viewport;
* отсутствует горизонтальная прокрутка.

---

# 15. Ограничения

Не выполнять:

* реальные уведомления упомянутым сотрудникам;
* backend API;
* сохранение в PostgreSQL;
* права доступа по ролям;
* отправку заметки клиенту;
* реальные сообщения;
* задачи;
* файлы;
* заказ покупателя;
* аудит редактирования;
* `localStorage`;
* глобальный state manager;
* commit;
* переход к следующей задаче.

---

# 16. Проверки

Из папки `frontend` выполни:

```text
npm run lint
npm run typecheck
npm run build
```

Если `typecheck` отсутствует:

```text
npx tsc --noEmit
```

Проверь сценарии:

1. Новая заметка создаётся.
2. Пустая заметка не создаётся.
3. Заметку текущего пользователя можно редактировать.
4. После редактирования отображается отметка «Изменено».
5. «Отмена» не меняет текст.
6. Заметку текущего пользователя можно удалить.
7. Перед удалением появляется подтверждение.
8. Чужую заметку нельзя редактировать и удалять.
9. Системное событие нельзя изменить.
10. Заметку можно закрепить.
11. Закреплённая заметка появляется над timeline.
12. Заметку можно открепить.
13. Удалённая заметка исчезает из закреплённых и истории.
14. Упоминание сотрудника отображается.
15. Вкладка «Комментарии» обновляется сразу.
16. Переключение вкладок не сбрасывает изменения.
17. На мобильной ширине нет горизонтальной прокрутки.
18. В Console нет ошибок.
19. Нет hydration mismatch.
20. Остальные события timeline продолжают работать.

# Критерии готовности

Задача считается выполненной, если:

* внутреннюю заметку можно создать;
* свою заметку можно изменить и удалить;
* чужие и системные записи защищены;
* заметки можно закреплять и откреплять;
* закреплённые заметки отображаются отдельно;
* работают упоминания сотрудников;
* изменения синхронизируются с timeline и фильтрами;
* отсутствуют ошибки TypeScript и hydration;
* lint, typecheck и build проходят успешно.
