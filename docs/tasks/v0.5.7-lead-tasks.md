# Задача Codex: v0.5.7 — Задачи и напоминания по лиду

## Контекст

Проект: `sport-leads`.

Завершены этапы:

* `v0.5.1 — Каркас страницы лида`;
* `v0.5.2 — Шапка карточки лида`;
* `v0.5.3 — Данные клиента и контакты`;
* `v0.5.4 — Коммерческие параметры`;
* `v0.5.5 — История активности`;
* `v0.5.6 — Внутренние заметки`.

Карточка лида доступна по маршруту:

```text
/sales/leads/[leadId]
```

На текущем этапе данные карточки работают на mock-источнике и локальном состоянии. Постоянное сохранение в PostgreSQL будет реализовано отдельным backend-этапом.

Текущая задача должна создать полноценный блок задач и напоминаний, связанных с лидом.

Не реализуй коммуникации, файлы, заказ покупателя или backend API.

---

# Цель

Менеджер должен иметь возможность:

* видеть ближайшую задачу по лиду;
* просматривать список задач;
* создавать задачу;
* редактировать задачу;
* завершать задачу;
* повторно открывать завершённую задачу;
* переносить срок;
* удалять локальную задачу;
* видеть просроченные задачи;
* видеть события задач в истории лида.

# 2. Создай компонент задач лида

Рекомендуемый компонент:

```text
LeadTasks
```

Предполагаемый файл:

```text
frontend/components/sales/lead-tasks.tsx
```

Компонент должен отображаться:

* в основной части карточки либо отдельной секцией;
* в правой панели — краткий блок ближайшей задачи.

Не дублируй данные в двух независимых состояниях.

Полный список и краткий блок должны получать данные из одного массива задач.

---

# 3. Модель задачи

Задача должна содержать:

```ts
type LeadTask = {
  id: string
  leadId: string
  title: string
  type: LeadTaskType
  status: "open" | "completed"
  priority: "low" | "medium" | "high" | "urgent"
  assignedTo: UserSummary
  dueAt: string
  description?: string
  result?: string
  createdAt: string
  completedAt?: string
  createdBy: UserSummary
}
```

Поддержи типы:

```text
call
message
email
send_proposal
clarify_sizes
receive_design
approve_design
check_payment
meeting
other
```

Пользовательские названия:

```text
Позвонить
Написать
Отправить предложение
Уточнить размеры
Получить макет
Согласовать дизайн
Проверить оплату
Встреча
Другое
```

Не используй `any`.

---

# 4. Список задач

Создай секцию:

```text
Задачи по лиду
```

Добавь фильтры:

* открытые;
* завершённые;
* все.

По умолчанию показывай открытые задачи.

Сортировка открытых задач:

1. просроченные;
2. ближайшие по сроку;
3. без срока, если такие допускаются.

Завершённые задачи сортируй по `completedAt`, новые сверху.

---

# 5. Карточка задачи

Каждая задача должна показывать:

* название;
* тип;
* исполнителя;
* срок;
* время;
* приоритет;
* статус;
* описание;
* результат выполнения, если завершена.

Дополнительно показывай текстовые признаки:

```text
Просрочено
Сегодня
Завтра
Завершено
```

Цвет не должен быть единственным способом передачи статуса.

---

# 6. Создание задачи

Добавь кнопку:

```text
Добавить задачу
```

Кнопка в шапке лида также должна открывать эту же форму, а не оставаться заглушкой.

Используй общий Dialog.

Поля:

* название;
* тип;
* исполнитель;
* дата;
* время;
* приоритет;
* описание.

Кнопки:

```text
Создать
Отмена
```

Правила:

* название обязательно;
* исполнитель обязателен;
* дата обязательна;
* время допускается по умолчанию, например `12:00`;
* нельзя сохранить некорректную дату;
* после создания задача сразу появляется в списке;
* Dialog закрывается;
* форма очищается;
* создаётся уникальный локальный ID.

Не использовать `alert()`.

---

# 7. Быстрые типы задач

В форме можно добавить быстрый выбор:

* позвонить;
* написать;
* отправить предложение;
* проверить оплату.

При выборе типа допускается автоматически подставить название, например:

```text
Позвонить клиенту
```

Пользователь должен иметь возможность изменить название вручную.

---

# 8. Редактирование задачи

Для открытой задачи добавь действие:

```text
Редактировать
```

Форма должна открываться с текущими значениями.

После сохранения:

* обновляется существующая задача;
* ID не меняется;
* порядок списка пересчитывается;
* ближайшая задача в правой панели обновляется.

«Отмена» не меняет данные.

---

# 9. Завершение задачи

Добавь действие:

```text
Завершить
```

При завершении открой небольшой Dialog.

Поля:

* результат выполнения;
* дата завершения формируется автоматически.

Результат может быть необязательным, но желательно показать поле:

```text
Результат
```

После завершения:

* статус становится `completed`;
* добавляется `completedAt`;
* задача исчезает из фильтра открытых;
* появляется в завершённых;
* ближайшая задача пересчитывается;
* в timeline добавляется событие `task_completed`.

---

# 10. Повторное открытие

Для завершённой задачи добавь действие:

```text
Открыть повторно
```

После действия:

* статус становится `open`;
* `completedAt` очищается;
* результат можно сохранить либо очистить — выбери единое логичное поведение;
* задача возвращается в список открытых;
* ближайшая задача пересчитывается;
* в timeline добавляется событие.

Не создавай новый объект задачи.

---

# 11. Перенос срока

Добавь быстрое действие:

```text
Перенести срок
```

Варианты:

* на завтра;
* на 3 дня;
* на неделю;
* выбрать дату и время.

После переноса:

* обновляется `dueAt`;
* список пересортировывается;
* статус просрочки пересчитывается;
* ближайшая задача обновляется;
* в timeline можно добавить локальное событие изменения задачи.

---

# 12. Удаление задачи

Добавь действие:

```text
Удалить
```

Перед удалением показывай подтверждение через существующий Dialog.

После удаления:

* задача исчезает из списка;
* ближайшая задача пересчитывается;
* другие задачи не меняются.

На текущем этапе удаление работает только локально.

Не использовать `window.confirm`, если есть Dialog.

---

# 13. Ближайшая задача

Обнови блок в правой панели:

```text
Ближайшая задача
```

Показывай ближайшую открытую задачу.

Данные:

* название;
* тип;
* дата и время;
* исполнитель;
* приоритет;
* признак просрочки.

Добавь действия:

```text
Открыть
Завершить
```

Если задач нет:

```text
Задачи отсутствуют
```

Кнопка:

```text
Добавить задачу
```

Ближайшая задача должна вычисляться из общего массива, а не храниться отдельно.

---

# 14. Просроченные задачи

Задача считается просроченной, если:

* статус `open`;
* `dueAt` раньше текущего времени.

Покажи:

```text
Просрочено
```

Не вычисляй текущее время по-разному на сервере и клиенте так, чтобы возникал hydration mismatch.

Допускается вычисление статуса в клиентском компоненте после инициализации либо использование стабильного времени из данных.

---

# 15. Синхронизация с timeline

При действиях добавляй события:

## Создание

```text
task_created
```

Пример:

```text
Создана задача «Позвонить клиенту»
Срок: 18 июля 2026, 12:00
Исполнитель: Мария Иванова
```

## Завершение

```text
task_completed
```

## Повторное открытие или перенос

Можно использовать `task_updated`, если расширение типа уместно, либо системное событие с существующим типом.

Не создавай полноценный аудит всех полей.

---

# 16. Mock-данные

Для `lead-1` добавь минимум четыре задачи:

1. открытая задача на сегодня;
2. просроченная задача;
3. будущая задача;
4. завершённая задача с результатом.

Пример:

```text

Тип: Звонок
Срок: сегодня, 16:00
Исполнитель: Мария Иванова
Приоритет: Высокий
```

```text
Срок: вчера, 12:00
Статус: Просрочено
```

```text
Отправить коммерческое предложение
Срок: через 3 дня
```

Не создавай одинаковые задачи для всех лидов.

---

# 17. Локальное состояние

Задачи должны находиться в едином локальном состоянии карточки лида.

Требования:

* не мутировать массив;
* ближайшая задача вычисляется через `useMemo`;
* переключение фильтров не сбрасывает данные;
* действия в шапке, правой панели и основном блоке используют одни callbacks;
* после ухода со страницы допустим сброс;
* не использовать `localStorage`;
* не использовать глобальный store.

---

# 18. Даты и время

Храни срок как ISO datetime.

Формат отображения:

```text
18 июля 2026, 16:00
```

Допускаются подписи «Сегодня» и «Завтра», если они не вызывают hydration mismatch.

Не добавляй тяжёлую библиотеку дат без необходимости.

Учитывай timezone проекта.

---

# 19. Адаптивность

На desktop:

* задачи отображаются компактным списком;
* основные действия доступны без перегрузки интерфейса;
* ближайшая задача помещается в правую панель.

На узком экране:

* карточки переходят в одну колонку;
* кнопки переносятся;
* Dialog помещается в viewport;
* длинные названия переносятся;
* нет горизонтальной прокрутки.

---

# 20. Доступность

Проверь:

* у формы есть labels;
* Dialog имеет заголовок;
* фильтры доступны с клавиатуры;
* меню действий имеет `aria-label`;
* focus-state виден;
* статус и просрочка передаются текстом;
* кнопки завершения имеют понятные названия;
* ошибки формы связаны с полями.

# 23. Проверки

Из папки `frontend` выполни:

```text
npm run lint
npm run typecheck
npm run build
```

Если `typecheck` отсутствует:

```text
npx tsc --noEmit
```

Проверь сценарии:

1. Список задач отображается.
2. Фильтры работают.
3. Новую задачу можно создать.
4. Кнопка в шапке открывает форму.
5. Задачу можно редактировать.
6. Задачу можно завершить.
7. Результат отображается.
8. Завершённую задачу можно открыть повторно.
9. Срок можно перенести.
10. Задачу можно удалить.
11. Просроченная задача помечена текстом.
12. Ближайшая задача отображается в правой панели.
13. После завершения ближайшая задача пересчитывается.
14. События создания и завершения появляются в timeline.
15. Переключение фильтров не сбрасывает состояние.
16. На мобильной ширине нет горизонтальной прокрутки.
17. В Console нет ошибок.
18. Нет hydration mismatch.
19. Остальные блоки карточки продолжают работать.
20. Канбан лидов продолжает работать.

# Критерии готовности

Задача считается выполненной, если:

* задачи отображаются в карточке лида;
* задачу можно создать, изменить, завершить и открыть повторно;
* работает перенос срока;
* просроченные задачи отмечаются;
* ближайшая задача отображается в правой панели;
* события задач синхронизируются с timeline;
* интерфейс адаптивен;
* отсутствуют ошибки TypeScript и hydration;
* lint, typecheck и build проходят успешно.
