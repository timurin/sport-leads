# Задача Codex: v0.5.8 — Интерфейс коммуникаций лида

## Контекст

Проект: `sport-leads`.

Завершены этапы:

* `v0.5.1 — Каркас страницы лида`;
* `v0.5.2 — Шапка карточки лида`;
* `v0.5.3 — Данные клиента и контакты`;
* `v0.5.4 — Коммерческие параметры`;
* `v0.5.5 — История активности`;
* `v0.5.6 — Внутренние заметки`;
* `v0.5.7 — Задачи и напоминания`.

Карточка лида доступна по маршруту:

```text
/sales/leads/[leadId]
```

Текущая задача должна создать единое окно переписки с клиентом.

На этом этапе не подключать реальные API Telegram, VK, WhatsApp, email или телефонии.

Использовать mock-данные и локальное состояние страницы.

---

# Цель

Менеджер должен иметь возможность:

* видеть переписку с клиентом;
* переключать канал связи;
* вводить сообщение;
* прикреплять mock-файлы;
* использовать шаблоны сообщений;
* отправлять сообщение локально;
* видеть входящие и исходящие сообщения;
* видеть статус доставки;
* видеть автора, дату и канал;
* синхронизировать сообщения с timeline.

# 3. Поддерживаемые каналы

Поддержи интерфейс для каналов:

```text
phone
email
telegram
whatsapp
vk
website
internal
```

Пользовательские названия:

```text
Телефон
Email
Telegram
WhatsApp
VK
Форма сайта
Внутренний
```

На текущем этапе отправка доступна только для mock-каналов:

* email;
* Telegram;
* WhatsApp;
* VK.

Телефон отображается как история звонков без отправки текстового сообщения.

Внутренний канал не должен отправлять сообщения клиенту.

---

# 4. Модель сообщения

Сообщение должно содержать:

```ts
type LeadMessage = {
  id: string
  leadId: string
  channel: CommunicationChannel
  direction: "incoming" | "outgoing"
  text: string
  author?: UserSummary
  senderName?: string
  recipientName?: string
  sentAt: string
  status?: "draft" | "sending" | "sent" | "delivered" | "read" | "failed"
  externalId?: string
  attachments?: LeadMessageAttachment[]
  isMock?: boolean
}
```

Вложение:

```ts
type LeadMessageAttachment = {
  id: string
  name: string
  type?: string
  size?: number
}
```

Не используй `any`.

---

# 5. Структура интерфейса

Интерфейс должен включать:

## Верхняя панель

* выбранный канал;
* основной контакт;
* адрес назначения;
* фильтр каналов;
* индикатор mock-режима.

## История переписки

* входящие сообщения слева;
* исходящие сообщения справа;
* дата и время;
* автор или отправитель;
* канал;
* статус доставки;
* вложения.

## Нижняя панель ввода

* textarea;
* выбор шаблона;
* прикрепление файла;
* кнопка отправки;
* счётчик длины текста.

---

# 6. Выбор канала

Добавь переключатель канала.

При выборе показывай адрес назначения:

* email;
* телефон;
* Telegram;
* WhatsApp;
* VK.

Источник адреса — основной контакт лида.

Если данных для выбранного канала нет, покажи:

```text
Для этого канала контакт не указан
```

В этом случае кнопка отправки должна быть disabled.

Не придумывай адреса, которых нет в данных клиента.

---

# 7. История сообщений

Сообщения должны отображаться по времени:

```text
старые сверху
новые снизу
```

При открытии панели прокрутка должна быть внизу, если это реализуется без сложного поведения.

Каждое сообщение должно показывать:

* текст;
* направление;
* канал;
* отправителя;
* дату;
* статус;
* вложения.

Не смешивай внутренние заметки с клиентской перепиской.

---

# 8. Отправка mock-сообщения

При отправке:

1. Проверить непустой текст или наличие вложения.
2. Создать локальное сообщение.
3. Добавить его в общий массив сообщений.
4. Очистить поле ввода.
5. Показать статус `sent`.
6. Через короткий локальный переход можно изменить статус на `delivered`.

Не выполнять реальный сетевой запрос.

Не показывать ложное сообщение о доставке через внешний сервис.

Добавь явную подпись:

```text
Тестовый режим
```

или:

```text
Сообщение сохранено локально
```

Не использовать `alert()`.

---

# 9. Статусы доставки

Поддержи:

```text
Черновик
Отправляется
Отправлено
Доставлено
Прочитано
Ошибка
```

Цвет не должен быть единственным способом передачи статуса.

Используй текст и иконку.

На mock-этапе достаточно состояний:

```text
sent
delivered
read
failed
```

---

# 10. Входящие сообщения

Для `lead-1` добавь mock-переписку:

* входящее сообщение Telegram;
* исходящий ответ;
* входящее email;
* исходящее сообщение VK;
* сообщение с вложением;
* сообщение со статусом `read`;
* сообщение со статусом `failed`.

Не использовать одинаковую переписку для всех лидов.

Для остальных лидов достаточно 2–4 сообщений.

---

# 11. Шаблоны сообщений

Добавь кнопку:

```text
Шаблоны
```

Поддержи минимум четыре шаблона:

```text
Первичный ответ
Уточнение деталей
Отправка предложения
Напоминание
```

Примеры:

```text
Здравствуйте! Спасибо за обращение. Уточните, пожалуйста, необходимое количество комплектов и желаемую дату готовности.
```

```text
Добрый день! Подготовили предварительное предложение. Готовы обсудить состав комплекта, нанесение и сроки производства.
```

При выборе шаблона:

* текст подставляется в поле;
* пользователь может его изменить;
* сообщение не отправляется автоматически.

---

# 12. Вложения

Добавь кнопку прикрепления файла.

На текущем этапе:

* использовать локальный `input type="file"`;
* не загружать файл на сервер;
* сохранять только метаданные в локальном состоянии;
* показывать имя и размер;
* позволить удалить вложение до отправки.

Ограничения:

* максимум 5 файлов;
* размер одного файла до 20 MB;
* показать ошибку при превышении;
* не читать весь файл в память без необходимости.

После отправки вложение отображается в сообщении.

Не создавать реальную загрузку.

---

# 13. Длина сообщения

Максимальная длина:

```text
5000 символов
```

Показывай счётчик.

При превышении:

* не отправлять;
* показать ошибку рядом с полем.

---

# 14. Синхронизация с timeline

При создании сообщения добавляй событие:

```text
outgoing_message
```

или:

```text
email_sent
```

в зависимости от канала.

Входящие mock-сообщения должны отображаться в timeline как:

```text
incoming_message
email_received
```

Не дублируй один и тот же объект в нескольких независимых состояниях без необходимости.

Предпочтительно хранить сообщения отдельно, а timeline формировать через событие-ссылку либо общий callback добавления активности.

---

# 15. Фильтрация

Добавь фильтры переписки:

```text
Все
Telegram
Email
WhatsApp
VK
```

Телефонные события остаются в timeline, а не в текстовой переписке, если это логичнее для текущего интерфейса.

Переключение фильтра не должно сбрасывать draft.

---

# 16. Draft-состояние

Текст сообщения должен сохраняться при переключении фильтров и каналов в рамках открытой карточки.

Можно хранить отдельный draft по каналу:

```ts
Record<CommunicationChannel, string>
```

Не использовать `localStorage`.

После ухода со страницы допустим сброс.

---

# 17. Ответ на сообщение

Для входящего сообщения добавь действие:

```text
Ответить
```

После нажатия:

* выбирается канал сообщения;
* поле ввода получает focus;
* можно добавить короткую цитату или подпись отправителя.

Не реализуй сложную систему threading.

---

# 18. Копирование текста

Добавь действие:

```text
Копировать текст
```

Используй Clipboard API.

Обрабатывай ошибку.

Покажи существующий toast или нейтральное подтверждение.

Не использовать `alert()`.

---

# 19. Пустое состояние

Если сообщений нет:

```text
Переписка пока отсутствует
```

Описание:

```text
Входящие и исходящие сообщения появятся здесь.
```

Форма отправки остаётся доступной при наличии контактного адреса.

---

# 20. Локальное состояние

Сообщения должны быть частью локального состояния карточки.

Требования:

* не мутировать массив;
* отправка добавляет новый объект;
* фильтрация выполняется через `useMemo`;
* draft не исчезает при переключении фильтров;
* выбранный канал сохраняется до ухода со страницы;
* не использовать глобальный store;
* не использовать `localStorage`.

---

# 21. Адаптивность

На desktop:

* история имеет ограниченную высоту;
* поле ввода закреплено внизу панели;
* сообщения читаемы;
* панель не растягивает бесконечно страницу.

На мобильной ширине:

* сообщения занимают большую часть ширины;
* кнопки переносятся;
* фильтры можно прокручивать внутри контейнера;
* поле ввода остаётся доступным;
* нет горизонтальной прокрутки всей страницы.

---

# 22. Доступность

Проверь:

* textarea имеет label;
* выбор канала доступен с клавиатуры;
* статус сообщения передаётся текстом;
* кнопки имеют `aria-label`;
* focus виден;
* ошибки связаны с полями;
* вложения имеют понятные названия;
* входящие и исходящие сообщения имеют доступные текстовые метки.

---

* телефонию;

* массовые рассылки;

* глобальный store;
* `localStorage`;
* commit;
* переход к следующей задаче.

# 25. Проверки

Из папки `frontend` выполни:

```text
npm run lint
npm run typecheck
npm run build
```

Если `typecheck` отсутствует:

```text
npx tsc --noEmit
```

Проверь сценарии:

1. Панель коммуникаций отображается.
2. Каналы переключаются.
3. Показывается адрес основного контакта.
4. Канал без контакта блокирует отправку.
5. Пустое сообщение не отправляется.
6. Сообщение отправляется локально.
7. Поле очищается после отправки.
8. Статус отображается.
9. Фильтры работают.
10. Draft не исчезает при переключении фильтров.
11. Шаблон подставляется в поле.
12. Шаблон не отправляется автоматически.
13. Файл можно выбрать и удалить до отправки.
14. Ограничение размера работает.
15. Максимум 5 файлов соблюдается.
16. Вложение отображается после отправки.
17. «Ответить» выбирает нужный канал.
18. Копирование текста работает.
19. Исходящее сообщение появляется в timeline.
20. На мобильной ширине нет горизонтальной прокрутки.
21. В Console нет ошибок.
22. Нет hydration mismatch.
23. Остальные блоки карточки продолжают работать.

# Критерии готовности

Задача считается выполненной, если:

* отображается единая панель переписки;
* работают переключение и фильтрация каналов;
* можно отправить mock-сообщение;
* работают шаблоны;
* поддерживаются mock-вложения;
* отображаются статусы доставки;
* сообщения синхронизируются с timeline;
* интерфейс адаптивен;
* отсутствуют ошибки TypeScript и hydration;
* lint, typecheck и build проходят успешно.
