# Задача Codex: v0.5.9 — Ядро коннекторов коммуникаций

## Контекст

Проект: `sport-leads`.

Завершены интерфейсные этапы карточки лида:

* `v0.5.1 — Каркас страницы лида`;
* `v0.5.2 — Шапка карточки лида`;
* `v0.5.3 — Данные клиента и контакты`;
* `v0.5.4 — Коммерческие параметры`;
* `v0.5.5 — История активности`;
* `v0.5.6 — Внутренние заметки`;
* `v0.5.7 — Задачи и напоминания`;
* `v0.5.8 — Интерфейс коммуникаций`.

В карточке уже существует mock-интерфейс переписки для:

* email;
* Telegram;
* WhatsApp;
* VK.

Текущая задача должна создать архитектурное ядро коннекторов, чтобы frontend и бизнес-логика не зависели от конкретного внешнего сервиса.

На этом этапе реальные внешние API не подключать.

Не выполнять реальные сетевые запросы к Telegram, VK, WhatsApp или почтовому серверу.

---

# Цель

Создать единый программный контракт коммуникационного коннектора и mock-реализацию.

Архитектура должна позволять в следующих версиях подключать:

* Telegram Bot API;
* VK API;
* email через SMTP/IMAP;
* WhatsApp Business API;
* формы сайта;
* другие каналы.

Все внешние сообщения должны преобразовываться в единую внутреннюю модель.

```python
class MessageStatus(str, Enum):
```

# 6. Нормализованная модель сообщения

Создай Pydantic-схему нормализованного сообщения.

Минимальные поля:

```python
class NormalizedMessage(BaseModel):
    id: UUID
    channel: CommunicationChannel
    direction: MessageDirection
    external_message_id: str | None
    external_conversation_id: str | None
    sender_external_id: str | None
    sender_name: str | None
    recipient_external_id: str | None
    recipient_name: str | None
    text: str
    status: MessageStatus
    sent_at: datetime | None
    received_at: datetime | None
    attachments: list[NormalizedAttachment]
    metadata: dict[str, JsonValue]
```

Дополнительно предусмотри:

* `lead_id`;
* `contact_id`;
* `connector_name`;
* `raw_event_id`;
* `reply_to_external_message_id`;
* `is_mock`.

Не используй `dict[str, Any]`, если можно применить Pydantic `JsonValue`.

Не хранить токены, секреты или полный raw webhook в открытом поле модели.

# 7. Модель вложения

Создай схему:

```python
class NormalizedAttachment(BaseModel):
    id: UUID
    external_id: str | None
    name: str
    mime_type: str | None
    size_bytes: int | None
    source_url: str | None
    storage_key: str | None
```

Валидация:

* размер не может быть отрицательным;
* имя обязательно;
* URL необязателен;
* содержимое файла не хранится в Pydantic-модели.

---

# 8. Получатель исходящего сообщения

Создай отдельную команду отправки:

```python
class SendMessageCommand(BaseModel):
    channel: CommunicationChannel
    recipient_external_id: str
    recipient_name: str | None
    text: str
    attachments: list[OutgoingAttachment]
    reply_to_external_message_id: str | None
    metadata: dict[str, JsonValue]
```

Требования:

* текст может быть пустым только при наличии вложения;
* канал `internal` нельзя отправлять через внешний connector;
* телефонный канал пока не поддерживает текстовую отправку;
* максимальная длина текста задаётся коннектором или общей валидацией.

---

# 9. Результат отправки

Создай схему:

```python
class SendMessageResult(BaseModel):

    external_conversation_id: str | None
    status: MessageStatus
    sent_at: datetime

```

# 10. Базовый интерфейс коннектора

Создай абстрактный контракт.

Допустимые варианты:

* `typing.Protocol`;
* `ABC`;
* существующий проектный подход.

Контракт должен поддерживать:

```python
class CommunicationConnector(Protocol):
    channel: CommunicationChannel
    name: str

    async def send_message(
        self,
        command: SendMessageCommand,
    ) -> SendMessageResult:
        ...

    async def normalize_webhook(
        self,
        payload: Mapping[str, object],
        headers: Mapping[str, str] | None = None,
    ) -> list[NormalizedMessage]:
        ...

    async def get_delivery_status(
        self,
        external_message_id: str,
    ) -> MessageStatus:
        ...

    async def validate_configuration(self) -> ConnectorHealth:
        ...
```

Не добавляй методы, реализацию которых нельзя логично определить.

`receive_message()` как постоянно работающий метод не требуется: входящие данные должны поступать через webhook или отдельный collector.

Для будущего email допускается отдельный polling collector, использующий тот же нормализованный формат.

# 13. Реестр коннекторов

Создай `ConnectorRegistry`.

Функции:

```python
register(connector)
get(channel)
has(channel)
list_registered()
```

Требования:

* нельзя зарегистрировать два коннектора одного канала без явной замены;
* запрос отсутствующего канала вызывает `ConnectorNotFoundError`;
* registry не должен создавать реальные подключения при импорте модуля;
* зависимости должны передаваться явно.

# 14. Mock-коннектор

Создай `MockCommunicationConnector`.

Он должен:

* поддерживать выбранный канал;
* не делать сетевых запросов;
* создавать детерминированный или уникальный `external_message_id`;
* возвращать `sent` или `delivered`;
* нормализовывать тестовый webhook;
* хранить отправленные сообщения только в памяти экземпляра;
* поддерживать получение статуса;
* уметь имитировать ошибку через безопасный параметр тестовой конфигурации.

Не использовать глобальный изменяемый список.

---

# 16. Сервис коммуникаций

Создай `CommunicationService`.

Обязанности:

* получить connector из registry;
* валидировать команду;
* вызвать `send_message`;
* преобразовать результат в нормализованное исходящее сообщение;
* нормализовать webhook через нужный connector;
* не содержать логику конкретного канала.

Пример:

```python
class CommunicationService:
    async def send(
        self,
        command: SendMessageCommand,
    ) -> NormalizedMessage:
        ...

    async def process_webhook(
        self,
        channel: CommunicationChannel,
        payload: Mapping[str, object],
        headers: Mapping[str, str] | None = None,
    ) -> list[NormalizedMessage]:
        ...
```

На этом этапе не сохранять сообщения в БД.

Подготовь точку расширения для будущего repository, но не создавай фиктивное хранилище PostgreSQL.

В комментарии укажи, что production-дедупликация должна выполняться через уникальные ограничения БД или Redis.

# 18. Безопасность webhook

Создай метод или контракт:

```python
async def verify_webhook(
    payload: bytes,
    headers: Mapping[str, str],
) -> bool:
    ...
```

Либо включи проверку внутрь `normalize_webhook`, если это соответствует архитектуре.

Mock-коннектор может принимать тестовый секрет.

Требования:

* сравнение секрета выполнять безопасным способом;
* не логировать секрет;
* при неверной подписи поднимать понятное исключение;
* не реализовывать подписи конкретных Telegram или VK API на этом этапе.

# Критерии готовности

Задача считается выполненной, если:

* существует единый контракт коммуникационного коннектора;
* создана нормализованная модель сообщений и вложений;
* работает registry;
* работает mock-коннектор;
* сервис не зависит от конкретного канала;
* предусмотрены ошибки, health-check, проверка webhook и идемпотентность;
* присутствуют unit-тесты;
* архитектура документирована;
* frontend продолжает собираться;
* backend-тесты проходят успешно.
