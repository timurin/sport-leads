# v0.5.99 — Подготовка контрольной точки v0.5.0

## Цель

Подготовить текущую рабочую версию Sport Leads к фиксации перед началом этапа v0.6.

Контрольная точка:

v0.5.0

Это функциональный checkpoint раздела «Лиды». Внешний вид карточки лида считается временным и будет финализирован после завершения всего функционала раздела.

## Важно

Не выполнять полный архитектурный аудит.

Не продолжать редизайн.

Не начинать v0.6.

Не удалять рабочие файлы только потому, что они пока не отслеживаются Git.

## Перед началом

Проверь:

git status --short
git diff --check
git diff --stat

Раздели файлы на группы:

1. Рабочий код текущего этапа.
2. Тесты.
3. Документация.
4. Задачи Codex.
5. Явно временные или генерируемые файлы.
6. Секреты и локальная конфигурация.

Сначала покажи краткий план и список файлов, которые предлагается включить в checkpoint.

## Включить в checkpoint

Включить рабочие изменения этапа v0.5:

- frontend карточки лида;
- список лидов и Kanban, если изменения являются частью текущей рабочей версии;
- коммуникационный интерфейс;
- задачи;
- заметки;
- timeline;
- demo-данные, необходимые для запуска;
- типы, используемые текущими компонентами;
- backend communications, если они уже являются частью текущей реализации;
- backend и frontend тесты;
- дизайн-систему;
- layout-компоненты;
- документацию архитектуры;
- документацию задач и релиза.

## Не включать

Не добавлять в Git:

- .env;
- реальные пароли и токены;
- .venv;
- node_modules;
- frontend/.next;
- __pycache__;
- .pytest_cache;
- coverage;
- логи;
- временные screenshots;
- системные lock-файлы;
- локальные IDE-файлы, если они не нужны проекту;
- скачанные архивы;
- временные копии HTML.

Проверь и при необходимости точечно обнови .gitignore.

Не удаляй docs/example-design автоматически. Если это устаревший эксперимент, оставь его вне checkpoint и укажи в отчёте. Если он уже используется документацией или задачами, сохрани.

## Проверка секретов

Проверь отслеживаемые и подготовленные файлы на:

- POSTGRES_PASSWORD;
- API keys;
- access tokens;
- private keys;
- реальные пароли;
- содержимое .env.

Файл .env не должен попадать в commit.

Допустим .env.example только с безопасными демонстрационными значениями.

## Backend-проверки

Из корня проекта активируй окружение и выполни:

python -m pytest

Также проверь импорт приложения:

python -c "from app.main import app; print(list(app.openapi()['paths'].keys()))"

Проверь отсутствие критических ошибок импорта и дублирующихся маршрутов.

Не исправляй крупные архитектурные проблемы в этой задаче. Критические ошибки сборки или запуска исправлять разрешено.

## Frontend-проверки

Из frontend выполни:

npx tsc --noEmit
npm run lint
npm run build

Не менять внешний вид карточки лида, кроме исправления ошибки, которая блокирует сборку или использование страницы.

## Ручная проверка

Проверь:

- /sales/leads;
- /sales/leads/lead-1;
- /sales/leads/lead-2;
- /sales/leads/lead-999.

Минимально проверить:

1. Список лидов открывается.
2. Kanban открывается.
3. Карточка lead-1 открывается.
4. Карточка lead-2 открывается.
5. lead-999 показывает not-found.
6. Переключаются вкладки карточки.
7. Работают задачи.
8. Работают заметки.
9. Переключаются каналы коммуникации.
10. Mock-сообщение отправляется.
11. В Console нет критических ошибок.

Внешний вид не оценивать и не перерабатывать.

## Git

После успешных проверок подготовь файлы к commit через git add.

Перед commit обязательно покажи:

git status --short
git diff --cached --stat
git diff --cached --check

Убедись, что в staged нет:

- .env;
- .next;
- node_modules;
- кэшей;
- логов;
- секретов;
- временных screenshots.

## Commit и tag

Сделай commit:

release: v0.5.0 leads functional checkpoint

После успешного commit создай annotated tag:

v0.5.0

Сообщение тега:

Sport Leads v0.5.0 — leads functional checkpoint

Не выполнять push, если remote origin не настроен или есть сомнения в выбранном репозитории.

Если origin настроен корректно, покажи URL remote и остановись перед push.

## Не делать

- не выполнять полный аудит;
- не перерабатывать дизайн;
- не менять API без критической необходимости;
- не удалять рабочую функциональность;
- не выполнять массовую очистку;
- не выполнять force push;
- не переписывать историю Git;
- не использовать git reset --hard;
- не использовать git clean -fd;
- не удалять untracked-файлы массово.
