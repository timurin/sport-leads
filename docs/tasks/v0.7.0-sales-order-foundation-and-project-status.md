Часть 5. Старт v0.7.0 — карточка заказа покупателя

После аудита начать следующий пункт roadmap:

«Карточка заказа покупателя — foundation»

Это первая итерация серии v0.7.x.

Требуется

Исследовать существующие:

модель заказа;
API заказов;
страницу списка заказов;
результат конвертации лида;
frontend-типы;
маршруты;
дизайн-шаблоны заказа;
связи заказа с лидом и клиентом.

Затем реализовать минимальный фундамент карточки заказа.

Минимальный объём
Маршрут карточки:
/sales/orders/[orderId]
Загрузка заказа из backend по ID.
Корректные состояния:
loading;
not found;
API error;
empty/partial data.
Базовый header карточки:
номер заказа;
дата создания;
статус;
клиент;
источник или ссылка на исходный лид, если связь существует;
ответственный;
итоговая сумма, если она уже хранится.
Навигация:
из списка заказов можно открыть карточку;
из карточки можно вернуться к списку;
существующая навигация CRM не ломается.
Не создавать пока полноценные:
позиции продукции;
техкарты;
производство;
склад;
закупки;
сложный жизненный цикл;
новый визуальный редизайн;
отдельные endpoint без необходимости.

Использовать текущую архитектуру лидов как ориентир, но не копировать её механически.

API и backend

Перед созданием нового endpoint проверить, существует ли подходящий:

GET /orders/{order_id}

или его эквивалент.

Если endpoint уже существует — использовать и привести к нужной полноте.

Если отсутствует — добавить только минимальный read endpoint карточки заказа.

Не расширять задачу до полного редактирования заказа.
