На этом шаге не реализовывать код, модели БД, миграции, API или frontend.

## Архитектурная концепция номенклатуры

Зафиксировать следующую модель:

> Номенклатура — единый справочник продаваемых, производимых, закупаемых и используемых объектов.
> Тип определяет бизнес-поведение.
> Иерархическая категория определяет назначение и наследуемый набор реквизитов.
> Характеристики описывают варианты исполнения.
> Производственная модель и спецификация являются отдельными сущностями и связываются только с продукцией.
> Позиция заказа хранит независимый коммерческий snapshot.

Не смешивать в одной сущности:

- тип номенклатуры;
- категорию;
- дополнительный реквизит;
- характеристику;
- вариант;
- модель изделия;
- спецификацию;
- позицию заказа.

## Типы номенклатуры

Предусмотреть четыре системных типа:

1. `SERVICE` — Услуга;
2. `PRODUCT` — Продукция собственного производства;
3. `GOODS` — Покупной товар;
4. `MATERIAL` — Материал.

Тип должен определять доступное бизнес-поведение и секции карточки.

### Общие поля

Общие для всех типов:

- идентификатор;
- внутренний код;
- артикул;
- полное наименование;
- краткое наименование;
- описание;
- тип;
- категория;
- базовая единица хранения;
- активность;
- базовая цена;
- валюта;
- изображения;
- дополнительные реквизиты;
- даты создания и изменения.

### Уникальные области типов

Зафиксировать направления развития без реализации:

#### Услуга

- не ведётся как складской запас;
- может иметь длительность или норму времени;
- может использовать единицы «услуга», «час», «макет», «заказ»;
- не требует модели изделия и материальной спецификации.

#### Продукция

- может иметь производственную модель;
- ревизию модели;
- размерную сетку;
- лекала;
- спецификации;
- технологические операции;
- техкарту;
- расчёт стоимости пошива;
- производство под заказ.

#### Товар

- закупается и перепродаётся;
- может иметь поставщиков;
- закупочную и продажную единицы;
- закупочную цену;
- штрихкод;
- складской учёт;
- характеристики.

#### Материал

- используется в производстве;
- имеет единицу хранения и списания;
- может иметь альтернативные единицы;
- цвет, ширину, плотность, состав;
- партии и поставщиков;
- входит в спецификации продукции.

Не добавлять перечисленные поля в текущую БД в рамках этой задачи.

## Иерархические категории

Запланировать persistent-справочник категорий:

`NomenclatureCategory`

Минимальная будущая структура:

- `id`;
- `parent_id`;
- `name`;
- `code`;
- `description`;
- `nomenclature_type`;
- `is_active`;
- `sort_order`;
- настройка наследования реквизитов;
- `created_at`;
- `updated_at`.

Требования:

- древовидная иерархия;
- категория может иметь родителя;
- запрет циклических связей;
- запрет конфликта типа дочерней и родительской категории;
- перемещение категории с проверкой совместимости;
- категория определяет шаблон реквизитов;
- номенклатура размещается в одной основной категории;
- архитектура может позднее поддержать дополнительные классификации, но не реализовывать их сейчас.

Пример:

```text
Продукция
└── Спортивная форма
    ├── Футбол
    │   ├── Футболки
    │   ├── Шорты
    │   └── Гетры
    ├── Волейбол
    └── Хоккей

Товары
└── Спортивные аксессуары

Материалы
├── Ткани
├── Фурнитура
└── Упаковка

Услуги
├── Дизайн
├── Нанесение
└── Доставка
Тип и категория не должны дублировать друг друга.

Реквизиты категорий

Категория должна задавать набор полей для всей вложенной в неё номенклатуры.

Пример для категории Футболки:

Для привязки поля к категории предусмотреть:

Наследование реквизитов

Зафиксировать правила:

дочерняя категория получает реквизиты всех родителей;
дочерняя категория может сделать необязательное поле обязательным;
может изменить порядок и секцию;
может установить значение по умолчанию;
может сузить допустимые значения;
не может изменить тип данных;
не может переопределить смысл поля с тем же кодом;
используемые поля не удаляются физически;
для устаревших полей применяются статусы inactive, hidden или deprecated.
Индивидуальные реквизиты

Предусмотреть возможность добавить поле конкретной номенклатуре.

При добавлении интерфейс в будущем должен предлагать:

только для этой карточки;
добавить в текущую категорию;
добавить в категорию и наследников;
добавить в общую библиотеку реквизитов.

Создание и изменение категорийных полей должно быть доступно только пользователям с соответствующими правами.

Архитектура должна предотвращать появление множества дублирующих полей с одинаковым смыслом.

Модели и спецификации

Зафиксировать, что поле «Модель» доступно только для типа PRODUCT.

Модель не является обычным дополнительным реквизитом или текстовым полем.

Будущая связь:

Nomenclature
└── ProductModel
    └── ProductModelRevision
        ├── PatternSet
        ├── SizeChart
        ├── MaterialSpecification
        ├── OperationSpecification
        └── TechnicalCard

Одна номенклатура может иметь:

основную модель;
несколько допустимых моделей;
версии модели;
несколько спецификаций;
стандартную или индивидуальную спецификацию заказа.

Предусмотреть будущие способы расчёта стоимости пошива:

по операциям спецификации;
фиксированная стоимость;
индивидуальный расчёт.

Не включать модели, лекала, спецификации и производство в этап v0.8, если текущий roadmap относит их к следующему самостоятельному этапу.
