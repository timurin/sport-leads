## Причина изменения roadmap

Текущий интерфейс справочника номенклатуры остаётся недостаточно удобным для ежедневной ERP-работы:

1. На основной странице одновременно отображаются список и встроенные поля создания категорий и номенклатуры.
2. Отсутствует полноценное рабочее место с отдельными кнопками создания, поиском, фильтрами и действиями.
3. Список визуально остаётся плоским и недостаточно отражает иерархию категорий.
4. Пользователь не получает полноценную отдельную карточку номенклатуры с понятными режимами просмотра и редактирования.
5. Новые реквизиты v0.8.4 уже реализованы, но должны быть встроены в устойчивую структуру карточки до реализации характеристик и вариантов.
6. Реализация характеристик и вариантов на текущем временном интерфейсе приведёт к повторной переработке frontend.

Поэтому перед Characteristics and Variants необходимо добавить отдельную итерацию рабочего места и карточки номенклатуры.

### v0.8.5 — Nomenclature Workspace and Editable Card

Статус: следующий пункт.

Цель:

Создать дружелюбное рабочее место справочника и полноценную отдельную карточку номенклатуры, пригодную для дальнейшего расширения характеристиками, вариантами, изображениями, моделями и спецификациями.

#### Рабочее место справочника

Запланировать переработку основной страницы номенклатуры.

Текущие встроенные формы создания убрать из постоянного содержимого списка.

В верхней панели предусмотреть:

- кнопку «Создать номенклатуру»;
- кнопку «Создать группу»;
- дополнительное меню действий;
- строку поиска;
- основные фильтры;
- сброс фильтров;
- переключение режима представления.

Создание номенклатуры и группы должно открываться:

- в отдельной форме;
- в диалоге;
- либо на отдельном маршруте согласно текущим UI-соглашениям.

Не оставлять постоянно раскрытые поля создания над списком.

#### Иерархическая навигация

Основной desktop-интерфейс должен состоять из:

- дерева категорий слева;
- панели поиска и фильтров;
- списка содержимого выбранной категории справа.

В дереве предусмотреть:

- «Все позиции»;
- «Без категории»;
- активные категории;
- архив или неактивные категории при соответствующем фильтре;
- раскрытие и сворачивание веток;
- количество элементов при доступности данных;
- контекстные действия категории;
- создание дочерней категории;
- редактирование;
- перемещение;
- деактивацию.

Категории не должны отображаться только как плоский фильтр.

#### Содержимое выбранной категории

В основной области отображать:

- дочерние категории;
- номенклатуру выбранной категории;
- при включённом режиме — номенклатуру вложенных категорий.

Предусмотреть два режима:

- «Папки и элементы» — основной;
- «Иерархический список» либо «Все вложенные» — дополнительный.

При выборе категории пользователь должен понимать:

- текущий путь;
- количество найденных элементов;
- применённые фильтры;
- включён ли просмотр вложенных категорий.

#### Поиск и фильтры

Запланировать поиск минимум по:

- артикулу;
- полному наименованию;
- краткому наименованию, когда оно будет доступно;
- внутреннему коду, если он существует;
- дополнительным реквизитам, поддерживающим поиск.

Основные фильтры:

- тип;
- категория;
- активность;
- единица хранения;
- наличие цены;
- наличие обязательных незаполненных реквизитов;
- дополнительные фильтруемые реквизиты.

Активные фильтры показывать явно и давать возможность снять их по одному.

#### Таблица номенклатуры

Рекомендуемые колонки:

- выбор строки;
- изображение или пиктограмма;
- артикул;
- наименование;
- тип;
- категория;
- единица хранения;
- базовая цена;
- статус;
- дата изменения;
- меню действий.

Наименование должно быть явной ссылкой на карточку.

Предусмотреть:

- сортировку;
- настройку видимых колонок;
- сохранение пользовательских настроек — planned, если это выходит за объём;
- массовые действия — planned либо минимально необходимый объём;
- состояния загрузки;
- пустое состояние;
- состояние ошибки;
- адаптивное представление на небольших экранах.

#### Отдельная карточка номенклатуры

Добавить в roadmap отдельный маршрут карточки, например:

`/settings/catalogs/nomenclature/[nomenclatureId]`

Точный маршрут определить по фактической структуре frontend.

Карточка должна открываться:

- по ссылке в таблице;
- после создания новой позиции;
- из связанных объектов, когда такие ссылки появятся.

Карточка не должна быть только временным модальным окном.

#### Шапка карточки

Предусмотреть:

- кнопку возврата;
- главное изображение или placeholder;
- полное наименование;
- артикул;
- тип;
- полный путь категории;
- статус;
- дату последнего изменения;
- индикатор наличия несохранённых изменений.

Основные действия:

- «Редактировать»;
- «Сохранить»;
- «Отмена»;
- дополнительное меню.

Дополнительные действия:

- создать копию;
- деактивировать или активировать;
- переместить в другую категорию;
- открыть связанные заказы;
- открыть историю изменений.

Не все дополнительные действия обязательно реализовывать сразу. Чётко отметить planned-функции.

#### Режим просмотра

По умолчанию карточка должна открываться в режиме просмотра.

Поля показывать как читаемые значения, а не как постоянно активные input-контролы.

Преимущества режима просмотра:

- меньше визуального шума;
- снижение риска случайных изменений;
- понятное различие просмотра и редактирования;
- удобное отображение большого количества реквизитов.

#### Режим редактирования

После команды «Редактировать» разрешённые поля становятся редактируемыми.

В согласованном объёме пользователь с соответствующими правами должен иметь возможность изменить:

- артикул;
- наименование;
- описание;
- тип с учётом ограничений и совместимости;
- категорию;
- единицу хранения;
- базовую цену;
- валюту в рамках текущей модели;
- активность;
- дополнительные реквизиты;
- индивидуальные реквизиты.

Backend остаётся источником бизнес-валидации.

При отмене изменений данные должны возвращаться к сохранённому состоянию.

При уходе со страницы с несохранёнными изменениями должно быть предусмотрено предупреждение.
