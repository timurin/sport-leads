# Задача Codex: v0.8.1 — Nomenclature Core

## Контекст

Проект: Sport-Lead.

Этап 2 «Заказы покупателей» закрыт в согласованном MVP-объёме. Начинается этап v0.8 «Номенклатура».

Перед началом обязательно изучить и использовать как источник истины:

* `AGENTS.md`;
* `docs/architecture/project-structure.md`;
* `docs/architecture/erp-check.md`;
* `docs/roadmap/roadmap.md`;
* `docs/architecture/decisions/ADR-003-sales-order-item-commercial-position.md`;
* `docs/architecture/decisions/ADR-005-pricing-and-tax-boundary.md`.

Сначала проверить фактическое состояние репозитория, активную ветку, существующие модели, API, миграции, тесты и связи позиций заказа.

## Цель

Реализовать persistent-ядро справочника номенклатуры готовых изделий, которое станет единым источником товарных позиций для заказов покупателей.

Номенклатура на этом этапе описывает продаваемое готовое изделие, но не производственную модель изделия.

## Объём v0.8.1

### 1. Сущность номенклатуры

Создать backend-модель и таблицу номенклатуры минимум с полями:

* `id`;
* уникальный внутренний код или артикул;
* наименование;
* краткое наименование — при необходимости;
* описание;
* категория или тип изделия;
* единица измерения;
* базовая цена;
* валюта в рамках текущей ценовой модели проекта;
* признак активности;
* дата создания;
* дата изменения.

Точные названия полей определить по действующим соглашениям проекта.

Денежные значения хранить через `Decimal/Numeric`, без `float`.

### 2. Категории

Реализовать минимально достаточную persistent-классификацию номенклатуры.

Не создавать чрезмерно сложную иерархию. Допустим отдельный справочник категорий либо простая связь, если это соответствует текущей архитектуре.

Должна поддерживаться как минимум классификация готовых изделий, например:

* футбольная форма;
* волейбольная форма;
* футболка;
* шорты;
* спортивный костюм;
* куртка;
* иное изделие.

Не зашивать демонстрационные категории в бизнес-логику.

### 3. API

Добавить API для:

* создания позиции номенклатуры;
* получения списка;
* получения одной позиции;
* редактирования;
* мягкого отключения или изменения активности;
* фильтрации по активности;
* поиска минимум по наименованию и артикулу;
* фильтрации по категории, если категория реализована отдельной сущностью.

Формат ошибок и валидаций должен соответствовать существующему API проекта.

### 4. Интеграция с позициями заказа

Изучить текущую модель позиции заказа покупателя.

Добавить необязательную ссылку позиции заказа на номенклатуру так, чтобы:

* существующие заказы и позиции продолжили работать;
* исторические данные не были разрушены;
* коммерческий снимок позиции заказа сохранялся;
* изменение карточки номенклатуры не изменяло автоматически ранее сохранённое наименование, цену, скидку, размеры, персонализацию и остальные данные заказа.

При выборе номенклатуры в новую позицию заказа разрешается первоначально копировать коммерческие значения в snapshot-поля позиции заказа.

Не превращать позицию заказа в динамическое представление текущей карточки номенклатуры.

### 5. Миграции

Создать безопасную Alembic-миграцию.

Миграция должна:

* применяться на существующей базе;
* не удалять текущие данные;
* корректно откатываться;
* не требовать ручного заполнения обязательных значений для старых позиций заказа.

### 6. Тесты

Добавить или обновить тесты минимум для:

* CRUD номенклатуры;
* уникальности артикула;
* поиска и фильтров;
* Decimal/Numeric-значений;
* активности;
* связи номенклатуры с позицией заказа;
* сохранения snapshot-данных позиции заказа после изменения номенклатуры;
* обратной совместимости существующих заказов;
* миграции и импорта приложения.

## Не входит в v0.8.1

Не реализовывать:

* модели изделий;
* варианты модели;
* лекала;
* размерные сетки номенклатуры;
* материалы и ткани;
* цвета материалов;
* спецификации;
* технологические операции;
* техкарты;
* производство;
* себестоимость;
* складские остатки;
* закупки;
* НДС и налоговые режимы;
* типы цен и прайс-листы;
* синхронизацию с 1С;
* frontend полноценного каталога, если он прямо не требуется существующей архитектурой.

Не смешивать сущности «Номенклатура», «Модель изделия» и «Позиция заказа».

## Архитектурные требования

* Следовать текущей слоистой структуре проекта.
* Не создавать параллельные модели, схемы или API для уже существующих сущностей.
* Использовать существующие соглашения именования.
* Не ломать API заказов покупателей.
* Денежные расчёты выполнять только через `Decimal`.
* Не добавлять временные mock/demo-данные в production-код.
* Не выполнять несвязанный рефакторинг.
* Не менять принятые границы ADR-003 и ADR-005 без отдельного ADR.

При выявлении архитектурного противоречия сначала зафиксировать его в отчёте и выбрать минимально рискованное решение.

## Проверки

Выполнить доступные проектные проверки:

* тесты backend;
* проверку миграции upgrade/downgrade;
* проверку OpenAPI;
* проверку импорта приложения;
* `git diff --check`;
* проверку UTF-8;
* проверку отсутствия случайных временных файлов;
* проверку чистоты архитектурных границ.

Удалить только явно ненужные временные файлы, созданные в рамках работы. Не удалять пользовательские или нераспознанные файлы.

## Критерии готовности

Задача завершена, когда:

1. Номенклатура хранится в БД.
2. CRUD, поиск и фильтрация работают через API.
3. Артикул уникален и валидируется.
4. Цена хранится через `Decimal/Numeric`.
5. Позиция заказа может ссылаться на номенклатуру.
6. Коммерческий snapshot позиции заказа не зависит от последующих изменений номенклатуры.
7. Существующие заказы не повреждены.
8. Миграции применяются и откатываются.
9. Тесты проходят.
10. Каноническая документация и два HTML-статуса актуализированы.
11. В diff отсутствуют несвязанные изменения.
