## Цель

Реализовать persistent-справочник единиц измерения и связать его с номенклатурой как базовую единицу хранения.

Текущие данные должны быть безопасно перенесены без потери совместимости.

## Архитектурная граница

В рамках v0.8.3 реализуется только одна базовая единица хранения для номенклатуры.

Не реализовывать:

- единицу продажи;
- единицу закупки;
- единицу списания;
- альтернативные единицы;
- коэффициенты пересчёта;
- упаковки;
- складской учёт;
- остатки;
- партии;
- закупки;
- производственное списание;
- автоматическую конвертацию количества.

Эти функции должны остаться будущими отдельными итерациями.

## Сущность UnitOfMeasure

Создать persistent-сущность единицы измерения.

Минимальные поля:

- `id`;
- уникальный стабильный `code`;
- полное наименование;
- краткое обозначение;
- категория величины;
- точность;
- признак активности;
- системный или пользовательский признак, если это соответствует архитектуре проекта;
- дата создания;
- дата изменения.

Точные названия полей определить по соглашениям проекта.

## Категории величин

Предусмотреть системные категории:

- `QUANTITY` — количество;
- `LENGTH` — длина;
- `AREA` — площадь;
- `MASS` — масса;
- `TIME` — время;
- `SERVICE` — услуга.

Категория величины должна быть системным enum или эквивалентным контролируемым значением.

Не разрешать произвольные строковые категории без валидации.

## Точность

Поле точности должно определять допустимое количество знаков после запятой при вводе количества.

Примеры:

- штука — 0;
- комплект — 0;
- метр — 3;
- килограмм — 3;
- час — 2.

Не использовать `float`.

Количество и точность должны быть совместимы с `Decimal/Numeric`.

В рамках этой задачи не требуется менять все существующие поля количества в заказах, если это выходит за границы roadmap. Необходимо только подготовить корректную модель единиц.

## Базовые единицы

Добавить минимальный набор системных единиц через безопасный механизм инициализации или миграции.

Рекомендуемый набор:

- `PCS` — Штука — `шт.`;
- `SET` — Комплект — `компл.`;
- `PAIR` — Пара — `пар.`;
- `M` — Метр — `м`;
- `M2` — Квадратный метр — `м²`;
- `G` — Грамм — `г`;
- `KG` — Килограмм — `кг`;
- `MIN` — Минута — `мин.`;
- `HOUR` — Час — `ч`;
- `SERVICE` — Услуга — `усл.`.

Коды должны быть стабильными и не зависеть от локализованного названия.

Не помещать seed-данные в production-код как frontend mock.

Повторное применение механизма инициализации не должно создавать дубликаты.

## CRUD API

Добавить API для:

- создания единицы измерения;
- получения списка;
- получения одной единицы;
- редактирования;
- изменения активности;
- поиска по коду, полному наименованию и краткому обозначению;
- фильтрации по категории величины;
- фильтрации по активности.

Физическое удаление используемой единицы не требуется.

Предпочтительно использовать деактивацию.

API должно соответствовать существующим соглашениям проекта по:

- маршрутам;
- схемам;
- ошибкам;
- валидациям;
- пагинации или лимитам;
- operationId.

## Правила валидации

Обязательно обеспечить:

- уникальность `code`;
- непустое наименование;
- непустое краткое обозначение;
- допустимую категорию величины;
- допустимый диапазон точности;
- невозможность деактивировать или изменять системную единицу способом, который разрушает существующие связи, если введён системный признак;
- невозможность присвоить номенклатуре несуществующую единицу;
- контролируемое поведение для неактивной единицы.

Новая номенклатура не должна выбирать неактивную единицу.

Существующая номенклатура с ранее назначенной единицей должна продолжать отображаться и открываться даже после деактивации единицы.

## Связь с номенклатурой

Добавить nullable или обязательную связь согласно фактическим legacy-данным:

- `Nomenclature.storage_unit_id` или эквивалентное поле;
- внешний ключ на `UnitOfMeasure`.

После миграции для всех корректных существующих записей должна быть назначена единица.

Если legacy-значение пустое или некорректное, выбрать безопасную стратегию:

- оставить nullable;
- либо назначить согласованную единицу по умолчанию только при наличии достаточных оснований.

Не подменять неизвестные данные молча.

## Legacy-поле единицы

Текущее текстовое поле единицы хранения необходимо проанализировать.

Предпочтительный порядок:

1. собрать фактические уникальные значения;
2. сопоставить известные значения с новыми единицами;
3. создать недостающие единицы при безопасной однозначной нормализации;
4. записать внешний ключ;
5. сохранить обратную совместимость на переходный период;
6. не удалять legacy-поле в этой же миграции, если оно используется API, frontend или историческими данными.

Если legacy-поле остаётся:

- пометить его deprecated в документации;
- не использовать как основной источник истины;
- новые изменения должны работать через связь со справочником;
- удалить поле только отдельной последующей миграцией после подтверждения отсутствия зависимостей.

## Миграция

Создать обратимую Alembic-миграцию.

Миграция должна:

- создать таблицу единиц измерения;
- создать системные единицы;
- добавить связь с номенклатурой;
- мигрировать известные legacy-значения;
- не потерять данные;
- применяться на существующей базе;
- корректно откатываться;
- проходить `alembic check`;
- не создавать дубликаты при штатном применении.

Проверить фактический upgrade и downgrade на тестовой базе.

## Frontend

Добавить страницу управления единицами измерения в настройках справочников.

Рекомендуемый маршрут определить по текущей структуре, например:

`/settings/catalogs/units-of-measure`

Интерфейс должен поддерживать:

- список единиц;
- поиск;
- фильтр по категории величины;
- фильтр активности;
- создание;
- редактирование;
- деактивацию;
- отображение кода;
- отображение полного и краткого наименования;
- отображение категории;
- отображение точности;
- понятные ошибки валидации.

Следовать текущему стилю SPORT UNIFORM и существующим компонентам настроек.

Не выполнять общий редизайн портала.

## Карточка номенклатуры

Заменить свободное текстовое поле единицы хранения на выбор из persistent-справочника.

Требования:

- показывать только активные единицы для нового выбора;
- отображать краткое обозначение и полное название;
- сохранять `storage_unit_id`;
- после перезагрузки отображать выбранную единицу;
- корректно показывать ранее назначенную деактивированную единицу;
- не позволять сохранить несуществующий идентификатор;
- не ломать редактирование других полей номенклатуры.

Searchable combobox допустим, если это соответствует текущим UI-компонентам. Для небольшого списка допустим обычный контрол выбора, но архитектура должна выдерживать расширение справочника.

## Позиция заказа

Не добавлять отдельный выбор единицы в позицию заказа без явного требования roadmap.

Проверить только отсутствие регрессии:

- выбор номенклатуры в заказе работает;
- snapshot наименования и цены сохраняется;
- количество и расчёты не ломаются;
- существующие позиции без новой связи продолжают работать.

Если текущая позиция заказа уже хранит текстовое обозначение единицы, не менять её семантику без отдельного ADR и задачи.

## Критерии готовности

Задача завершена, когда:

1. существует persistent-справочник единиц измерения;
2. категории величин валидируются;
3. код единицы уникален;
4. поддерживается точность;
5. базовые единицы созданы безопасно;
6. номенклатура связана с единицей через внешний ключ;
7. legacy-значения мигрированы без потери данных;
8. карточка номенклатуры использует справочник;
9. CRUD и фильтры работают;
10. неактивные единицы корректно обрабатываются;
11. существующая номенклатура и заказы не повреждены;
12. миграция применяется и откатывается;
13. тесты и сборки проходят;
14. документация и HTML-статусы обновлены;
15. P0/P1 отсутствуют.
