## Цель

Реализовать persistent-механизм дополнительных реквизитов номенклатуры:

- библиотеку определений полей;
- варианты значений для полей выбора;
- привязку полей к категориям;
- наследование полей по дереву категорий;
- обязательность и значения по умолчанию;
- хранение типизированных значений в карточке номенклатуры;
- индивидуальные реквизиты конкретной номенклатуры.

## Архитектурная граница

Дополнительный реквизит описывает номенклатуру в целом.

Примеры:

- тип воротника;
- длина рукава;
- посадка;
- пол;
- возрастная группа;
- состав;
- плотность;
- технология нанесения.

Дополнительный реквизит не является характеристикой или вариантом исполнения.

Не реализовывать в этой задаче:

- размеры и цвета как варианты;
- генерацию SKU;
- характеристики номенклатуры;
- варианты номенклатуры;
- выбор вариантов в заказе;
- модели изделий;
- лекала;
- спецификации;
- фотографии;
- склад;
- производство.

## Сущности

Предусмотреть persistent-сущности в соответствии с фактической архитектурой проекта.

### CustomFieldDefinition

Минимальные поля:

- `id`;
- уникальный стабильный `code`;
- `name`;
- `description`;
- тип данных;
- признак активности;
- признак системного или пользовательского поля при необходимости;
- единица измерения для числовых полей при необходимости;
- возможность поиска;
- возможность фильтрации;
- видимость в списке номенклатуры;
- дата создания;
- дата изменения.

### CustomFieldOption

Для полей одиночного и множественного выбора:

- `id`;
- `field_definition_id`;
- стабильный `code`;
- отображаемое значение;
- порядок;
- активность.

Код варианта должен быть уникален в пределах определения поля.

### CategoryField

Связь определения поля с категорией:

- `category_id`;
- `field_definition_id`;
- обязательность;
- значение по умолчанию;
- порядок;
- секция карточки;
- признак наследования;
- настройка видимости;
- допустимые значения или ограничение вариантов при необходимости;
- дата создания;
- дата изменения.

Не дублировать одно и то же поле в одной категории.

### NomenclatureFieldValue

Типизированное значение поля для номенклатуры:

- `nomenclature_id`;
- `field_definition_id`;
- ссылка на источник назначения поля при необходимости;
- типизированное значение;
- признак индивидуального реквизита при необходимости;
- дата создания;
- дата изменения.

Не хранить все значения в одном нетипизированном JSON без схемы и валидации.

Допустимо использовать одну таблицу значений с отдельными колонками для типов либо другой проверяемый EAV-подход, если он обеспечивает целостность и типизацию.

## Поддерживаемые типы данных

Минимально реализовать:

- `STRING` — строка;
- `TEXT` — многострочный текст;
- `INTEGER` — целое число;
- `DECIMAL` — десятичное число;
- `BOOLEAN` — логическое значение;
- `DATE` — дата;
- `SINGLE_SELECT` — одиночный выбор;
- `MULTI_SELECT` — множественный выбор;
- `COLOR` — цвет.

Ссылку на произвольный справочник можно оставить planned, если её безопасная реализация чрезмерно расширяет объём.

Для `DECIMAL` не использовать `float`.

## Правила определения полей

Обеспечить:

- уникальность `code`;
- непустое название;
- неизменяемость типа данных после появления сохранённых значений либо безопасную блокировку такого изменения;
- запрет физического удаления используемого поля;
- деактивацию вместо удаления;
- валидацию значений по типу;
- проверку допустимых вариантов выбора;
- контролируемое поведение неактивных полей и вариантов.

Неактивное поле:

- не предлагается для новых назначений;
- сохраняет существующие значения;
- корректно отображается в существующей карточке с отметкой статуса.

## Назначение полей категориям

В интерфейсе категории добавить управление набором реквизитов.

Пользователь должен иметь возможность:

- выбрать поле из библиотеки;
- назначить его категории;
- определить обязательность;
- задать значение по умолчанию;
- определить порядок;
- выбрать секцию карточки;
- удалить назначение, если это не приводит к потере данных;
- увидеть унаследованные поля отдельно от собственных.

Не создавать новое определение поля автоматически при каждом назначении.

## Наследование

Дочерняя категория получает поля всех родительских категорий.

Пример структуры:

    Спортивная форма
      - Вид спорта
      - Пол

        Футбол
          - Тип комплекта

            Футболки
              - Тип воротника
              - Длина рукава

Номенклатура в категории `Футболки` должна получить весь итоговый набор:

- Вид спорта;
- Пол;
- Тип комплекта;
- Тип воротника;
- Длина рукава.

Правила:

- наследование проходит по всей цепочке родителей;
- циклы категорий уже запрещены и не должны появиться;
- дочерняя категория может сделать поле обязательным;
- может изменить порядок и секцию;
- может задать значение по умолчанию;
- может сузить допустимые варианты выбора;
- не может изменить тип данных;
- не может переопределить смысл поля;
- конфликтующие назначения должны выявляться и возвращать понятную ошибку.

При перемещении категории необходимо проверить итоговый набор полей и существующие значения номенклатуры.

## Обязательные поля

При создании и редактировании номенклатуры backend обязан проверять обязательные реквизиты итоговой категории.

Проверка должна учитывать:

- собственные поля категории;
- унаследованные поля;
- override обязательности;
- значения по умолчанию;
- индивидуальные реквизиты, если они обязательны.

Frontend-валидация не заменяет backend-валидацию.

Существующие карточки, созданные до появления новых обязательных полей, не должны стать недоступными для просмотра.

Определить безопасное поведение:

- разрешать открыть карточку;
- показывать незаполненные обязательные поля;
- требовать заполнение при следующем содержательном сохранении;
- либо использовать иной явно документированный режим совместимости.

Не подставлять фиктивные значения.

## Значения по умолчанию

Значение по умолчанию должно:

- соответствовать типу поля;
- проходить валидацию;
- использоваться при создании новой номенклатуры;
- не перезаписывать уже сохранённое значение;
- корректно наследоваться;
- поддерживать override в дочерней категории.

## Индивидуальные реквизиты

Добавить возможность назначить поле только одной номенклатуре.

Пользователь должен иметь возможность:

- выбрать существующее поле из библиотеки;
- назначить его только этой карточке;
- определить обязательность при необходимости;
- заполнить значение;
- удалить индивидуальное назначение, если это безопасно.

В рамках этой итерации не обязательно разрешать создавать новое определение поля прямо из карточки, если это усложняет контроль дублей.

Предпочтительно:

- определения полей создаются в отдельном справочнике;
- карточка выбирает существующее определение;
- расширение поля на категорию выполняется в настройках категории.

## Frontend: категории

В карточке или интерфейсе категории добавить раздел «Реквизиты».

Показывать:

- собственные поля;
- унаследованные поля;
- категорию-источник;
- обязательность;
- значение по умолчанию;
- порядок;
- секцию;
- статус поля.

Действия:

- добавить поле;
- изменить настройки собственного назначения;
- удалить собственное назначение;
- переопределить разрешённые настройки унаследованного поля;
- перейти к определению поля.

Унаследованное поле нельзя физически удалить из дочерней категории.

## Frontend: карточка номенклатуры

Добавить вкладку или секцию «Реквизиты».

Поля должны строиться по итоговой схеме категории.

Требования:

- группировка по секциям;
- отображение обязательности;
- корректный контрол для каждого типа данных;
- значения по умолчанию;
- понятные ошибки валидации;
- сохранение вместе с карточкой либо отдельным надёжным действием;
- повторная загрузка сохранённых значений;
- отображение унаследованного источника при необходимости;
- управление индивидуальными реквизитами;
- корректная адаптивность.

При смене категории:

- пересчитать итоговую схему;
- сохранить совместимые значения;
- не удалять данные молча;
- показать предупреждение о несовместимых или больше не назначенных полях;
- определить безопасное поведение для исторических значений.

## Исторические значения

Если поле перестало входить в текущую схему категории, но значение уже существует:

- не удалять его автоматически;
- хранить как историческое или orphaned-значение;
- показывать пользователю предупреждение;
- дать осознанное действие очистки;
- не включать такое поле в обычную обязательную валидацию новой схемы.

Выбранную стратегию зафиксировать в ADR.

## Миграция

Создать обратимую Alembic-миграцию.

Миграция должна:

- создать таблицы определений;
- создать таблицы вариантов;
- создать связи категорий;
- создать хранение значений;
- создать необходимые ограничения и индексы;
- не изменять существующие данные номенклатуры;
- применяться на существующей базе;
- корректно откатываться;
- проходить `alembic check`;
- проходить фактический upgrade/downgrade.

Не переносить текущие базовые поля номенклатуры в custom fields.

Поля `article`, `name`, `type`, `category`, `storage_unit`, `base_price`, `currency`, `is_active` остаются системными.

## Критерии готовности

Задача завершена, когда:

1. существует библиотека типизированных реквизитов;
2. варианты выбора хранятся persistent;
3. поля назначаются категориям;
4. наследование работает по всей цепочке;
