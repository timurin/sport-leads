# Задача Codex: финализация и push этапа v0.8.1–v0.8.5

## Контекст

Проект: Sport-Lead.

Пользователь выполнил ручную визуальную и функциональную проверку `v0.8.5 — Nomenclature Workspace and Editable Card`.

Результат принят. Интерфейс признан пригодным для дальнейшей работы. Дополнительная UX-шлифовка будет выполняться позднее отдельными задачами.

Завершены:

- v0.8.1 — Nomenclature Core;
- patch v0.8.1 — выбор номенклатуры в позиции заказа;
- документационная архитектура этапа v0.8;
- v0.8.2 — Nomenclature Types and Category Hierarchy;
- v0.8.3 — Units of Measure Directory;
- v0.8.4 — Category Custom Fields;
- актуализация roadmap с добавлением рабочего места;
- v0.8.5 — Nomenclature Workspace and Editable Card.

P0/P1 по результатам автоматических и ручных проверок отсутствуют.

## Цель

Провести финальную проверку всех незакоммиченных изменений этапа номенклатуры, создать commit и отправить текущую ветку в `origin`.

Не выполнять merge в `main`.

## Предварительная проверка

Перед commit проверить:

- текущую ветку;
- `git status`;
- список изменённых файлов;
- список новых файлов;
- отсутствие несвязанных изменений;
- отсутствие секретов, локальных конфигураций и временных файлов;
- отсутствие случайно добавленных файлов кэша;
- соответствие изменений этапу v0.8.1–v0.8.5;
- актуальность канонической документации;
- актуальность двух HTML-статусов;
- непротиворечивость roadmap;
- отсутствие ложных отметок о реализации v0.8.6 и v0.8.7.

Не удалять пользовательские или нераспознанные файлы.

Если обнаружен P0 или P1, commit и push не выполнять. Остановиться и предоставить отчёт.

## Границы зафиксированного этапа

В commit должны войти относящиеся к этапу изменения:

- persistent-ядро номенклатуры;
- CRUD и API;
- связь с позициями заказов;
- выбор номенклатуры в заказе;
- коммерческий snapshot;
- типы номенклатуры;
- иерархические категории;
- справочник единиц измерения;
- custom fields;
- типизированные значения;
- workspace номенклатуры;
- дерево категорий;
- поиск и фильтры;
- отдельная карточка номенклатуры;
- режимы просмотра и редактирования;
- миграции этапа;
- backend и frontend тесты;
- ADR;
- roadmap;
- project-structure;
- ERP-check;
- оба HTML-статуса;
- задачи Codex, относящиеся к этапу, если по правилам репозитория они хранятся в Git.

Не включать несвязанные изменения других модулей.

## Контрольные проверки

Перед commit повторно выполнить доступные проверки:

- backend pytest;
- frontend tests;
- TypeScript;
- ESLint;
- production build;
- `scripts/check_project.py`;
- OpenAPI;
- уникальность operationId;
- Alembic check;
- проверку текущей цепочки миграций;
- UTF-8;
- HTML5;
- отсутствие внешних зависимостей в HTML-статусах;
- `git diff --check`.

Не выполнять полный downgrade production-базы. Использовать безопасную тестовую среду и действующие проектные процедуры.

## Проверка миграций

Убедиться, что цепочка миграций содержит и последовательно связывает:

- `c3d4e5f6a789_add_nomenclature_core.py`;
- `d4e5f6a7b890_add_nomenclature_types_and_categories.py`;
- `e5f6a7b8c901_add_units_of_measure.py`;
- `f6a7b8c9d012_add_nomenclature_custom_fields.py`.

Проверить:

- единственную голову Alembic;
- корректные `down_revision`;
- отсутствие параллельных голов;
- отсутствие пропущенных миграций.

## Git

После успешных проверок:

1. добавить относящиеся к этапу файлы в индекс;
2. проверить staged diff;
3. создать один итоговый commit;
4. отправить текущую ветку в `origin`;
5. настроить upstream, если ветка ещё не связана с удалённой.

Рекомендуемое сообщение commit:

`feat: complete nomenclature workspace through v0.8.5`

Если фактический diff уже логически разделён на подготовленные commits, не переписывать историю без необходимости. Зафиксировать оставшиеся изменения осмысленным commit.

Не выполнять:

- merge в `main`;
- rebase опубликованной истории;
- force push;
- удаление веток;
- создание релиза;
- изменение тегов;
- squash существующих commits без отдельного указания.

## После push

Проверить:

- commit существует локально;
- commit отправлен в `origin`;
- локальная ветка связана с удалённой;
- рабочее дерево чистое;
- локальная и удалённая ветки синхронизированы.

## Итоговый отчёт

Указать:

1. текущую ветку;
2. результаты предварительной проверки;
3. результаты всех тестов и сборок;
4. состояние цепочки миграций;
5. список файлов или групп изменений, вошедших в commit;
6. commit hash;
7. сообщение commit;
8. результат push;
9. имя удалённой ветки;
10. состояние upstream;
11. итоговый `git status`;
12. оставшиеся P0/P1/P2;
13. подтверждение, что merge в `main` не выполнялся;
14. следующий пункт roadmap: `v0.8.6 — Characteristics and Variants`.

После успешного push не начинать v0.8.6 в рамках этой задачи.
