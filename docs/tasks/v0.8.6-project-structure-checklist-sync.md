# v0.8.6 — Project Structure Checklist and Status Sync

## Контекст

Текущий документ структуры проекта слишком подробный, смешивает фактическое состояние, архитектурные пояснения и будущие планы.

Нужно заменить его компактным рабочим чек-листом. Этот чек-лист станет основным документом контроля структуры Sport-Lead.

После завершения любого пункта разработки Codex должен закрывать соответствующий чекбокс и синхронно обновлять HTML-представление.

## Цель

1. Обновить канонический Markdown-файл структуры проекта.
2. Добавить в `AGENTS.md` обязательные правила его ведения.
3. Создать синхронное автономное HTML-представление.

## Файлы

Обновить:

```
docs/architecture/project-structure.md
AGENTS.md
docs/erp/status/project-structure.html
```

Если канонический Markdown структуры сейчас находится по другому пути:

* определить действующий путь;
* не создавать конкурирующие документы;
* перенести новую структуру в единственный канонический файл;
* исправить ссылки на него в документации;
* в итоговом отчёте указать фактический путь.

## 1. Обновить Markdown-структуру

Заменить старое содержимое канонического `project-structure.md` на новую сокращённую структуру с чекбоксами из задания.

Перед сохранением сверить каждый уже закрытый пункт с:

* текущим кодом;
* Git-историей;
* миграциями;
* backend API;
* frontend routes и components;
* тестами;
* отчётами завершённых этапов.

Не оставлять `[x]`, если реализация не подтверждена.

Если крупный пункт реализован только частично:

* не использовать нестандартный статус `[~]`;
* разбить его на отдельные подпункты;
* реализованным поставить `[x]`;
* оставшимся поставить `[ ]`.

В начале файла указать:

* дату обновления;
* текущую версию;
* ветку;
* короткий Git SHA.

## 2. Добавить правила в AGENTS.md

Добавить отдельный раздел:

```
## Project structure checklist
```

Зафиксировать обязательные правила:

1. `docs/architecture/project-structure.md` — канонический чек-лист функциональной структуры проекта.

2. После завершения задачи Codex обязан проверить, закрывает ли она один или несколько пунктов структуры.

3. Завершённые и проверенные пункты отмечаются `[x]`.

4. Незавершённые пункты остаются `[ ]`.

5. Частично реализованный пункт необходимо разделить на более точные подпункты.

6. Нельзя закрывать чекбокс только по наличию UI, API, модели или документации.

7. Чекбокс закрывается после проверки полного объёма, обозначенного конкретной строкой.

8. Новые функциональные области добавляются в соответствующий раздел структуры.

9. Удалённые или изменённые требования должны отражаться в структуре.

10. При каждом изменении Markdown-файла одновременно обновляется:

    docs/erp/status/project-structure.html

11. Markdown и HTML должны содержать одинаковые разделы, пункты и состояния чекбоксов.

12. Итоговый отчёт каждой задачи должен указывать:

    * какие чекбоксы закрыты;
    * какие добавлены;
    * какие изменены;
    * обновлён ли HTML.

13. Если задача не влияет на структуру, в отчёте явно писать:

    Project structure checklist: changes not required.

14. Структуру обновлять в той же задаче, которая завершает функциональность, а не отдельной будущей задачей.

15. Roadmap не считается заменой структуры: roadmap определяет последовательность, структура — фактическую готовность модулей.

Не дублировать эти правила в нескольких разделах `AGENTS.md`.

## 3. Создать HTML-представление

Создать или полностью обновить:

```
D:\Projects\sport-leads\docs\erp\status\project-structure.html
```

HTML должен строиться на основании актуального `project-structure.md` и содержать те же:

* разделы;
* подпункты;
* формулировки;
* состояния чекбоксов;
* метаданные версии и Git-коммита.

Требования:

* автономный single-file HTML;
* встроенные CSS и JavaScript;
* без CDN;
* без внешних шрифтов;
* без внешних библиотек;
* адаптивный desktop, laptop, tablet и mobile;
* светлый ERP-интерфейс;
* боковая или верхняя навигация по разделам;
* поиск по пунктам;
* фильтры:

  * все;
  * выполнено;
  * не выполнено;
* сворачивание разделов;
* общий прогресс;
* прогресс каждого раздела;
* количество выполненных и оставшихся пунктов;
* визуальные чекбоксы;
* дата обновления;
* версия;
* ветка;
* Git SHA;
* кнопка раскрытия всех разделов;
* кнопка сворачивания всех разделов;
* режим печати;
* корректное открытие напрямую из файловой системы.

Чекбоксы в HTML не должны изменять состояние проекта. HTML является отчётом, а источником истины остаётся Markdown.

## 4. Проверить старую документацию

Найти упоминания старого файла структуры и проверить:

* `README.md`;
* `AGENTS.md`;
* `docs/erp-check.md`;
* `docs/roadmap.md`;
* файлы в `docs/erp/status`;
* документы архитектуры;
* активные task-файлы.

Исправлять только ссылки и правила, необходимые для перехода на новый чек-лист.

Не обновлять содержание roadmap в этой задаче.

## 5. Ограничения

Не изменять:

* production-код;
* backend API;
* database models;
* migrations;
* frontend application;
* бизнес-логику;
* roadmap и порядок этапов.

Не удалять старые документы без доказанной необходимости.

Если найден второй документ структуры:

* определить канонический;
* убрать конфликт ссылок;
* не удалять файл автоматически;
* указать его в отчёте как кандидат на архивирование.

## Проверки

Выполнить:

```
git diff --check
python scripts/check_project.py
```

Дополнительно проверить:

* Markdown корректно отображает чекбоксы;
* количество чекбоксов Markdown совпадает с HTML;
* количество выполненных чекбоксов совпадает;
* количество невыполненных чекбоксов совпадает;
* названия разделов совпадают;
* HTML не содержит внешних HTTP/HTTPS-зависимостей;
* HTML открывается локально;
* поиск работает;
* фильтры работают;
* прогресс рассчитывается корректно;
* адаптивная версия не создаёт горизонтальный скролл.

При необходимости добавить небольшой скрипт проверки синхронизации в `scripts/`, но не вводить генератор или новую зависимость без необходимости.

## Критерии готовности

* канонический `project-structure.md` заменён компактным чек-листом;
* фактические `[x]` сверены с кодом;
* незавершённые пункты имеют `[ ]`;
* частичные пункты разбиты;
* `AGENTS.md` содержит обязательное правило обновления структуры;
* HTML создан по требуемому пути;
* Markdown и HTML синхронизированы;
* roadmap не изменён;
* production-код не изменён;
* проверки пройдены.

## Итоговый отчёт

Вывести:

1. текущую ветку и HEAD;
2. фактический путь канонического Markdown;
3. количество разделов;
4. общее количество чекбоксов;
5. количество `[x]`;
6. количество `[ ]`;
7. пункты, статус которых был исправлен после проверки кода;
8. изменения `AGENTS.md`;
9. путь HTML-файла;
10. результаты проверки синхронизации;
11. результаты `git diff --check`;
12. результат `scripts/check_project.py`;
13. подтверждение, что roadmap и production-код не изменялись.
