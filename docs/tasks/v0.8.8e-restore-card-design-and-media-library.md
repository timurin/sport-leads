# v0.8.8e — Restore Card Design and Media Library

## Контекст

Последние изменения сохранили механику вкладок и редактирования, но визуально разрушили утверждённый дизайн карточки.

Использовать как единственный визуальный источник истины:

- `docs/design/nomenclature-card-reference-v1.html`;
- утверждённый screenshot этого эталона.

Не проектировать новый интерфейс и не интерпретировать макет свободно.

## Цель

Вернуть визуальный дизайн HTML-эталона, сохранив всю текущую рабочую механику:

- переключение вкладок;
- отдельное редактирование блоков;
- сохранение;
- custom fields;
- характеристики и варианты;
- media API;
- предупреждение о несохранённых изменениях.

## 1. Общая композиция

Повторить из HTML-эталона:

- ширину рабочего контейнера;
- фон страницы;
- шапку карточки;
- размеры и начертания текста;
- цвета;
- вкладки;
- карточки;
- границы;
- радиусы;
- тени;
- внутренние и внешние отступы;
- высоту полей и кнопок.

Не использовать текущую упрощённую компоновку со случайными узкими блоками и большим пустым пространством.

## 2. Вкладки

Текущую механику переключения сохранить.

Содержимое каждой вкладки должно использовать доступную ширину страницы.

### Основное

Desktop:

- левая колонка около 65%;
- правая колонка около 35%.

Слева:

- Основная информация;
- системные поля;
- кнопка редактирования блока.

Справа:

- Карточка;
- краткая история изменений.

### Реквизиты

- основной блок занимает полную полезную ширину;
- таблица реквизитов не должна занимать только половину экрана;
- кнопка редактирования в заголовке;
- поля и таблица оформлены как в HTML-эталоне.

### Характеристики и варианты

- блок занимает полную полезную ширину;
- сохранить текущие назначения, значения, генерацию и варианты;
- таблица должна совпадать с эталоном;
- убрать чрезмерное пустое пространство;
- действия выровнять в единую toolbar-строку.

### Изображения и файлы

- media library занимает полную полезную ширину;
- не размещать её в узкой правой колонке;
- галерея должна выглядеть как полноценная библиотека;
- на desktop показывать несколько миниатюр в строке;
- на mobile адаптивную сетку.

### История

- полноценная timeline на полезную ширину;
- не использовать маленький пустой блок.

## 3. Редактирование

Сохранить отдельное редактирование каждого блока.

Каждый редактируемый блок имеет:

- Редактировать;
- Сохранить;
- Отмена;
- Сохранение…;
- Сохранено;
- Ошибка сохранения.

Не возвращать одну глобальную кнопку, которая редактирует только основную информацию.

Не менять существующие API-контракты.

## 4. Media plugin

В frontend установить:

    npm install @uppy/core @uppy/react @uppy/thumbnail-generator

Использовать Uppy в headless-режиме.

Не использовать готовый Uppy Dashboard, так как он изменит утверждённый дизайн.

Uppy должен отвечать за:

- выбор нескольких изображений;
- drag-and-drop;
- локальные previews;
- очередь загрузки;
- прогресс;
- состояния success/error;
- ограничения MIME и размера;
- удаление файла из очереди;
- повтор загрузки после ошибки.

Загрузка должна использовать существующий backend NomenclatureMedia API.

Не менять локальное файловое хранилище backend.

Если текущий API использует JSON/base64:

- реализовать собственный upload handler Uppy;
- преобразовывать выбранный файл в текущий transport contract;
- не добавлять второй backend endpoint только ради Uppy.

## 5. Media library UI

Повторить media-блок HTML-эталона:

- кнопка «Загрузить»;
- dropzone;
- сетка миниатюр;
- локальный preview до завершения загрузки;
- прогресс на миниатюре;
- отметка «Главное»;
- действие назначения главным;
- изменение порядка;
- удаление;
- редактирование alt-текста;
- плитка «Добавить»;
- понятное пустое состояние.

После успешной загрузки:

- новая миниатюра появляется без полной перезагрузки;
- данные синхронизируются с backend;
- после повторного открытия карточки изображения сохраняются.

Не использовать внешние demo-изображения из HTML-файла.

## 6. Responsive

Проверить:

- 1920×1080;
- 1440×900;
- 1366×768;
- 1024×768;
- 768×1024;
- 390×844;
- 360×800.

Требования:

- desktop использует всю рабочую ширину;
- tablet переходит в одну колонку;
- mobile не имеет общего горизонтального scroll;
- вкладки прокручиваются локально;
- таблицы прокручиваются только внутри блока;
- media grid адаптивна;
- действия блока доступны на мобильном.

- механику custom fields;
- механику характеристик и вариантов;
- существующее media storage;

## Критерии готовности

1. Внешний вид снова соответствует HTML-эталону.
2. Вкладки продолжают переключаться.
3. Каждый блок редактируется и сохраняется отдельно.
4. Реквизиты и характеристики используют полную полезную ширину.
5. Media library не зажата в узкой колонке.
6. Uppy интегрирован в headless-режиме.
7. Загрузка, previews, главное фото, сортировка и удаление работают.
8. Текущие backend/API механизмы не сломаны.
9. Desktop, tablet и mobile соответствуют эталону.
10. P0/P1 отсутствуют.

- сохранённую механику;

Commit и push не выполнять.
