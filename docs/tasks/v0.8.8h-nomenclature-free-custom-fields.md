# v0.8.8h — Свободные дополнительные реквизиты номенклатуры

## Критическое ограничение

НЕ ИЗМЕНЯТЬ:

- утверждённый дизайн карточки номенклатуры;
- HTML/CSS-композицию страницы;
- сетку;
- размеры блоков;
- вкладки;
- шапку;
- расположение секций;
- responsive-поведение;
- механику остальных вкладок.

Разрешено менять только содержимое и механику блока «Реквизиты», а также необходимые backend/API сущности.

## Цель

Разрешить пользователю добавлять произвольные реквизиты прямо в карточке номенклатуры.

Примеры:

- Бренд;
- ТН ВЭД;
- Страна происхождения;
- Коллекция;
- Состав;
- любое другое пользовательское поле.

Общий справочник «Дополнительные реквизиты» должен автоматически собирать созданные определения и значения и предлагать их повторно.

## Механика в карточке

В существующем блоке «Реквизиты» добавить кнопку:

`+ Добавить реквизит`

После нажатия открыть компактную форму внутри текущего блока или существующий диалог проекта.

Пользователь вводит наименование реквизита.

Во время ввода выполнять поиск по существующему справочнику дополнительных реквизитов:

- по точному названию;
- по частичному совпадению;
- без учёта регистра.

Если найдено существующее поле:

- показать его в подсказках;
- после выбора использовать существующий тип;
- подгрузить существующие допустимые значения.

Если совпадений нет:

- предложить `Создать реквизит «... »`;
- запросить тип значения;
- создать новое определение;
- сразу назначить его текущей номенклатуре.

## Поддерживаемые типы

Использовать существующую typed-модель custom fields:

- строка;
- текст;
- целое число;
- decimal;
- да/нет;
- дата;
- одиночный выбор;
- множественный выбор;
- цвет.

Не создавать второй параллельный механизм хранения реквизитов.

## Значения справочника

Для полей типа одиночного и множественного выбора:

- показывать существующие значения;
- разрешить выбрать значение;
- разрешить добавить новое значение;
- новое значение сохранять в общий справочник поля;
- после сохранения оно доступно для других карточек.

Пример:

Категория объекта: Номенклатура  
Наименование: ТН ВЭД  
Тип: Одиночный выбор  
Значения:

- 6109100000;
- 6109902000;
- 6110309100.

Для строковых полей разрешить свободный ввод.

Ранее использованные строковые значения можно показывать как подсказки, но не превращать автоматически в закрытый список.

## Справочник «Дополнительные реквизиты»

Существующий справочник должен отображать и хранить:

- категория объекта: `Номенклатура`;
- наименование;
- стабильный code;
- тип значения;
- активность;
- значения для select-типов;
- количество назначений или использования, если легко получить;
- поиск;
- фильтр по типу и активности.

Не создавать отдельный справочник только для этой задачи, если существующий custom-fields каталог уже выполняет эту роль.

## Назначение реквизита

При добавлении предусмотреть область применения:

- только текущая номенклатура;
- текущая категория;
- текущая категория и дочерние категории.

По умолчанию выбирать:

`Только текущая номенклатура`.

Использовать существующие связи индивидуальных и категорийных реквизитов.

## Защита от дублей

Перед созданием нового определения:

- нормализовать название;
- проверить похожие существующие поля;
- показать найденные варианты;
- не создавать точный дубль с тем же нормализованным названием в области `Номенклатура`.

Не выполнять автоматическое объединение разных реквизитов.

## Редактирование блока

В существующем дизайне блока «Реквизиты» сохранить:

- заголовок;
- кнопку редактирования;
- таблицу или список;
- текущие размеры и стили.

Добавить действия:

- добавить;
- изменить значение;
- удалить назначение из карточки;
- перейти к определению реквизита, если это соответствует текущей навигации.

Удаление назначения не должно физически удалять общее определение или значения справочника.

## Backend

Проверить и при необходимости расширить текущие API/service:

- поиск определений реквизитов;
- создание определения из карточки;
- получение options;
- добавление option;
- индивидуальное назначение номенклатуре;
- назначение категории;
- сохранение typed-значения;
- удаление назначения без удаления определения.

Backend остаётся источником валидации.

## Миграция

Использовать текущие таблицы custom fields, если они покрывают механику.

Создавать миграцию только при реальной нехватке структуры.

Не изменять системные поля номенклатуры.

## Тесты

Добавить regression-тесты:

1. поиск существующего реквизита по части названия;
2. выбор существующего реквизита;
3. создание нового произвольного реквизита;
4. защита от точного дубля;
5. создание значения select;
6. повторное предложение созданного значения;
7. индивидуальное назначение;
8. назначение категории;
9. сохранение typed-значения;
10. удаление назначения без удаления определения;
11. повторная загрузка карточки;
12. отсутствие визуальной и функциональной регрессии других блоков.

## Проверки

Выполнить обязательные проверки по AGENTS.md:

- backend pytest;
- frontend tests;
- TypeScript;
- ESLint;
- production build;
- OpenAPI;
- Alembic check при изменении моделей;
- git diff --check.

## Критерии готовности

- произвольный реквизит создаётся из карточки;
- существующие реквизиты находятся и предлагаются;
- существующие значения подставляются;
- новые select-значения пополняют общий справочник;
- значения сохраняются типизированно;
- реквизит можно назначить карточке или категории;
- дизайн и схема карточки визуально не изменились;
- остальные вкладки и механики не затронуты;
- P0/P1 отсутствуют.

В отчёте отдельно подтвердить:

`Дизайн, JSX-композиция, сетка и CSS карточки номенклатуры не изменялись.`

Commit и push не выполнять.
