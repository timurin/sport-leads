Создать один последовательный глобальный roadmap, который:

* отражает фактическое состояние проекта;
* использует подтверждённые статусы из `project-structure.md`;
* разбивает будущую разработку на этапы, функциональные пункты и микрозадачи;
* позволяет отдельно выполнять и проверять небольшие задачи Codex;
* синхронизируется с автономным HTML-файлом;
* становится единственным источником порядка дальнейшей разработки.

## 2. Источники фактического состояния

Использовать следующий приоритет:

1. Текущий production-код.
2. Миграции.
3. Автоматические тесты.
4. `docs/architecture/project-structure.md`.
5. Git-история.
6. `docs/architecture/erp-check.md`.
7. Старый roadmap.
8. Отчёты задач Codex.

Новый roadmap не должен возвращать завышенные статусы, исправленные в структуре.

В частности:

* persistent-функциональность отделять от demo/local UI;
* не считать модуль завершённым только по наличию страницы;
* не считать модуль завершённым только по наличию модели или API;
* не переносить `[x]` из старого roadmap без проверки;
* использовать структуру проекта как основной документ фактической готовности.

## 3. Иерархия roadmap

Использовать четыре уровня.

### Уровень 1 — Этап

Крупный бизнес-контур.

Пример:

```
Этап 5 — Модели изделий
```

### Уровень 2 — Функциональный блок

Самостоятельная часть этапа.

Пример:

```
5.2 — Backend моделей
```

### Уровень 3 — Функциональный пункт

Проверяемая пользовательская или системная возможность.

Пример:

```
5.2.4 — Update API модели
```

### Уровень 4 — Микрозадача Codex

Небольшая техническая задача, выполняемая отдельно.

Пример:

```
5.2.4.1 — Добавить Update schema
5.2.4.2 — Добавить repository update
5.2.4.3 — Добавить PATCH endpoint
5.2.4.4 — Добавить regression tests
```

Не создавать микрозадачи ради формального количества. Каждый уровень добавлять только там, где он действительно помогает разделить реализацию и проверку.

## 4. Статусы

В Markdown использовать только стандартные чекбоксы:

```
[x] — завершено и проверено
[ ] — не завершено
```

Не использовать:

```
[>]
[~]
[!]
[-]
```

Состояние «в работе», блокировки и исключения описывать текстовой строкой внутри пункта:

```
Статус: в работе
Блокер: ...
Исключено из текущего scope: ...
```

Это необходимо для совместимости Markdown и автоматического подсчёта HTML.

## 6. Глубина детализации завершённых этапов

Для уже реализованных этапов не требуется создавать задним числом десятки искусственных микрозадач.

Использовать:

* этап;
* функциональные блоки;
* проверенные завершённые пункты;
* ссылки на существующие task-файлы или версии при необходимости.

Микрозадачи подробно создавать прежде всего для:

* текущего незавершённого функционального блока;
* следующего этапа;
* ближайших зависимых этапов.

Это сохранит roadmap читаемым.

## 7. Первый подробно декомпозированный будущий контур

Подробно разложить на микрозадачи следующий контур:

Зависимости:

Микрозадачи:
- [ ] 5.2.4.1 — Update schema
- [ ] 5.2.4.2 — Repository update
- [ ] 5.2.4.3 — Service validation

- [ ] 5.2.4.5 — Frontend edit form

## 9. Формат файлов микрозадач

Файлы создаются позже, по мере выбора задач, строго в:

```
docs/tasks/
```

Формат имени:

```
<версия>-<код>-<short-name>.md
```

Пример:

```
docs/tasks/v0.9.1-5.2.1-model-database-core.md
docs/tasks/v0.9.2-5.3.1-model-list-api.md
docs/tasks/v0.9.3-5.4.1-model-card-route.md
```

Не создавать сейчас task-файл для каждой будущей микрозадачи.

Roadmap должен содержать список микрозадач, а отдельный task-файл создаётся непосредственно перед началом конкретной работы.

## 10. Баги

Баги оформлять отдельными микрозадачами:

```
5.4.2.B1
5.4.2.B2
```

Task-файлы:

```
docs/tasks/v0.9.3-5.4.2-B1-model-save-regression.md
```

Баг-задача должна включать:

* шаги воспроизведения;
* ожидаемый результат;
* фактический результат;
* область изменения;
* regression test;
* критерий исправления.

Баг не смешивать с новой функциональностью.

## 14. Реальные пути frontend и backend

Во всех пояснениях использовать реальные префиксы:

```
backend/app/
backend/alembic/
backend/tests/

frontend/app/
frontend/components/
frontend/lib/
frontend/types/
```

Next.js route group:

```
frontend/app/(workspace)/
```

не входит в URL.

Пример:

```
URL: /sales/leads
File: frontend/app/(workspace)/sales/leads/page.tsx
```

Не создавать URL вида:

```
/(workspace)/sales/leads
```

## Проверки

Обязательные:

```
git diff --check
python scripts/check_project.py
```

Если `scripts/check_project.py` снова блокируется только из-за недоступного PostgreSQL или Docker:

* не считать это продуктовой ошибкой документационной задачи;
* зафиксировать фактический результат;
* указать точную проваленную проверку;
* указать ошибку подключения;
* не изменять конфигурацию Docker или базы в рамках этой задачи.

Дополнительные проверки:

* канонический roadmap существует только по одному пути;
* Markdown и HTML содержат одинаковое количество этапов;
* количество чекбоксов совпадает;
* количество `[x]` совпадает;
* количество `[ ]` совпадает;
* названия этапов совпадают;
* HTML не содержит внешних HTTP/HTTPS-зависимостей;
* ссылки используют реальные пути;
* `(workspace)` не попадает в URL;
* `docs/architecture/project-structure.md` не изменён;
* production-код не изменён.

## Критерии готовности

* `docs/roadmap/roadmap.md` заменён единым глобальным roadmap;
* завершённые статусы сверены со структурой проекта;
* будущие этапы идут в правильной последовательности;
* этапы моделей, размеров, лекал, спецификаций и техкарт подробно декомпозированы;
* микрозадачи имеют отслеживаемые коды;
* task-файлы пока не созданы массово;
* `AGENTS.md` содержит правила roadmap и микрозадач;
* `docs/erp/status/roadmap.html` синхронизирован;
* структура и ERP-check не изменены;
* production-код не изменён.
